# 工程化与工具面试题

## 一、前端工程化基础

### 什么是前端工程化？

前端工程化指的是用标准化、自动化、模块化、可维护的方式来管理前端项目开发和交付过程。

它通常包括：

1. 模块化。
2. 组件化。
3. 构建与打包。
4. 代码规范。
5. 自动化测试。
6. 环境管理。
7. 部署发布。
8. CI/CD。

一句话总结：前端工程化的核心目标是提升开发效率、协作质量和交付稳定性。

### 与 Webpack 类似的工具有哪些？它们有什么区别？

常见工具有：

1. Webpack
2. Rollup
3. Vite
4. Parcel
5. esbuild

区别大致可以这样理解：

1. Webpack 功能全面，生态成熟，适合复杂工程。
2. Rollup 更擅长打包库，产物更干净。
3. Vite 开发启动快，适合现代前端项目。
4. esbuild 极快，但生态和深度优化能力与传统打包器定位不同。
5. Parcel 更强调开箱即用。

## 二、Webpack 核心原理

### 说说你对 Webpack 的理解？它解决了什么问题？

Webpack 本质上是一个模块打包器和构建工具。

它解决的核心问题有：

1. 浏览器原生不方便直接处理复杂模块依赖。
2. CSS、图片、字体等资源需要统一构建。
3. 开发阶段需要热更新、代理、Source Map 等能力。
4. 生产环境需要压缩、分包、缓存和优化。

一句话总结：Webpack 把“开发阶段的模块化代码和资源”转换成“浏览器可以高效运行的产物”。

### 说说 Webpack 的构建流程。

Webpack 构建流程可以概括为：

1. 从入口文件开始。
2. 分析模块依赖。
3. 对不同资源通过 Loader 做转换。
4. 递归构建模块依赖图。
5. 把模块打包成 chunk。
6. 通过 Plugin 在不同阶段扩展构建能力。
7. 最终输出打包结果。

如果面试官继续追问，可以补充关键词：

1. `entry`
2. `module`
3. `loader`
4. `plugin`
5. `chunk`
6. `bundle`

### 说说 Webpack 中常见的 Loader？解决了什么问题？

Loader 的作用是让 Webpack 能处理非 JavaScript 资源，或者在打包前转换源码。

常见 Loader：

1. `babel-loader`：处理 JS 语法转换。
2. `ts-loader` 或 `swc-loader`：处理 TypeScript。
3. `css-loader`：解析 CSS。
4. `style-loader`：把样式注入页面。
5. `sass-loader` / `less-loader`：处理预处理器样式。
6. `vue-loader`：处理 Vue 单文件组件。
7. 资源类 Loader 或 asset module：处理图片、字体等静态资源。

### 说说 Webpack 中常见的 Plugin？解决了什么问题？

Plugin 是对 Webpack 构建流程的扩展，能够在更广范围介入编译生命周期。

常见 Plugin：

1. `HtmlWebpackPlugin`：生成 HTML 并注入资源。
2. `MiniCssExtractPlugin`：抽离 CSS。
3. `DefinePlugin`：注入全局常量。
4. `CopyWebpackPlugin`：拷贝静态资源。
5. `BundleAnalyzerPlugin`：分析打包体积。
6. 压缩类插件：压缩 JS/CSS。

一句话理解：Loader 主要处理“文件内容转换”，Plugin 主要处理“构建流程扩展”。

### 说说 Loader 和 Plugin 的区别？编写 Loader、Plugin 的思路是什么？

区别主要有：

1. Loader 处理单个模块的转换。
2. Plugin 作用于整个构建流程。

编写 Loader 的思路：

1. 接收源代码。
2. 对代码做转换。
3. 返回转换结果。

编写 Plugin 的思路：

1. 编写包含 `apply` 方法的类。
2. 通过 compiler 或 compilation 生命周期钩子介入构建流程。
3. 在合适阶段完成扩展逻辑。

### 说说 Webpack 的热更新是如何做到的？原理是什么？

HMR 即 Hot Module Replacement，目标是在不刷新整个页面的前提下，只替换发生变化的模块。

大致过程：

1. 文件变更后，Webpack 重新编译对应模块。
2. Dev Server 通知浏览器有更新。
3. 浏览器请求更新后的模块。
4. 运行时替换旧模块并执行更新逻辑。
5. 页面状态尽量保留，不做整页刷新。

### 说说 Webpack proxy 工作原理？为什么能解决跨域？

Webpack Dev Server 的 proxy 本质上是开发服务器代替浏览器去请求目标接口。

浏览器访问的是本地开发服务器，同源；开发服务器再把请求转发到真实后端。

所以它解决跨域的关键是：

1. 浏览器只和同源开发服务器通信。
2. 跨域发生在服务端转发层，不受浏览器同源策略限制。

## 三、Webpack 优化与进阶问题

### 说说如何借助 Webpack 来优化前端性能。

常见方式有：

1. 代码分割，减少首屏包体积。
2. 懒加载和异步加载。
3. 压缩 JS、CSS、图片。
4. 开启 Tree Shaking。
5. 抽离公共依赖。
6. 利用长期缓存策略。
7. 生成体积分析报告，持续治理依赖。

### 如何提高 Webpack 的构建速度？

常见思路：

1. 缩小 Loader 处理范围。
2. 使用缓存。
3. 多进程处理，如 `thread-loader`。
4. 使用更快的编译器，如 SWC、esbuild。
5. 减少不必要插件。
6. 优化依赖结构和模块数量。
7. 使用持久化缓存。

### 如何更好地利用 long term cache？

长期缓存的核心目标是：内容不变时文件名尽量不变，内容变了文件名再变。

常见做法：

1. 使用基于内容的 hash。
2. 把 runtime 单独抽离。
3. 公共依赖和业务代码分包。
4. 避免无关改动导致大面积 hash 变化。

### Tree Shaking 的原理是什么？

Tree Shaking 指的是在打包阶段移除未使用代码。

它依赖：

1. ES Module 的静态结构。
2. 构建阶段能分析导入导出关系。
3. 尽量减少副作用干扰。

### 对于已经 import 但未实际使用的模块，Webpack 还会打包吗？

结论不是绝对的，要看模块格式和副作用情况。

1. 如果是 ES Module 且可静态分析，未使用导出有机会被 Tree Shaking 移除。
2. 如果模块有副作用，即使部分导出没用，也可能保留。
3. CommonJS 场景下 Tree Shaking 能力会弱很多。

### Webpack 的 runtime 做了什么事情？

Webpack runtime 是打包产物中负责模块加载和运行时管理的那部分代码。

它通常负责：

1. 模块缓存。
2. 模块执行。
3. chunk 动态加载。
4. HMR 相关运行时代码。

### Webpack 中 code splitting 是如何动态加载 chunk 的？

动态分包后，Webpack 会在运行时按需加载 chunk。

大致过程：

1. 代码里遇到动态 `import()`。
2. 运行时根据映射关系找到对应 chunk 文件。
3. 浏览器加载这个 chunk。
4. 加载完成后注册模块并执行。

### 打包器如何将打包后的 JS 资源注入 HTML 中？

常见做法是借助插件在构建阶段生成或处理 HTML。

比如：

1. 读取 HTML 模板。
2. 获取本次构建产物列表。
3. 把脚本和样式资源插入模板。
4. 输出最终 HTML 文件。

### 打包器如何加载 JSON、图片等非 JavaScript 资源？

原理上通常有两步：

1. 在构建阶段识别资源类型。
2. 把资源转换成 JS 可引用的模块形式。

比如：

1. JSON 可转成对象导出。
2. 图片可转成 URL、Base64 或资源文件输出。

### 打包器如何加载 style 样式资源？

常见流程：

1. 读取 CSS 文件。
2. 解析 `@import` 和 `url()`。
3. 在开发环境中把样式插入页面。
4. 在生产环境中把 CSS 抽离成独立文件。

### JS 代码压缩 minify 的原理是什么？

常见原理包括：

1. 删除空格、换行、注释。
2. 压缩变量名和函数名。
3. 合并表达式和声明。
4. 常量折叠和预计算。
5. 删除不可达代码。

## 四、Vite、Rollup 与现代构建

### Vite 现在使用的底层打包工具是什么？

Vite 在不同阶段使用的底层能力不同：

1. 开发阶段主要依赖原生 ESM 和 esbuild 做预构建与快速转换。
2. 生产构建阶段默认使用 Rollup。

### Vite 为什么不用 esbuild 做最终打包和代码优化？

核心原因不是 esbuild 不够快，而是 Rollup 在产物优化和生态兼容上更适合生产构建。

常见原因：

1. Rollup 在 Tree Shaking 和产物组织方面更成熟。
2. 生态插件更完善。
3. 复杂库构建和代码输出控制更细。
4. 开发阶段和生产阶段目标不同，Vite 会分别选更合适工具。

### 那在开发模式下使用 esbuild，不用对代码做优化吗？

开发模式的核心目标是快速反馈，而不是最优产物。

所以开发阶段通常只做：

1. 必要的语法转换。
2. 依赖预构建。
3. HMR 支持。

而压缩、深度 Tree Shaking、产物精细优化，通常放到生产构建阶段再做。

### 什么是 Vite 依赖预构建？

依赖预构建指的是：Vite 启动时先把第三方依赖做一次预处理，通常转成更适合浏览器直接加载的 ESM 形式。

它的作用：

1. 提升开发服务器启动速度。
2. 减少浏览器请求碎片化依赖。
3. 兼容 CommonJS 等格式依赖。

### bundless 的优势与不足是什么？

bundless 指开发阶段尽量不整体打包，而是按模块直接提供资源。

优势：

1. 启动快。
2. 增量更新快。
3. 开发体验好。

不足：

1. 生产环境通常仍然需要打包优化。
2. 对复杂依赖和兼容场景要求更高。
3. 模块数量很大时，浏览器请求管理会更复杂。

## 五、包管理与依赖治理

### 如何加速 npm install？

常见做法：

1. 使用稳定且更近的镜像源。
2. 锁定依赖版本，减少频繁解析。
3. 使用缓存。
4. 减少不必要依赖。
5. 在 CI 环境使用更稳定的安装策略。

### npm i 与 npm ci 的区别是什么？

1. `npm install` 更适合日常开发，会根据 `package.json` 和锁文件安装依赖。
2. `npm ci` 更适合 CI 环境，严格依据锁文件安装，速度更快、结果更稳定。

一句话总结：`npm ci` 更强调可重复构建，`npm install` 更强调开发灵活性。

### package-lock.json 有什么作用？如果没有它会怎么样？

锁文件的核心作用是固定依赖树，保证不同环境安装结果一致。

如果没有它：

1. 同样的 `package.json` 可能安装出不同版本依赖。
2. 更容易出现“我这里没问题，你那里有问题”。
3. CI 和本地环境一致性下降。

### package-lock.json 与 yarn.lock 有什么区别？

两者本质上都是锁文件，目标一致：

1. 固定依赖版本。
2. 保证安装一致性。
3. 加快安装解析。

区别主要来自所服务的包管理器不同：

1. `package-lock.json` 对应 npm。
2. `yarn.lock` 对应 Yarn。

### semver 指什么？`~1.2.3` 与 `^1.2.3` 的范围是多少？

semver 指语义化版本规范，通常格式是：`主版本号.次版本号.修订号`。

含义一般是：

1. 主版本：可能有不兼容变更。
2. 次版本：向下兼容的新功能。
3. 修订号：向下兼容的问题修复。

常见范围：

1. `~1.2.3` 通常允许升级到 `<1.3.0`
2. `^1.2.3` 通常允许升级到 `<2.0.0`

### dependencies、devDependencies、peerDependencies、optionalDependencies 有什么区别？

1. `dependencies`：运行时依赖。
2. `devDependencies`：开发阶段依赖。
3. `peerDependencies`：宿主环境应提供的依赖，常见于插件和组件库。
4. `optionalDependencies`：可选依赖，安装失败也不一定中断整个安装。

### 当你引入某个依赖时，你引入的是该依赖下的哪一个文件？

这取决于包的入口解析规则，常见会参考：

1. `exports`
2. `main`
3. `module`
4. `browser`

不同构建工具和运行环境会根据这些字段和解析策略决定最终入口文件。

### package.json 中 main、module、browser、exports 字段有何区别？

1. `main`：CommonJS 默认入口。
2. `module`：通常给 ESM 构建工具使用的入口。
3. `browser`：浏览器环境的替代入口或映射。
4. `exports`：更现代、更严格的导出控制方式。

### npm workspaces 解决了什么问题？

npm workspaces 用于在 monorepo 场景下统一管理多个包。

它主要解决：

1. 多包项目依赖管理复杂。
2. 本地包联调不方便。
3. 重复安装依赖浪费空间。

### npm publish 时 npm script 的生命周期是什么？

发布时 npm 会按一定顺序执行一些生命周期脚本。

面试里不必死背所有钩子，但要知道：

1. 发布前可以做构建、校验。
2. 发布后可以做收尾工作。
3. 常见流程会围绕 `prepublishOnly`、`prepare`、`postpublish` 等脚本展开。

### 前端项目每次 npm install 之后需要执行一些处理工作，应该怎么办？

通常可以通过 npm scripts 生命周期或安装后脚本来做。

比如：

1. 执行代码生成。
2. 补充本地配置。
3. 打 patch。
4. 初始化 husky 等工具。

但要注意：安装后脚本能力很强，也要评估安全性和透明度。

### 如何修复某个 npm 包的紧急 bug？

常见方案：

1. 升级到官方修复版本。
2. 临时 fork 并引用自定义版本。
3. 使用 `patch-package` 对 `node_modules` 中的包打补丁。

工程上最常见的临时方案是 `patch-package`。

### 简述 npm cache。

npm cache 是 npm 的本地缓存机制，用于减少重复下载依赖、提升安装速度。

它的意义：

1. 加快安装。
2. 降低网络依赖。
3. 提高重复构建效率。

## 六、模块规范、编译与 AST

### 前端打包时 CJS、ESM、UMD 模块有什么不同？

1. CommonJS 主要用于 Node 生态，运行时加载。
2. ESM 是官方模块标准，支持静态分析。
3. UMD 是兼容多种环境的通用模块格式。

现代前端工程里，ESM 最适合做构建优化和 Tree Shaking。

### 在 Babel 编译为低版本 ES 时，为什么能编译可选链这类语法，但无法直接编译 API？

因为 Babel 更擅长处理“语法转换”，而不是自动补齐运行时环境能力。

1. 可选链属于语法，可以转译成旧语法。
2. `Promise`、`Map` 这类 API 属于运行时能力，需要 polyfill。

所以语法降级和 API 兼容是两类问题。

### 什么是 browserslist？有什么意义？

`browserslist` 用来声明项目需要兼容的浏览器范围。

它的意义：

1. 指导 Babel 转译范围。
2. 指导 Autoprefixer 补前缀。
3. 避免无意义地兼容过多旧环境。
4. 让构建目标更明确。

### 什么是 AST？它有哪些应用？

AST 即抽象语法树，是把代码解析后的树形结构表示。

常见应用：

1. 编译器实现。
2. Babel 转译。
3. ESLint 规则分析。
4. 代码自动修改。
5. 依赖收集和静态分析。

## 七、Node 版本、环境与日常工具

### 如何查看项目应采用的 Node 版本？如何为项目指定 Node 版本号？

常见做法：

1. 查看 `package.json` 里的 `engines` 字段。
2. 查看 `.nvmrc` 或其他版本管理配置。

指定版本常见方式：

1. `.nvmrc`

### 刚刚启动了一个服务，如何知道这个服务对应的端口号是多少？

常见方法：

1. 看启动日志。
2. 查看项目配置文件或环境变量。
3. 使用系统命令查看端口占用。
4. 查看服务进程的启动参数。

### 如何查看 `node_modules` 某个文件夹的体积有多大？

常见方式：

1. 用系统命令查看目录大小。
2. 使用包分析工具。
3. 在打包层面用 bundle 分析工具看实际进入产物的体积。

### 如何查看当前 Node 版本及相关环境信息？

常见方式：

1. 查看 Node 版本。
2. 查看 npm 版本。
3. 查看运行时信息和平台信息。
4. 查看当前进程或环境变量中的版本配置。

## 八、性能分析与 Web 指标

### 什么是浏览器的关键渲染路径？

关键渲染路径指浏览器把 HTML、CSS、JS 转换为可见页面的关键步骤链路。

核心过程包括：

1. 解析 HTML。
2. 解析 CSS。
3. 构建 DOM 和 CSSOM。
4. 生成渲染树。
5. 布局和绘制。

优化关键渲染路径，本质上就是让首屏内容更快出现。

### 你使用过哪些前端性能分析工具？

常见工具有：

1. Chrome DevTools Performance
2. Chrome DevTools Memory
3. Lighthouse
4. Webpack Bundle Analyzer
5. Source Map 辅助分析
6. 线上监控平台

### Core Web Vitals 是什么？有哪些指标？

Core Web Vitals 是 Google 提出的核心用户体验指标。

常见核心指标有：

1. LCP：最大内容绘制，关注加载体验。
2. INP 或历史上的 FID：关注交互响应。
3. CLS：累计布局偏移，关注视觉稳定性。

面试时知道这三个指标分别衡量什么就够了。

## 九、权限、安全与资源治理

### 权限设计中的 RBAC 是指什么？

RBAC 指 Role-Based Access Control，即基于角色的访问控制。

它的核心思路是：

1. 给用户分配角色。
2. 给角色分配权限。
3. 用户通过角色获得权限。

它比直接给用户逐个配置权限更适合中大型系统。

### 什么是 Open Graph 协议？用来做什么？

Open Graph 是一套页面元信息协议，常用于社交平台分享页面时生成卡片预览。

它主要用于定义：

1. 标题
2. 描述
3. 图片
4. 页面类型
5. 链接地址

### 图片防盗链原理是什么？

图片防盗链的核心是：服务器通过请求头信息判断资源是否来自允许的来源。

最常见的是检查：

1. `Referer`
2. 有时也会结合签名、鉴权、时间戳等方式

## 十、部署、发布与交付

### 什么是服务器端渲染（SSR）？

SSR 指页面 HTML 先在服务端生成，再返回给浏览器。

它的价值：

1. 提升首屏速度。
2. 改善 SEO。

代价是：

1. 服务端压力增加。
2. 开发复杂度上升。

### 前端如何对分支环境进行部署？

常见思路是为不同分支绑定不同环境配置和部署目标。

比如：

1. 开发分支部署到测试环境。
2. 预发布分支部署到 staging。
3. 主分支部署到生产环境。

关键点包括：

1. 环境变量隔离。
2. 构建配置区分。
3. 自动化流水线控制。

### 如何使用 Docker 部署前端？

常见流程：

1. 在构建阶段生成静态资源。
2. 用 Nginx 或其他轻量 Web 服务承载静态文件。
3. 编写 Dockerfile 构建镜像。
4. 部署到目标环境运行。

现代前端部署里，常见做法是多阶段构建：前面打包，后面仅保留运行所需文件。

### 什么是 CI/CD？

CI/CD 指持续集成和持续交付/部署。

它的目标：

1. 自动化构建。
2. 自动化测试。
3. 自动化发布。
4. 降低人工操作风险。

常见工具有：

1. GitHub Actions
2. GitLab CI
3. Jenkins
4. 云厂商流水线服务

### 如何对 npm package 进行发包？

常见流程：

1. 确认包名和版本。
2. 配置好入口、类型、导出字段。
3. 构建产物。
4. 登录 npm 仓库。
5. 执行发布。
6. 必要时加 tag 管理版本通道。

### 在项目中，如何平滑升级 npm 包？

常见思路：

1. 先确认升级范围和变更日志。
2. 分批升级而不是一次性全升。
3. 优先升级低风险依赖。
4. 配合自动化测试和构建验证。
5. 对核心依赖做灰度或分支验证。

一句话总结：升级依赖不是“直接点更新”，而是一个有验证和回滚意识的工程动作。

### 在 nginx 中如何配置负载均衡？

Nginx 负载均衡本质上是把请求分发到多台后端服务器。

常见策略有：

1. 轮询
2. 加权轮询
3. IP 哈希
4. 最少连接

前端工程里虽然不一定直接写这类配置，但要知道它常用于多实例部署和网关转发。
