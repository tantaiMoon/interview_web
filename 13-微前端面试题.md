# 微前端面试题

## 一、微前端基础认知

### 什么是微前端？

微前端是一种将前端应用按业务域拆分成多个可独立开发、独立部署、独立运行子应用的架构思想。

它借鉴了微服务的拆分理念，但作用对象是前端应用。

核心目标通常有：

1. 降低单体前端项目的复杂度。
2. 提升团队并行开发效率。
3. 支持不同技术栈共存。
4. 实现子应用独立发布，减少相互影响。

一句话总结：微前端不是某个框架，而是一种前端架构方案。

### 微前端解决了什么问题？

微前端主要解决的是大型前端项目在组织和演进上的问题。

常见问题包括：

1. 单体项目过大，维护成本高。
2. 多团队协作时发布耦合严重。
3. 技术栈升级困难。
4. 不同业务线节奏不同，彼此影响大。
5. 旧系统和新系统难以平滑共存。

微前端的价值不在“把项目拆小”本身，而在于更适合复杂组织和复杂业务的演进模式。

### 微前端的优缺点是什么？

优点：

1. 子应用可以独立开发、测试、部署。
2. 多团队并行协作更清晰。
3. 支持技术栈渐进升级。
4. 有利于系统逐步重构。

缺点：

1. 架构复杂度明显上升。
2. 子应用间通信、路由、样式隔离处理更复杂。
3. 公共依赖治理难度更高。
4. 调试、监控、部署链路更复杂。
5. 首屏性能和运行时开销需要额外关注。

## 二、架构设计与拆分思路

### 什么样的场景适合使用微前端？

适合微前端的场景通常有：

1. 大型中后台系统。
2. 多业务线共用统一壳应用。
3. 多团队长期协作的项目。
4. 需要旧系统和新系统并存演进的场景。
5. 业务模块边界清晰、可独立交付的系统。

如果项目本身很小、团队很少、发布频率不高，直接上微前端通常是过度设计。

### 微前端拆分的原则是什么？

拆分时常见原则有：

1. 按业务域拆分，而不是纯技术层面拆分。
2. 子应用边界要清晰。
3. 每个子应用要尽量高内聚、低耦合。
4. 公共能力尽量抽到基础层，而不是互相直接依赖。
5. 路由、权限、菜单、监控等横切能力要统一治理。

面试时可以强调：拆分不是越细越好，拆得过细会引入更高治理成本。

### 微前端和微服务有什么关系和区别？

两者思想相似，都是“按边界拆分”，但作用层不同。

1. 微服务主要作用在后端服务层。
2. 微前端主要作用在前端应用层。

它们的共同点：

1. 都强调独立开发、独立部署。
2. 都需要统一治理。
3. 都关注边界划分和协作效率。

区别在于：

1. 微服务解决服务拆分问题。
2. 微前端解决前端应用拆分问题。

## 三、常见实现方案

### 微前端常见实现方案有哪些？

常见实现方案有：

1. 路由分发式。
2. iframe 嵌入式。
3. Web Components。
4. JavaScript 运行时组合。
5. 基于模块联邦的组合方式。

现代项目中更常见的是：

1. qiankun / single-spa 一类运行时方案。
2. Module Federation 一类模块共享方案。

### iframe 方案在微前端中有什么优缺点？

优点：

1. 天然隔离性强。
2. 样式和 JS 不容易互相污染。
3. 不同技术栈兼容简单。

缺点：

1. 通信复杂。
2. 用户体验较差，路由和历史记录处理麻烦。
3. SEO 和性能较差。
4. 主子应用样式与交互整合感弱。

所以 iframe 更适合强隔离场景，但不适合大多数强调一体化体验的系统。

### 什么是 Web Components 方案？

Web Components 是浏览器原生组件化能力，包括自定义元素、Shadow DOM 和模板机制。

在微前端中，它可以作为封装子应用或组件边界的一种方式。

优点：

1. 标准化能力。
2. 较好的样式隔离。
3. 跨框架集成能力较强。

缺点：

1. 工程复杂度较高。
2. 生态习惯与主流前端框架并不完全一致。
3. 大型业务应用中仍需补足更多治理能力。

### 说说你对 single-spa 的理解。

single-spa 是一个微前端编排框架，核心作用是管理多个子应用的注册、加载、挂载和卸载。

它的核心思想：

1. 主应用负责路由和生命周期调度。
2. 子应用在匹配的路由下被加载和挂载。
3. 子应用离开后可以被卸载。

一句话理解：single-spa 更偏“微前端调度内核”。

### 说说你对 qiankun 的理解。

qiankun 可以理解为基于 single-spa 思想进一步封装的微前端框架。

它在工程上补充了很多能力，比如：

1. 更易用的应用注册。
2. 样式隔离。
3. JS 沙箱。
4. 预加载。
5. 生命周期治理。

所以很多团队会说：single-spa 更偏底层，qiankun 更偏工程可落地方案。

### 什么是 Module Federation？适合什么场景？

Module Federation 是 Webpack 5 提供的一种模块联邦机制，允许不同构建产物在运行时共享模块。

它的特点：

1. 可以动态加载远程模块。
2. 可以共享公共依赖。
3. 更适合模块级复用，而不只是页面级嵌入。

适用场景：

1. 多应用共享组件或页面模块。
2. 多团队协作下的运行时模块集成。
3. 需要减少重复打包依赖的场景。

## 四、运行机制与核心问题

### 微前端的核心运行机制是什么？

无论具体方案如何，微前端通常都围绕几个核心动作：

1. 子应用注册。
2. 路由匹配。
3. 资源加载。
4. 生命周期触发。
5. 挂载与卸载。
6. 通信与状态协同。
7. 隔离与共享治理。

所以微前端的关键不是“能不能嵌进去”，而是“能不能长期稳定运行和演进”。

### 微前端中的生命周期一般有哪些？

在很多微前端框架里，子应用通常会暴露类似生命周期：

1. `bootstrap`
2. `mount`
3. `unmount`
4. 有些方案还会支持 `update`

含义大致是：

1. 初始化。
2. 挂载到页面。
3. 从页面卸载。
4. 更新运行状态。

### 微前端中的 JS 沙箱是什么？为什么需要它？

JS 沙箱的目标是限制子应用对全局环境的污染。

因为多个子应用运行在同一个页面中，如果都直接修改 `window`、全局变量或原型链，很容易互相影响。

沙箱的价值：

1. 隔离全局变量污染。
2. 避免子应用相互干扰。
3. 在卸载时恢复环境。

### 微前端中的样式隔离怎么做？

常见方式有：

1. 命名规范约束。
2. CSS Modules。
3. Shadow DOM。
4. 运行时样式作用域限制。
5. BEM 或命名空间前缀。

面试时要说明：样式隔离没有绝对万能方案，通常是多种手段组合使用。

### 微前端中的依赖共享怎么做？

常见思路：

1. 主应用统一注入公共依赖。
2. 使用 externals。
3. 使用 Module Federation 的 shared 能力。
4. 对 React、Vue、路由库等核心依赖做统一版本治理。

如果依赖共享处理不好，常见问题包括：

1. 多份框架重复加载。
2. 版本冲突。
3. 包体积增大。
4. 运行时行为不一致。

### 微前端中的路由一般怎么设计？

常见思路：

1. 主应用负责一级路由分发。
2. 子应用负责自身内部路由。
3. 路由前缀通常按业务域划分。
4. 刷新、回退、深链接都要统一考虑。

一句话总结：主路由负责“进哪个子应用”，子路由负责“子应用内部怎么走”。

## 五、子应用通信与状态管理

### 微前端中的通信方式有哪些？

常见方式：

1. Props 传参。
2. 全局事件总线。
3. 自定义事件。
4. URL / query / hash。
5. 共享状态仓库。
6. postMessage。
7. 框架提供的通信 API。

选型原则通常是：

1. 简单场景优先轻量通信。
2. 复杂全局状态才考虑统一状态中心。

### 微前端中的全局状态怎么管理？

常见方式有：

1. 主应用维护全局状态。
2. 子应用通过订阅和派发共享数据。
3. 使用统一状态容器。
4. 把真正的业务数据状态尽量留在各自子应用内部，只共享必要的全局状态。

共享状态通常适合：

1. 用户信息。
2. 权限信息。
3. 主题配置。
4. 全局菜单和布局状态。

### 微前端通信时需要注意什么问题？

常见注意点：

1. 不要过度共享状态。
2. 通信协议要清晰。
3. 子应用不要强依赖彼此内部实现。
4. 事件监听要及时解绑。
5. 要考虑卸载后的状态清理。

## 六、性能、监控与工程落地

### 微前端会带来哪些性能问题？

常见问题：

1. 首屏加载资源变多。
2. 公共依赖重复加载。
3. 子应用切换时资源请求增加。
4. 沙箱和隔离机制带来运行时开销。
5. 多应用并存时内存占用更高。

所以微前端不是天然更快，而是通常要通过治理把性能问题控制住。

### 微前端项目怎么做性能优化？

常见做法：

1. 子应用按需加载。
2. 预加载高频应用。
3. 公共依赖共享。
4. 资源缓存和 CDN。
5. 减少重复打包。
6. 控制子应用粒度，不要拆得过碎。
7. 对低频应用延迟加载。

### 微前端项目怎么做监控和错误治理？

常见思路：

1. 主应用统一接入监控。
2. 子应用补充自身错误上报。
3. 埋点里要带子应用标识。
4. 区分主应用错误和子应用错误。
5. 路由切换、资源加载失败、运行时异常都要单独记录。

### 微前端项目的部署模式通常是什么样的？

常见模式：

1. 主应用独立部署。
2. 各子应用独立部署。
3. 主应用通过配置中心或注册表获取子应用地址。
4. 子应用可独立发版，不一定需要主应用重新上线。

这也是微前端的重要价值之一：解耦发布节奏。

## 七、工程实践与高频综合题

### 如果让你设计一个微前端架构，你会怎么考虑？

比较完整的回答可以按这个顺序：

1. 先判断是否真的需要微前端。
2. 再划分业务边界和子应用粒度。
3. 确定主应用职责：路由、权限、菜单、监控、公共能力。
4. 确定子应用职责：独立业务闭环。
5. 设计通信、样式隔离、依赖共享方案。
6. 最后考虑发布、监控、回滚和性能优化。

### 微前端项目最容易踩的坑有哪些？

常见坑有：

1. 拆分边界不清晰。
2. 子应用之间耦合过重。
3. 公共依赖版本冲突。
4. 样式和全局变量污染。
5. 路由和历史记录处理混乱。
6. 发布链路过于复杂。
7. 盲目追求技术栈共存，增加治理成本。

### 微前端不是银弹，这句话怎么理解？

这句话的意思是：微前端解决的是特定规模和组织协作下的问题，但它也会引入新的复杂度。

如果项目规模、团队规模、演进压力并没有大到那个程度，直接使用普通单体前端架构反而更简单、更高效。

所以正确的问题不是“要不要上微前端”，而是“当前问题值不值得用微前端来解决”。

## 附：常见代码实现

### 1. qiankun 注册子应用示例

```javascript
import { registerMicroApps, start } from 'qiankun'

registerMicroApps([
  {
    name: 'sub-app',
    entry: '//localhost:7100',
    container: '#subapp-container',
    activeRule: '/sub-app'
  }
])

start()
```

### 2. 子应用生命周期示例

```javascript
export async function bootstrap() {}
export async function mount(props) {
  console.log('mount', props)
}
export async function unmount() {}
```

### 3. 微前端通信示例

```javascript
const bus = new EventTarget()
bus.addEventListener('change-user', e => console.log(e.detail))
bus.dispatchEvent(new CustomEvent('change-user', { detail: { id: 1 } }))
```
