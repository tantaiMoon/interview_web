## 一、基础类型与语言机制

### 说说 JavaScript 中的数据类型？它们有什么区别？

```javascript
const num = 1
const str = 'hello'
const bool = true
const und = undefined
const nul = null
const sym = Symbol('id')
const big = 10n
const obj = { name: 'Tom' }

console.log(typeof num)   // number
console.log(typeof str)   // string
console.log(typeof bool)  // boolean
console.log(typeof und)   // undefined
console.log(typeof nul)   // object
console.log(typeof sym)   // symbol
console.log(typeof big)   // bigint
console.log(typeof obj)   // object
```


JavaScript 的数据类型通常分为两大类：

1. 基本数据类型：`Number`、`String`、`Boolean`、`Undefined`、`Null`、`Symbol`、`BigInt`
2. 引用数据类型：`Object`，以及基于对象派生出来的 `Array`、`Function`、`Date`、`RegExp` 等。

它们的核心区别：

1. 基本类型一般按值存储，保存在栈中或栈上的标识里。
2. 引用类型保存的是对象地址，实际内容通常在堆中。
3. 基本类型复制时复制的是值本身；引用类型复制时复制的是地址。

面试时可以补一句：`typeof null === 'object'` 是历史遗留问题，不代表 `null` 真的是对象。

### typeof 与 instanceof 的区别是什么？

```javascript
console.log(typeof 1)               // number
console.log(typeof [])              // object
console.log(typeof null)            // object
console.log([] instanceof Array)    // true
console.log({} instanceof Object)   // true
console.log(new Date() instanceof Date) // true
```


两者都能做类型判断，但侧重点不同。

1. `typeof` 更适合判断基本类型，比如 `number`、`string`、`boolean`、`undefined`、`function`。
2. `instanceof` 用来判断某个对象是否在某个构造函数的原型链上，更适合判断引用类型。

典型问题：

1. `typeof null` 返回 `object`，不准确。
2. `typeof []` 和 `typeof {}` 都是 `object`，区分不了数组和对象。
3. `instanceof` 无法准确判断基本类型字面量。

所以更完整的判断方案通常是：

1. 基本类型优先用 `typeof`
2. 引用类型可用 `instanceof`
3. 更精确时使用 `Object.prototype.toString.call()`

### 谈谈 JavaScript 中的类型转换机制。

```javascript
console.log(Number('123'))      // 123
console.log(String(123))        // '123'
console.log(Boolean(''))        // false
console.log('5' - 1)            // 4
console.log('5' + 1)            // '51'
console.log(!!0)                // false
console.log(!!'hello')          // true
```


JavaScript 存在隐式类型转换和显式类型转换。

显式转换常见方式：

1. `Number()`
2. `String()`
3. `Boolean()`
4. `parseInt()`、`parseFloat()`

隐式转换常见场景：

1. `==` 比较
2. 字符串拼接
3. 条件判断
4. 算术运算

一些常见规则：

1. 转布尔值时，`0`、`''`、`null`、`undefined`、`NaN` 会变成 `false`。
2. 字符串和数字运算时，很多场景会先转成数字。
3. 对象参与运算时，会先尝试 `valueOf`，再尝试 `toString`。

面试里要强调：JavaScript 的坑很多来自隐式类型转换，因此实际开发中应尽量控制隐式转换的使用。

### == 和 === 的区别，分别在什么情况下使用？

```javascript
console.log(1 == '1')     // true
console.log(1 === '1')    // false
console.log(null == undefined)  // true
console.log(null === undefined) // false
```


1. `==` 会先做类型转换，再比较值是否相等。
2. `===` 不会做类型转换，要求类型和值都相等。

开发中通常优先使用 `===`，因为行为更明确、可预测性更高。

`==` 主要的问题在于它的隐式转换规则复杂，容易出现非预期结果，比如：

1. `'' == 0` 为 `true`
2. `null == undefined` 为 `true`

所以面试时最好回答：除非明确知道转换规则并且有意利用，否则应该优先使用 `===`。

### 深拷贝和浅拷贝的区别是什么？如何实现一个深拷贝？

```javascript
function deepClone(obj, map = new WeakMap()) {
  if (obj === null || typeof obj !== 'object') return obj
  if (map.has(obj)) return map.get(obj)
  const result = Array.isArray(obj) ? [] : {}
  map.set(obj, result)
  for (const key in obj) {
    if (Object.prototype.hasOwnProperty.call(obj, key)) {
      result[key] = deepClone(obj[key], map)
    }
  }
  return result
}
```


浅拷贝只复制对象的第一层属性，如果属性值还是对象，那么复制的仍然是引用地址。

深拷贝会递归复制对象的每一层，最终得到一份完全独立的新数据。

常见浅拷贝方式：

1. `Object.assign()`
2. 展开运算符 `...`
3. 数组的 `slice()`、`concat()`

常见深拷贝方式：

1. `JSON.parse(JSON.stringify(obj))`
2. 手写递归拷贝
3. `structuredClone()`
4. 使用第三方库，比如 `lodash.cloneDeep`

如果面试官追问缺点，要补充：

1. `JSON` 方式会丢失 `undefined`、函数、`Symbol`、`Date`、`RegExp` 等特殊值。
2. 手写深拷贝要处理循环引用、原型链、特殊对象类型。

### 不使用递归如何实现一个深克隆?

**根本原因**

递归深拷贝的真正风险是**调用栈溢出**:JS 每个函数调用占一帧栈空间,V8 默认栈深度约 1 万左右,遇到深度嵌套对象(如线性链表 `a.next.next.next...`)就会 `RangeError: Maximum call stack size exceeded`。改用**显式栈/队列 + 循环**,把"递归调用栈"变成"堆里的一个数组",栈空间瓶颈就消失了——这就是把递归改成迭代的通用思路。

**具体机制**

1. **核心思路**:用一个数组(栈)或队列保存"待拷贝任务",每个任务是 `{ source, target, key }`,主循环不断从中取出处理。
2. **循环引用处理**:用 `WeakMap` 记录 `源对象 → 克隆对象` 的映射,遇到已访问过的对象直接复用,既防死循环又省内存。
3. **拷贝顺序**:用栈得到 DFS、用队列得到 BFS,业务上结果一致——任选其一。
4. **特殊类型识别**:在循环里对 `Date`、`RegExp`、`Map`、`Set` 等做分支处理,普通对象/数组才进栈继续展开。

**对比表**

| 维度 | 递归实现 | 迭代实现 |
|------|---------|---------|
| 嵌套深度上限 | 受调用栈(约 1 万) | 受堆内存(可达千万级) |
| 实现复杂度 | 简洁,易读 | 多一层任务结构,稍繁 |
| 性能 | 一般场景相当或略快 | 极深结构下显著优势 |
| 调试 | 栈帧清晰 | 需打印任务队列 |
| 适用场景 | 普通业务对象 | 链表、树、AST、用户输入数据 |

**代码**

```javascript
function deepCloneIterative(source) {
  // 基础类型直接返回
  if (source === null || typeof source !== 'object') return source

  // 1. 先克隆"外壳",作为根
  const root = Array.isArray(source) ? [] : {}
  const map = new WeakMap()           // 解决循环引用
  map.set(source, root)

  // 2. 用栈代替递归
  const stack = [{ src: source, target: root }]

  while (stack.length) {
    const { src, target } = stack.pop()

    // 用 Reflect.ownKeys 同时拿到 string 和 symbol 键
    for (const key of Reflect.ownKeys(src)) {
      const value = src[key]

      if (value !== null && typeof value === 'object') {
        // 循环引用命中:直接指向已有克隆
        if (map.has(value)) {
          target[key] = map.get(value)
          continue
        }

        // 特殊对象快速路径
        if (value instanceof Date)   { target[key] = new Date(value.getTime()); continue }
        if (value instanceof RegExp) { target[key] = new RegExp(value.source, value.flags); continue }

        // 普通对象/数组:先建空壳,入栈待处理
        const clone = Array.isArray(value) ? [] : {}
        target[key] = clone
        map.set(value, clone)
        stack.push({ src: value, target: clone })
      } else {
        // 基础类型 + 函数(函数通常按引用拷)
        target[key] = value
      }
    }
  }

  return root
}

// 测试:超深嵌套(递归会爆栈,迭代正常)
let deep = {}
let cur = deep
for (let i = 0; i < 100000; i++) { cur.next = {}; cur = cur.next }
const cloned = deepCloneIterative(deep)
console.log('OK')

// 循环引用
const a = { name: 'a' }; a.self = a
const b = deepCloneIterative(a)
console.log(b.self === b)             // true
```

**实操建议**

1. **绝大多数业务对象用 `structuredClone(obj)`**,浏览器和 Node 17+ 内置,正确处理 Date/RegExp/Map/Set/循环引用,且天然非递归实现,**首选**。
2. 真要面试手写,迭代版本展示三个亮点即可:**显式栈、`WeakMap` 解循环、特殊类型分支**——这是面试官想听的全部。
3. 函数、Symbol、`#` 私有字段、`getter/setter`、`Proxy` 包装的对象,任何深拷贝方案都不能完美还原,业务里要避免依赖。
4. 性能极端场景(大对象高频拷贝),与其优化深拷贝,不如改成**不可变数据 + 结构共享**(Immer / Immutable.js),从源头避免拷贝。

### JavaScript 中参数是按值传递还是按引用传递？

```javascript
function change(a, b) {
  a = 20
  b.name = 'Jerry'
}

let x = 10
let y = { name: 'Tom' }
change(x, y)
console.log(x)       // 10
console.log(y.name)  // Jerry
```


更准确的说法是：JavaScript 中函数参数一律按值传递。

区别在于传递的“值”是什么：

1. 基本类型传递的是值本身。
2. 引用类型传递的是对象地址这个值。

所以很多人会误以为对象是“按引用传递”，其实不是。对象参数在函数里如果修改对象内部属性，外部会感知到；但如果在函数内部把参数重新赋值为另一个对象，不会影响外部变量本身。

面试时最好表述为：JavaScript 没有严格意义上的“按引用传递”，只有按值传递，只是对象的值恰好是地址。

### 什么是类数组对象？`arguments` 有什么特点？

```javascript
function demo() {
  console.log(arguments.length)
  console.log(arguments[0])

  console.log(arr.map(x => x * 2))
```

类数组对象指的是“长得像数组，但并不是数组”的对象，通常具有：

### 说说 var、let、const 之间的区别。

```javascript
var a = 1
let b = 2
const c = 3

if (true) {
  var a = 10
  let b = 20
  console.log(a, b) // 10 20
}
console.log(a) // 10
console.log(b) // 2
```


三者都能声明变量，但差异明显。

1. `var` 存在变量提升，且没有块级作用域。
2. `let` 有块级作用域，存在暂时性死区，不允许重复声明。
3. `const` 也有块级作用域，声明后必须初始化，且不能重新赋值。

补充说明：

1. `const` 保证的是绑定地址不变，不代表对象内容绝对不可修改。
2. 在现代开发中，优先使用 `const`，需要重新赋值时再用 `let`。

### JavaScript 为什么会有变量提升?

**根本原因**

JS 引擎执行代码分两步:**先编译,再执行**。编译阶段会扫描当前作用域,把 `var` 声明和 `function` 声明**提前注册到词法环境**中(分配内存、给默认值),执行阶段才一行行跑代码。这种"声明在编译期处理、赋值留到执行期"的机制,在视觉上就表现为"变量好像被提到了顶部",但实际上**变量从未真的被移动**,只是引擎在编译期已经知道它存在。

**具体机制**

1. **`var` 声明**:编译期注册,默认值 `undefined`。访问它得到 `undefined`,不报错。
2. **`function` 声明**:编译期连函数体一起提升,所以提前调用也能跑。
3. **`function` 表达式**:右边的函数对象不会被提升,只有左边的 `var` 名字被提升。
4. **`let` / `const` / `class`**:也会被引擎记录("半提升"),但**进入暂时性死区(TDZ)**,声明语句执行前访问会抛 `ReferenceError`,不是 `undefined`。
5. **同名提升优先级**:函数声明 > 变量声明;后定义的同名函数会覆盖前面的。

**对比表**

| 声明方式 | 是否提升 | 提升后初值 | 提前访问的结果 |
|---------|---------|----------|--------------|
| `var` | ✅ | `undefined` | `undefined` |
| `function` 声明 | ✅(连函数体) | 函数本身 | 可正常调用 |
| `function` 表达式 | 仅变量名 | `undefined` | `TypeError`(不是函数) |
| `let` / `const` | ✅(进入 TDZ) | 无 | `ReferenceError` |
| `class` | ✅(进入 TDZ) | 无 | `ReferenceError` |

**代码**

```javascript
console.log(a)        // undefined  —— var 被提升,值为 undefined
var a = 1

console.log(b)        // ReferenceError —— let 在 TDZ
let b = 1

foo()                 // ✅ 正常调用 —— 函数声明整体提升
function foo() { console.log('foo') }

bar()                 // TypeError: bar is not a function —— 只提升 var
var bar = function () { console.log('bar') }
```

**实操建议**

1. 全部用 `let` / `const`,变量提升和 TDZ 带来的"看似无关代码却报错"问题基本消失。
2. 函数定义优先用**函数声明**(顶层)或 `const fn = () => {}`(块内),不要混用让代码顺序难懂。
3. 面试被追问 TDZ 时,关键是说清"`let/const` 也提升,只是访问会抛错"——很多人误以为 TDZ 等于"不提升"。

### const 声明的变量值想要改变该怎么做?

**根本原因**

`const` 在 JS 里只锁定**变量到值的绑定**,不锁定值本身的内容。也就是说,`const obj = {}` 保证 `obj` 这个标识符不能再指向新对象,但**对象内部的属性是可以随意改的**。要想真正改变 `const` 变量的可观察值,只能从"内容可变"这条路走。

**具体机制**

1. **`const` 的限制**:不能 `=` 重新赋值;但读 / 写对象属性、调用对象方法、push 数组、Map.set 等**完全不受限制**。
2. **三层"不可变"层级**:
   - 第一层:`const`(绑定不可变,内容可变)。
   - 第二层:`Object.freeze()`(冻结第一层属性,深层仍可变)。
   - 第三层:递归深冻结 / Immutable.js / Immer(任何层级都不能改)。
3. **想"换值"的场景**:重新声明一个新 `const`,或者改用 `let`。
4. **特殊情况**:`Object.freeze` 在严格模式下改属性会抛错,非严格模式下静默失败。

**代码**

```javascript
const obj = { count: 0 }

// ❌ 不行:重新赋值给 obj 本身
obj = { count: 1 }              // TypeError: Assignment to constant variable

// ✅ 改属性可以
obj.count = 1
obj.name = 'Tom'

// ✅ 用 Object.assign 批量改
Object.assign(obj, { count: 2, name: 'Jerry' })

// ✅ 数组同理
const arr = [1, 2, 3]
arr.push(4)                     // OK,arr 还是同一个数组引用
arr[0] = 999                    // OK

// 真要"不可变",用 Object.freeze 或解构产生新对象
const frozen = Object.freeze({ count: 0 })
frozen.count = 1                // 严格模式抛错,非严格模式静默失败

// 函数式风格:不改原对象,生成新对象(配 let)
let state = { count: 0 }
state = { ...state, count: state.count + 1 }
```

**对比表**

| 写法 | 绑定可变 | 第一层属性可变 | 深层属性可变 |
|------|--------|-------------|------------|
| `let obj = {}` | ✅ | ✅ | ✅ |
| `const obj = {}` | ❌ | ✅ | ✅ |
| `const obj = Object.freeze({})` | ❌ | ❌ | ✅ |
| 递归深 freeze | ❌ | ❌ | ❌ |

**实操建议**

1. 默认 `const`,只有在确实需要重新赋值时才改 `let`——这是社区共识。
2. 想"语义上不可变"的常量(如配置表、枚举映射),配上 `Object.freeze` 或 `as const`(TS),让误改在开发期就报错。
3. React / Redux 这类"状态不可变"的体系下,改值要靠**生成新对象**(`...state` 或 Immer),不要去改 `const` 内部属性,否则视图不会刷新。
4. 不要用 `Object.freeze` 当深拷贝替代品,它只冻一层。

## 二、作用域、原型与面向对象

### 说说你对作用域链的理解。

**根本原因**

JS 是**词法作用域(静态作用域)** 语言:函数能访问哪些变量,在**写代码时**就由它在源码中的物理位置决定,不是运行时谁调它就用谁的环境。当函数访问一个变量时,需要一条"由内向外的查找路径"——这条路径就是**作用域链**。它的本质是变量查找算法,作用是把"嵌套结构"变成"线性查找表"。

**具体机制**

1. **函数定义时**就保存一个 `[[Environment]]` 内部属性,指向定义所在的词法环境。
2. **函数调用时**创建新的"执行上下文",其作用域链 = `当前词法环境 → 外层词法环境 → ... → 全局环境`。
3. **变量查找**:从链头开始,找不到就沿链向外走,直到全局;还找不到就 `ReferenceError`。
4. **作用域链不可变**:函数定义在哪,链就是哪,**和调用位置无关**(这是和 `this` 最大的区别)。
5. **闭包**:函数即使在原作用域销毁后,只要被外部引用,它的作用域链就不会被回收——这正是闭包能"记住"外层变量的原因。

**对比:作用域链 vs 原型链 vs this 绑定**

| 机制 | 决定时机 | 用于 | 路径方向 |
|------|---------|------|---------|
| 作用域链 | 函数**定义时** | 变量查找 | 内 → 外(词法嵌套) |
| 原型链 | 对象创建时 | 属性查找 | 实例 → 原型 → 原型的原型 |
| this | 函数**调用时** | 决定函数运行时上下文 | 由调用方式决定 |

**代码**

```javascript
const a = 1
function outer() {
  const b = 2
  function inner() {
    const c = 3
    console.log(a, b, c)   // 1 2 3:沿作用域链向外找
  }
  return inner
}

// 即使 outer() 已执行完,inner 的作用域链仍持有 b
const fn = outer()
fn()                      // 闭包让 b 不被回收

// 词法作用域 vs 动态作用域:JS 是词法的
const x = 'global'
function readX() { return x }     // x 在定义时就锁定到 global
function caller() {
  const x = 'local'
  return readX()                  // 仍然返回 'global',不是 'local'
}
console.log(caller())             // 'global'
```

**实操建议**

1. 不要写"靠层层向外找变量"的代码,可读性极差;每个变量尽量在使用它的最小作用域内声明。
2. 闭包持有作用域链 = 持有外层变量,**用完后置 `null` 或解绑**,防止意外内存占用(详见"闭包内存泄漏"一题)。
3. 大量循环里访问外层变量会有微小性能成本,把它**先用 `const local = outer` 缓存到当前作用域**,JIT 优化更好。
4. 模块化(ESM)就是天然的作用域隔离,不要用 IIFE 和 `var` 模拟旧时代的私有作用域。

### 说说你对闭包的理解。

**根本原因**

闭包不是"一种特殊语法",而是**词法作用域 + 函数是一等公民**的自然产物。当一个函数被传到原作用域之外执行时,它仍然持有定义时的作用域链,从而**让外层局部变量"活过"了原本该销毁的时机**。本质上,闭包 = 函数 + 该函数被创建时所处的词法环境的引用。

**具体机制**

1. **变量持有**:内层函数引用外层变量,引擎不能回收外层环境,即使外层函数已 `return`。
2. **每次调用产生新闭包**:每次执行外层函数,都生成一份独立的环境,各闭包之间互不干扰。
3. **共享同一份环境**:同一外层函数返回多个内部函数时,这些函数共享一个外层环境(经典面试题陷阱)。
4. **何时释放**:当所有引用闭包的外部入口都被解除(变量置 `null`、监听解绑、定时器清除),环境才能被 GC。

**闭包的两大作用**

| 作用 | 解决的问题 | 典型场景 |
|------|----------|---------|
| **数据私有化(封装)** | JS 没有 `private`,需要藏数据 | 模块模式、计数器、单例工厂 |
| **状态持久化(记忆)** | 函数调用之间需要"记住"上次的值 | 防抖节流、柯里化、缓存 memo、迭代器 |

衍生场景:函数工厂、回调中的"快照"参数、私有事件总线。

**闭包一定会内存泄漏吗?**

**不会**,只在"该释放却没释放"时才算泄漏。

1. **正常闭包**:用完会被 GC 正常回收,**不算泄漏**(只是生命周期更长)。
2. **真正泄漏的场景**:
   - 闭包被挂到长期存活的对象上(全局变量、单例、`window`),却**永远不被解除**。
   - 闭包内**意外引用了大对象**(如整个 DOM 节点、大数组),拖着它们不释放。
   - **定时器 / 监听器中的闭包**忘记 `clearInterval` / `removeEventListener`。
   - **DOM 已移除但 JS 仍持有引用**(detached DOM)。
3. **历史遗留**:IE 老版本对 DOM + JS 闭包有循环引用问题,现代浏览器已基本解决。

**代码**

```javascript
// ① 数据私有化
function createCounter() {
  let count = 0                  // 外部访问不到
  return {
    inc: () => ++count,
    get: () => count,
  }
}
const c = createCounter()
c.inc(); c.inc()
console.log(c.get())             // 2

// ② 状态持久化(防抖)
function debounce(fn, delay) {
  let timer = null               // 闭包记住 timer
  return (...args) => {
    clearTimeout(timer)
    timer = setTimeout(() => fn(...args), delay)
  }
}

// ③ 共享环境陷阱
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0)   // 3 3 3:三个闭包共享同一个 i
}
for (let j = 0; j < 3; j++) {
  setTimeout(() => console.log(j), 0)   // 0 1 2:每轮 let 产生新作用域
}

// ④ 真实泄漏:DOM 引用挂到全局
let leakedHandler
function bind() {
  const huge = new Array(1e6).fill('x')
  leakedHandler = () => console.log(huge.length)   // huge 永远不释放
}
```

**如何预防内存泄漏**

1. **配对清理**:`addEventListener` 必有对应 `removeEventListener`;`setInterval` 必有对应 `clearInterval`。
2. **解除引用**:闭包用完显式置 `fn = null`,或用 `WeakMap` / `WeakRef` 持有"可回收"的引用。
3. **作用域最小化**:闭包内只引用真正需要的字段,不要把整个大对象捎进去——一个不该被引用的 `event` 对象就能拖死一棵 DOM 树。
4. **组件/页面销毁时统一清理**:Vue 的 `onUnmounted`、React 的 `useEffect` 返回函数,统一收拾监听、定时器、订阅。
5. **DevTools 验证**:Chrome Memory 面板 → Heap snapshot → 搜 `Detached` 关键字,能直接看到没释放的 DOM 闭包。
6. **代码评审重点**:"这个回调谁来解绑?"——大部分泄漏死在"绑了没解"。

**实操建议**

1. 闭包是工具不是负担——**不要为了避免泄漏就不用闭包**,绝大多数闭包是安全的。
2. React 里 `useEffect` 内闭包捕获的是**渲染时的快照**,出现"读到旧值"问题用 `useRef` 或依赖数组解决,不要靠 `setTimeout` 兜底。
3. 长期运行的应用(后台 Tab、Electron、Node 服务)必须做定期内存监控,泄漏不是问题,问题是没人发现。

### JavaScript 中执行上下文和执行栈是什么？

```javascript
function foo() {
  console.log('foo start')
  bar()
  console.log('foo end')
}

function bar() {
  console.log('bar')
}

foo()
```


执行上下文可以理解为代码执行时所需的环境信息，主要包括：

1. 变量对象或词法环境
2. 作用域链
3. `this` 指向

执行栈则是管理执行上下文的栈结构。

代码执行过程通常是：

1. 全局代码先进入全局执行上下文。
2. 函数调用时创建函数执行上下文并压栈。
3. 函数执行完成后弹栈。
4. 最终回到全局上下文。

所以执行栈本质上是 JavaScript 管理函数调用顺序的机制。

### 谈谈 this 对象的理解。

```javascript
const obj = {
  name: 'Tom',
  say() {
    console.log(this.name)
  }
}
obj.say() // Tom

const fn = obj.say
fn() // 非严格模式下通常是 undefined 或 window.name
```


`this` 不是函数定义时决定的，而是函数调用时决定的。

常见绑定规则：

1. 默认绑定：普通函数直接调用，非严格模式下指向 `window`，严格模式下是 `undefined`。
2. 隐式绑定：对象调用方法时，`this` 指向调用它的对象。
3. 显式绑定：通过 `call`、`apply`、`bind` 指定 `this`。
4. `new` 绑定：构造函数调用时，`this` 指向新创建的实例。
5. 箭头函数：没有自己的 `this`，继承外层作用域的 `this`。

面试时通常会强调：判断 `this` 的关键不是函数在哪里定义，而是函数如何调用。

### JavaScript 原型、原型链是什么？有什么特点？

```javascript
function Person(name) {
  this.name = name
}
Person.prototype.sayHi = function () {
  return 'hi ' + this.name
}
const p = new Person('Tom')
console.log(p.sayHi())
console.log(p.__proto__ === Person.prototype) // true
```


JavaScript 是基于原型的语言。每个对象内部都可以关联到另一个对象，这个被关联的对象就是它的原型。

当访问对象属性时，如果对象本身没有，就会去它的原型对象上查找；原型对象没有，就继续向上查找，这条链路就是原型链。

特点：

1. 属性和方法可以复用。
2. 是 JavaScript 实现继承的基础。
3. 查找是逐层向上的，直到 `Object.prototype`。
4. 原型链顶端的原型再往上是 `null`。

### JavaScript 如何实现继承？各方式的优缺点是什么?

**根本原因**

JS 没有 Java 那样的类继承机制,**对象的"继承"本质上是把另一个对象设为自己的原型**(`__proto__` 链)。要做到"子能复用父的属性和方法",就要解决两件事:**①让子实例能拿到父的实例属性(每个实例独有);②让子实例能共享父的原型方法(节省内存)**。社区围绕这两点摸索出了 5 代方案,每代都在修补上一代的缺陷,最终演进出 ES6 的 `class extends`。

**演进与对比表**

| 方式 | 实例属性 | 原型方法复用 | 主要缺陷 |
|------|---------|------------|---------|
| ① 原型链继承 | ❌ 多实例共享 | ✅ | 引用类型属性被所有子实例共享、无法传参 |
| ② 借用构造函数 | ✅ 各自独立 | ❌ 无法继承原型方法 | 方法只能写在构造函数里,无法复用 |
| ③ 组合继承 | ✅ | ✅ | 父构造函数被调用 2 次(浪费 + 多余属性) |
| ④ 原型式继承(`Object.create`) | ❌ 共享 | ✅ | 同①的引用类型问题 |
| ⑤ 寄生式继承 | ❌ | ✅ | 方法不能复用(每次新建) |
| ⑥ **寄生组合继承(最优)** | ✅ | ✅ | 写法相对繁琐 |
| ⑦ **`class extends`(推荐)** | ✅ | ✅ | 本质就是 ⑥ 的语法糖 |

**具体机制 + 代码**

**① 原型链继承**:把父实例作为子的原型。

```javascript
function Parent() { this.colors = ['red'] }
function Child() {}
Child.prototype = new Parent()

const a = new Child(), b = new Child()
a.colors.push('blue')
console.log(b.colors)         // ['red', 'blue']  ⚠️ 引用类型被共享
```

**② 借用构造函数**:在子构造函数里 `Parent.call(this)`。

```javascript
function Parent(name) { this.name = name; this.colors = ['red'] }
function Child(name) { Parent.call(this, name) }   // 借用

Parent.prototype.say = function () {}              // 子实例拿不到这个方法
const c = new Child('Tom')
c.say()                                            // ❌ TypeError
```

**③ 组合继承**:①+② 拼起来。

```javascript
function Parent(name) { this.name = name; this.colors = ['red'] }
Parent.prototype.say = function () { return this.name }

function Child(name, age) {
  Parent.call(this, name)        // 第 1 次调用 Parent —— 拿到实例属性
  this.age = age
}
Child.prototype = new Parent()   // 第 2 次调用 Parent —— 浪费
Child.prototype.constructor = Child
```

**⑥ 寄生组合继承(社区最优)**:用 `Object.create` 替代第 2 次 `new Parent`。

```javascript
function inherit(Child, Parent) {
  Child.prototype = Object.create(Parent.prototype)   // 不调 Parent
  Child.prototype.constructor = Child
}

function Parent(name) { this.name = name }
Parent.prototype.say = function () { return this.name }

function Child(name, age) {
  Parent.call(this, name)
  this.age = age
}
inherit(Child, Parent)
```

**⑦ ES6 `class extends`(推荐写法)**

```javascript
class Animal {
  constructor(name) { this.name = name }
  say() { return this.name }
}
class Dog extends Animal {
  constructor(name, breed) {
    super(name)                  // 必须在 this 之前调
    this.breed = breed
  }
  bark() { return 'wang' }
}
```

**优缺点深度对照**

| 维度 | 原型链 | 借用构造 | 组合 | 寄生组合 | class |
|------|-------|---------|------|---------|-------|
| 父实例属性独立 | ❌ | ✅ | ✅ | ✅ | ✅ |
| 父原型方法复用 | ✅ | ❌ | ✅ | ✅ | ✅ |
| 能向父构造传参 | ❌ | ✅ | ✅ | ✅ | ✅ |
| 父构造调用次数 | 1 | 1 | **2** | 1 | 1 |
| `instanceof` 正确 | ✅ | ❌ | ✅ | ✅ | ✅ |
| `super` 调用父方法 | ❌ | ❌ | ❌ | ❌ | ✅ |
| 是否推荐 | ❌ | ❌ | ❌ | ✅ 老项目 | ✅ 新项目 |

**实操建议**

1. **新项目一律 `class extends`**,语义清晰、`super` 调用方便、TS 友好,本质就是寄生组合的语法糖。
2. 老项目维护遇到原型链相关代码,先认出是哪一代方案,再对照"它解决了什么、还有什么坑没修"——避免乱改。
3. 真业务里**继承用得越来越少**,前端社区共识是"组合优于继承":遇到复用就先想"能不能拆成 hook / 高阶函数 / mixin",再考虑继承。
4. 别在前端业务代码里手写"寄生组合",维护成本太高;真要继承直接 `class`。

### 说说 new 操作符具体干了什么？

```javascript
function myNew(Constructor, ...args) {
  const obj = Object.create(Constructor.prototype)
  const result = Constructor.apply(obj, args)
  return result !== null && (typeof result === 'object' || typeof result === 'function')
    ? result
    : obj
}
```


`new` 的执行过程可以概括为：

1. 创建一个新对象。
2. 把新对象的原型指向构造函数的 `prototype`。
3. 以这个新对象作为 `this` 执行构造函数。
4. 如果构造函数返回的是对象，就返回该对象；否则返回新创建的对象。

所以 `new` 的本质是“创建实例对象并建立原型关联”。

手写一个简化版 `new`，核心实现可以这样写：

```javascript
function myNew(Constructor, ...args) {
  const obj = Object.create(Constructor.prototype)
  const result = Constructor.apply(obj, args)
  return result !== null && (typeof result === 'object' || typeof result === 'function')
    ? result
    : obj
}
```

示例：

```javascript
function Person(name) {
  this.name = name
}

Person.prototype.sayHi = function () {
  return 'hi ' + this.name
}

const p = myNew(Person, 'Tom')
console.log(p.name)      // Tom
console.log(p.sayHi())   // hi Tom
```

### bind、call、apply 的区别是什么？如何实现一个 bind？

```javascript
Function.prototype.myCall = function (context, ...args) {
  context = context == null ? globalThis : Object(context)
  const key = Symbol('fn')
  context[key] = this
  const result = context[key](...args)
  delete context[key]
  return result
}

Function.prototype.myApply = function (context, args = []) {
  context = context == null ? globalThis : Object(context)
  const key = Symbol('fn')
  context[key] = this
  const result = context[key](...args)
  delete context[key]
  return result
}
```


三者本质上都是改变函数执行时的 `this` 指向。

区别：

1. `call`：立即执行，参数逐个传入。
2. `apply`：立即执行，参数以数组形式传入。
3. `bind`：不立即执行，返回一个绑定了 `this` 的新函数。

如果手写 `bind`，核心要点有：

1. 返回一个新函数。
2. 新函数执行时把 `this` 绑定到指定对象。
3. 支持柯里化参数。
4. 处理作为构造函数时的 `new` 场景。

可以顺手把 `call`、`apply`、`bind` 的简化实现一起记住。

1. 手写 `call`

```javascript
Function.prototype.myCall = function (context, ...args) {
  context = context == null ? globalThis : Object(context)
  const key = Symbol('fn')
  context[key] = this
  const result = context[key](...args)
  delete context[key]
  return result
}
```

2. 手写 `apply`

```javascript
Function.prototype.myApply = function (context, args = []) {
  context = context == null ? globalThis : Object(context)
  const key = Symbol('fn')
  context[key] = this
  const result = context[key](...args)
  delete context[key]
  return result
}
```

3. 手写 `bind`

```javascript
Function.prototype.myBind = function (context, ...outerArgs) {
  const self = this

  function boundFn(...innerArgs) {
    const isNew = this instanceof boundFn
    const thisArg = isNew ? this : context
    return self.apply(thisArg, [...outerArgs, ...innerArgs])
  }

  boundFn.prototype = Object.create(self.prototype)
  return boundFn
}
```

示例：

```javascript
function greet(greeting, name) {
  return greeting + ', ' + name
}

console.log(greet.myCall(null, 'Hello', 'Tom'))           // Hello, Tom
console.log(greet.myApply(null, ['Hello', 'Jerry']))      // Hello, Jerry

const sayHello = greet.myBind(null, 'Hello')
console.log(sayHello('Jack'))                             // Hello, Jack
```

## 三、异步、事件与浏览器交互

### 说说 JavaScript 中的事件模型。

```javascript
const box = document.getElementById('box')
box.addEventListener('click', () => console.log('capture'), true)
box.addEventListener('click', () => console.log('bubble'), false)
```


JavaScript 的事件模型通常包含三个阶段：

1. 捕获阶段
2. 目标阶段
3. 冒泡阶段

事件从顶层向目标元素传播是捕获，目标元素处理后再从目标元素向外传播是冒泡。

实际开发里最常见的是冒泡模型，因为它便于做事件代理。

### 解释下什么是事件代理？应用场景是什么？

```javascript
document.getElementById('list').addEventListener('click', e => {
  if (e.target.matches('li')) {
    console.log('clicked', e.target.textContent)
  }
})
```


事件代理指的是不把事件直接绑定到子元素上，而是绑定到它们的父元素或祖先元素上，利用事件冒泡统一处理。

优点：

1. 减少事件绑定数量。
2. 提高性能。
3. 动态新增元素也能自动响应。

典型场景：

1. 列表点击
2. 表格操作按钮
3. 菜单项交互

### 说说你对事件循环的理解。

```javascript
console.log('start')
setTimeout(() => console.log('timeout'))
Promise.resolve().then(() => console.log('promise'))
console.log('end')
// start end promise timeout
```


JavaScript 是单线程的，但需要处理异步任务，所以引入了事件循环机制。

可以把任务分为：

1. 同步任务，进入主线程立即执行。
2. 异步任务，执行时机成熟后进入任务队列。

常见又可分成：

1. 宏任务：`setTimeout`、`setInterval`、I/O、UI 渲染
2. 微任务：`Promise.then`、`MutationObserver`、`queueMicrotask`

执行顺序通常是：

1. 执行当前宏任务
2. 清空当前所有微任务
3. 必要时浏览器渲染
4. 再执行下一个宏任务

面试时要说清楚：`Promise` 的回调一般先于 `setTimeout` 执行，因为微任务优先级更高。

### ajax 原理是什么？如何实现？

```javascript
const xhr = new XMLHttpRequest()
xhr.open('GET', '/api/user', true)
xhr.onreadystatechange = function () {
  if (xhr.readyState === 4 && xhr.status === 200) {

  }
}
xhr.send()
```


Ajax 的本质是在不刷新页面的前提下，与服务器进行异步数据通信。

传统实现依赖 `XMLHttpRequest`：

5. 接收响应并更新页面

所以面试回答可以分层：

1. 原理上是异步 HTTP 通信。
2. 传统实现是 XHR。
3. 现代项目多用 `fetch` / `axios`。

### 什么是防抖和节流？有什么区别？如何实现？

```javascript
function debounce(fn, delay = 300) {
  let timer = null
  return function (...args) {
    clearTimeout(timer)
    timer = setTimeout(() => fn.apply(this, args), delay)
  }
}

function throttle(fn, delay = 300) {
  let last = 0
  return function (...args) {
    const now = Date.now()
    if (now - last >= delay) {
      last = now
      fn.apply(this, args)
    }
  }
}
```


两者都是高频事件优化手段。

1. 防抖：事件连续触发时，只在最后一次触发后经过指定时间才执行。
2. 节流：事件连续触发时，在固定时间间隔内只执行一次。

应用场景：

1. 防抖：搜索联想、窗口 resize、表单校验。
2. 节流：滚动监听、拖拽、鼠标移动。

实现思路：

1. 防抖通常借助定时器，每次触发都清空上一次。
2. 节流通常通过时间戳或定时器控制执行频率。

### JavaScript 中如何实现函数缓存？函数缓存有哪些应用场景？

```javascript
function memo(fn) {
  const cache = new Map()
  return function (...args) {
    const key = JSON.stringify(args)
    if (cache.has(key)) return cache.get(key)
    const result = fn.apply(this, args)
    cache.set(key, result)
    return result
  }
}
```


函数缓存也叫记忆化，核心思想是：相同输入直接返回上次计算结果，避免重复计算。

实现思路：

1. 用对象或 `Map` 记录参数和结果的映射。
2. 每次执行前先查缓存。
3. 命中则直接返回，未命中再计算并存储。

应用场景：

1. 复杂计算结果复用。
2. 重复接口请求的本地去重。
3. 递归优化。
4. 模板编译缓存。

### 观察者模式和发布订阅模式有什么区别？

```javascript

  constructor() { this.list = [] }
  add(fn) { this.list.push(fn) }
  notify(data) { this.list.forEach(fn => fn(data)) }
}
```


这两个模式都和“事件通知”有关，但耦合关系不同。

观察者模式通常是：

1. 观察者直接订阅目标对象。
2. 目标对象状态变化后直接通知观察者。

发布订阅模式通常是：

1. 发布者和订阅者之间通过事件中心或消息中心解耦。

### 说说 JavaScript 数字精度丢失的问题，如何解决？

```javascript

```


JavaScript 中 `Number` 采用 IEEE 754 双精度浮点数表示，因此某些小数计算会出现精度误差，比如 `0.1 + 0.2 !== 0.3`。

解决思路：

1. 对金额等场景转成整数运算，比如“元转分”。
2. 使用 `toFixed` 做展示层格式化，但它不一定适合做精确计算。
3. 使用高精度运算库，比如 `decimal.js`、`big.js`。
4. 超大整数可以考虑 `BigInt`，但它不处理小数。

### 大文件上传如何做断点续传？

```javascript
function createChunks(file, size = 1024 * 1024) {
  const chunks = []
  for (let i = 0; i < file.size; i += size) {
    chunks.push(file.slice(i, i + size))
  }

}
```

4. 中断后重新上传时，只补传未完成分片。
5. 最后服务端合并分片。

1. 文件 hash 校验
2. 秒传
3. 并发上传
4. 上传进度展示

### Web 常见的攻击方式有哪些？如何防御?

**根本原因**

Web 攻击本质上都来自一句话:**"任何来自浏览器的输入都不可信"**。Web 体系是开放的——HTML/JS/CSS 是文本、HTTP 是无状态、Cookie 自动携带、`<script>` 不受同源限制——攻击者只要在某个环节注入"看起来合法但其实有害的内容",就能借浏览器的信任搞事。所以**所有防御的核心都是"不混淆数据和代码、不在错误的地方信任错误的来源"**。

**九种主流攻击 + 防御对照表**

| 攻击 | 一句话原理 | 典型场景 | 核心防御 |
|------|----------|---------|---------|
| **XSS(存储/反射/DOM)** | 用户输入被当代码执行 | 评论区 `<script>`、URL 参数渲染 | 输出转义、CSP、`textContent` |
| **CSRF** | 借受害者已登录 cookie 伪造请求 | 钓鱼站点 `<img src=...转账url>` | SameSite、CSRF Token、关键操作二次验证 |
| **点击劫持** | 用透明 iframe 诱导用户点 | "免费领礼包",底下其实是付款按钮 | `X-Frame-Options: DENY`、`frame-ancestors` |
| **SQL 注入** | 拼接字符串让 SQL 变结构 | `where id='${input}'` | 参数化查询、ORM、最小权限账号 |
| **命令注入** | 后端 shell 拼字符串 | `exec('ping ' + ip)` | 不拼 shell,用安全 API |
| **文件上传** | 上传 `.php` / `.html` 当代码执行 | 头像接口 | 白名单 + 重命名 + 不在 web 根目录 + 内容校验 |
| **SSRF** | 让服务器去访问内网 | 后端代为下载用户给的 URL | 内网 IP 黑名单、白名单域 |
| **中间人(MITM)** | 拦截改 HTTP 流量 | 公共 WiFi | HTTPS + HSTS + 证书钉扎 |
| **依赖投毒/原型污染** | 三方包/`__proto__` 注入 | `lodash` 漏洞、`merge` 类工具 | 锁版本、SCA 扫描、`Object.create(null)` |

**重点攻击详解**

**1. XSS(跨站脚本)**

把 JS 注入到别人浏览器执行,可以偷 cookie、改页面、自动转账。

```javascript
// ❌ 危险:用户输入直接拼到 HTML
el.innerHTML = `<div>${user.name}</div>`
// 输入 <img src=x onerror=alert(document.cookie)> 就执行了

// ✅ 安全:文本接口
el.textContent = user.name

// ✅ 富文本场景用白名单库
import DOMPurify from 'dompurify'
el.innerHTML = DOMPurify.sanitize(html)
```

**纵深防御**

1. 后端入库时不用过度转义(可能丢数据);**输出到哪种上下文,按那种规则转义**(HTML / 属性 / JS / URL 各不同)。
2. Cookie 加 `HttpOnly`,XSS 偷不到。
3. 配 CSP:`Content-Security-Policy: default-src 'self'; script-src 'self'`,杜绝内联脚本和外域脚本。

**2. CSRF(跨站请求伪造)**

用户在 A 站登录后,B 站偷偷构造一个对 A 站的请求,浏览器自动带上 A 站 cookie,A 站误以为是用户本人操作。

```html
<!-- 攻击站点 -->
<img src="https://bank.com/transfer?to=hacker&amount=10000">
```

**防御**

1. **`SameSite=Lax/Strict` Cookie**:浏览器跨站不带 cookie,**目前最有效的一招**(现代浏览器默认 Lax)。
2. **CSRF Token**:服务端发一次性随机 token,前端表单/请求头带上,B 站拿不到。
3. **关键操作二次验证**:支付/改密码要再输密码或短信。
4. 不要用 GET 做"会改数据"的接口。

**3. 点击劫持**

```http
# 响应头加上,禁止被嵌套
X-Frame-Options: DENY
Content-Security-Policy: frame-ancestors 'none'
```

**4. 原型污染(前端 JS 特有)**

```javascript
// ❌ 危险的 merge 实现
function merge(a, b) {
  for (const k in b) a[k] = b[k]   // 可被 __proto__ 污染
}
const evil = JSON.parse('{"__proto__": {"isAdmin": true}}')
merge({}, evil)
console.log({}.isAdmin)            // true,所有对象都中招

// ✅ 防御
Object.create(null)                // 原型为 null,无法被污染
Object.freeze(Object.prototype)    // 冻结 Object.prototype
```

**统一防御清单(前端 + 后端)**

1. **输入校验**:类型、长度、格式;前端方便用户,后端真校验。
2. **输出编码**:按上下文转义;别用 `innerHTML`,用 `textContent`。
3. **Cookie 三件套**:`HttpOnly` + `Secure` + `SameSite`。
4. **CSP**:禁内联、禁外域、禁 `eval`。
5. **HTTPS + HSTS**:全站 HTTPS 是底线。
6. **依赖治理**:`npm audit`、Renovate、锁定版本、内部镜像。
7. **最小权限**:数据库账号、Token scope、API 权限收敛到必要范围。
8. **监控告警**:异常请求频率、敏感接口失败率打点报警。

**实操建议**

1. 安全 80% 在后端,但**前端是第一道门**:URL 参数渲染、富文本、第三方脚本、postMessage 这些地方 100% 是高风险。
2. 写代码时养成两个反射动作:**"输出 HTML 必转义"**、**"GET 不改数据"**——能挡掉一半 XSS 和 CSRF。
3. 项目接入 SCA(`npm audit` / Snyk)、SAST(ESLint security 规则)、CSP report-uri 三件套,把安全做成持续过程,不是上线前临时检查。
4. 面试被追问时记住框架级答案:**"按风险点分层 + 输入输出双侧治理 + 浏览器安全头 + 监控兜底"**。

### 跨域的原因及解决方案

**根本原因**

跨域问题来自浏览器的**同源策略(Same-Origin Policy)**:不同源的页面之间默认**不能读对方的资源**(DOM、Cookie、HTTP 响应),为了防止恶意站点窃取你在其他站点的数据。"同源" = `协议 + 域名 + 端口`三者全相同;**任意一个不同就算跨域**。注意三件事:

1. 同源策略**只针对浏览器**,服务端之间互访没有跨域问题。
2. **请求其实发出去了**(简单请求),只是响应被浏览器拦下不给 JS 读。
3. `<img>`、`<script>`、`<link>` 这些标签**天然允许跨域加载**——这也是 JSONP / CDN / 图片打点的基础。

**哪些算跨域?**

| 当前页面 | 目标地址 | 是否跨域 | 原因 |
|---------|---------|---------|------|
| `http://a.com/page` | `http://a.com/api` | ❌ | 同源 |
| `http://a.com` | `https://a.com` | ✅ | 协议不同 |
| `http://a.com:80` | `http://a.com:8080` | ✅ | 端口不同 |
| `http://a.com` | `http://b.com` | ✅ | 域名不同 |
| `http://a.com` | `http://api.a.com` | ✅ | 子域不同 |

**主流解决方案对比**

| 方案 | 适用 | 是否需要后端配合 | 是否生产推荐 |
|------|------|---------------|------------|
| **CORS** | 接口跨域(主流) | ✅ 需要 | ⭐⭐⭐⭐⭐ |
| **Nginx 反向代理** | 接口跨域(同域分发) | ❌ 前端独立做 | ⭐⭐⭐⭐⭐ |
| **dev-server proxy** | 开发期 | ❌ | ⭐⭐⭐⭐ 仅开发 |
| **JSONP** | 老接口 GET | ✅ | ⭐ 仅兼容 |
| **postMessage** | iframe / 多窗口通信 | 双方约定 | ⭐⭐⭐⭐ |
| **document.domain** | 同主域不同子域 | 双方设置 | ⭐ 已被废弃 |
| **WebSocket** | 双向长连接 | ✅ | ⭐⭐⭐⭐ |

**方案 1:CORS(跨域资源共享,生产首选)**

浏览器与服务端通过响应头协商。分**简单请求**和**预检请求**两种。

```nginx
# 服务端响应头
Access-Control-Allow-Origin: https://a.com           # 允许的来源(不要写 * 配 cookie)
Access-Control-Allow-Credentials: true               # 允许带 cookie
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400                        # 预检结果缓存
```

**简单请求 vs 预检请求**

| 维度 | 简单请求 | 预检请求(Preflight) |
|------|---------|-------------------|
| 触发条件 | GET/POST/HEAD + 安全 header + 标准 Content-Type | 其它情况 |
| 流程 | 直接发请求 | 先发 OPTIONS,通过后再发真实请求 |
| 典型 | `fetch('/api')` | `fetch('/api', { method: 'PUT' })` |

**前端配合**:

```javascript
// 默认不带 cookie,跨域要带必须设
fetch('/api', { credentials: 'include' })
axios.defaults.withCredentials = true
```

**方案 2:Nginx 反向代理(生产推荐)**

把跨域问题在网关层抹平,浏览器看到的始终是同源。

```nginx
server {
  listen 80;
  server_name a.com;
  location /api/ {
    proxy_pass http://backend.b.com:3000/;     # 浏览器认为还在 a.com
  }
}
```

详见 `14-Git与DevOps面试题.md` 的 Nginx 配置题。

**方案 3:开发态 dev-server proxy**

```javascript
// vite.config.js
export default {
  server: {
    proxy: {
      '/api': {
        target: 'http://backend:3000',
        changeOrigin: true,
        rewrite: p => p.replace(/^\/api/, ''),
      },
    },
  },
}
```

**注意**:这个**只在开发服务器有效**,上线必须由 Nginx 或 CORS 接管。

**方案 4:JSONP(只看不用)**

```javascript
function jsonp(url, callbackName) {
  window[callbackName] = data => console.log(data)
  const s = document.createElement('script')
  s.src = `${url}?callback=${callbackName}`
  document.body.appendChild(s)
}
// 服务端返回:callbackName({...})
```

**只支持 GET、易被 XSS、几乎不再用**,只在维护老系统时遇到。

**方案 5:postMessage(跨窗口通信)**

主页面与 iframe / 弹窗 / Worker 之间通信。

```javascript
// 父页面
const iframe = document.querySelector('iframe')
iframe.contentWindow.postMessage({ type: 'hi' }, 'https://child.com')

// iframe 内
window.addEventListener('message', e => {
  if (e.origin !== 'https://parent.com') return     // 必须校验来源
  console.log(e.data)
})
```

**关键点**:`postMessage` 第二个参数指定 `targetOrigin`,接收侧必须**校验 `e.origin`**,否则就是 XSS 入口。

**特殊问题**

1. **跨域 cookie**:CORS 允许带 cookie 必须满足三条:`Access-Control-Allow-Origin` 不能是 `*`、必须 `Allow-Credentials: true`、前端 `credentials: 'include'`。
2. **跨域字体 / 图片 canvas 取像素**:被同源策略限制,需要服务端加 `Access-Control-Allow-Origin` + 标签加 `crossorigin`。
3. **跨域 cookie 第三方限制**:Chrome 已逐步禁第三方 cookie,SaaS 嵌入场景要走 OAuth + 重定向方案,别再依赖跨域 cookie。

**实操建议**

1. **生产首选 Nginx 代理**:前端不必关心后端域名,后端不必加 CORS 头,运维统一管理,部署最简单。
2. **要做 SaaS / 开放 API 才用 CORS**:把允许的 origin、method、header 写在后端中间件里,严禁 `Access-Control-Allow-Origin: *` 配合 cookie。
3. **开发用 devServer.proxy 模拟生产代理**,代码里写相对路径 `/api/xxx`,本地、测试、生产用同一份代码。
4. 不要再写 JSONP;不要用 `document.domain`(Chrome 109+ 默认禁用);跨子域 cookie 用 `Domain=.a.com` + CORS 才安全。
5. 排查跨域问题三步走:**看控制台报错(CORS / Mixed Content / Cookie)→ 看请求是否发出 → 看响应头是否齐全**。

### 什么是 JSONP？它的原理和局限性是什么？

```javascript
function jsonp(url, callbackName) {
  const script = document.createElement('script')
  script.src = `${url}?callback=${callbackName}`
  document.body.appendChild(script)
}
```


JSONP 是早期前端解决跨域请求的一种方式，本质上利用了 `script` 标签不受同源策略限制这一点。

原理大致是：

1. 前端动态创建 `script` 标签。
2. 请求服务端接口。
3. 服务端返回一段函数调用代码。
4. 浏览器加载脚本后直接执行，从而把数据传回前端。

它的局限性很明显：

1. 只支持 GET 请求。
2. 安全性较差。
3. 现代项目中基本被 CORS 替代。

### ES6 中扩展运算符和 rest 参数怎么理解？有什么应用场景？

```javascript
const arr1 = [1, 2]
const arr2 = [...arr1, 3]
function sum(...nums) {
  return nums.reduce((a, b) => a + b, 0)
}
```


两者都写作 `...`，但使用位置和语义不同。

1. 扩展运算符用于“展开”，比如把数组、可迭代对象拆开。
2. rest 参数用于“收集”，把多个参数收集成一个数组。

常见应用场景：

1. 数组合并。
2. 参数收集。
3. 浅拷贝。
4. 把类数组转为数组。
5. 对象属性展开。

面试时要强调：同样是 `...`，放在不同语法位置，语义完全不同。

### ES6 中 Iterator 和 `for...of` 怎么理解？

```javascript

  console.log(item)
}
```


Iterator 即迭代器，是一种统一的遍历接口。

核心思想是：

2. 每次返回 `{ value, done }`。

`for...of` 则是基于 Iterator 协议进行遍历的语法。

常见可迭代对象有：

### ES6 中新增的 Set、Map 两种数据结构怎么理解？

```javascript

```


1. `Set` 类似“值的集合”，成员唯一，常用于数组去重。
2. `Map` 是“键值对集合”，键可以是任意类型。

相较于对象：

1. `Map` 的键更灵活。
2. `Map` 更适合频繁增删查。
3. `Set` 更适合处理去重和集合运算。

### WeakMap 和 WeakSet 怎么理解？和 Map、Set 有什么区别？

```javascript
const wm = new WeakMap()
const obj = {}
wm.set(obj, 'data')
console.log(wm.get(obj))
```


WeakMap 和 WeakSet 可以理解为“弱引用版本”的 Map 和 Set。

核心区别：

1. WeakMap 的键只能是对象。
2. WeakSet 的成员只能是对象。
3. 它们持有的是弱引用，不会阻止对象被垃圾回收。
4. 一般不能像普通 Map/Set 那样完整遍历。

常见场景：

1. 给对象附加元数据。
2. 缓存对象相关信息，又不希望影响垃圾回收。
3. 避免因缓存导致内存泄漏。

### 你是怎么理解 ES6 中 Promise 的？使用场景是什么？

```javascript
const p = new Promise(resolve => {
  setTimeout(() => resolve('ok'), 1000)
})
p.then(console.log)
```


Promise 是异步编程的一种解决方案，用来表示一个未来才会完成的操作结果。

它有三种状态：

1. `pending`
2. `fulfilled`
3. `rejected`

特点：

1. 状态一旦改变就不可逆。
2. 支持链式调用。
3. 可以统一处理异步成功和失败。

常见场景：

1. 网络请求
2. 异步流程编排
3. 多个异步任务汇总，如 `Promise.all`

### Promise.all、Promise.race、Promise.allSettled 有什么区别？

```javascript
Promise.all([Promise.resolve(1), Promise.resolve(2)]).then(console.log)
Promise.race([
  new Promise(r => setTimeout(() => r('fast'), 100)),
  new Promise(r => setTimeout(() => r('slow'), 200))
]).then(console.log)
Promise.allSettled([Promise.resolve(1), Promise.reject('err')]).then(console.log)
```


这几个都是 Promise 的静态方法，但聚合策略不同。

1. `Promise.all`：全部成功才成功，一个失败就整体失败。
2. `Promise.race`：谁先结束就采用谁的结果。
3. `Promise.allSettled`：等待所有 Promise 都结束，并返回每个结果状态。

常见应用场景：

1. `all`：多个接口都成功后再继续。
2. `race`：超时控制、抢最快结果。
3. `allSettled`：批量请求后统一查看成功和失败情况。

现在实际业务中，很多异步控制场景已被 `async/await` 替代，但理解 Generator 对理解迭代器和异步演进过程很重要。

### Reflect 是什么？它和 Proxy 有什么关系？

### 1. 简易防抖

```javascript
function debounce(fn, delay = 300) {
  let timer = null;
  return function (...args) {
    clearTimeout(timer)
    timer = setTimeout(() => fn.apply(this, args), delay)
  }
}
```

### 2. 简易节流

```javascript
function throttle(fn, delay = 300) {
  let last = 0
  return function (...args) {
    const now = Date.now()
    if (now - last >= delay) {
      last = now
      fn.apply(this, args)
    }
  }
}
```

### 3. 手写深拷贝

```javascript
function deepClone(obj, map = new WeakMap()) {
  if (obj === null || typeof obj !== 'object') return obj
  if (map.has(obj)) return map.get(obj)
  const result = Array.isArray(obj) ? [] : {}
  map.set(obj, result)
  for (const key in obj) {
    if (Object.prototype.hasOwnProperty.call(obj, key)) {
      result[key] = deepClone(obj[key], map)
    }
  }
  return result
}

/**
  1. 先拷贝外壳
  2. 再一层层处理内部属性
  3. 遇到对象就继续展开
  4. 遇到基础值就直接复制
  5. 用 Map 防止重复拷贝和循环引用
*/
function deepCloneIterative(source) {
    if (source === null || typeof source !== 'object') return source
    const root = Array.isArray(source) ? [] : {}
    const map = new Map()
    map.set(source, root)

    const stack = [
      { source, target: root }
    ]
    while (stack.length) {
      const { source, target } = stack.pop()
      for (const key in source) {
        if (!Object.prototype.hasOwnProperty.call(source, key)) continue
        const value = source[key]
        if (value !== null && typeof value === 'object') {
          if (map.has(value)) {
            target[key] = map.get(value)
          } else {
            const clone = Array.isArray(value) ? [] : {}
            target[key] = clone
            map.set(value, clone)
            stack.push({ source: value, target: clone })
          }
        } else {
          target[key] = value
        }
      }
    }

    return root
  }
```

### 4. 手写发布订阅

```javascript
class EventBus {
  constructor() {
    this.events = {}
  }
  on(name, fn) {
    ;(this.events[name] ||= []).push(fn)
  }
  emit(name, ...args) {
    ;(this.events[name] || []).forEach(fn => fn(...args))
  }
  off(name, fn) {
    this.events[name] = (this.events[name] || []).filter(item => item !== fn)
  }
}
```
