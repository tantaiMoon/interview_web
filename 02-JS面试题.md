## 一、基础类型与语言机制

### 说说 JavaScript 中的数据类型？它们有什么区别？

```javascript
const num = 1
const str = 'hello'
const bool = true
const und = undefined
const nul = null
const sym = Symbol('id')
const big = 10n
const obj = { name: 'Tom' }

console.log(typeof num)   // number
console.log(typeof str)   // string
console.log(typeof bool)  // boolean
console.log(typeof und)   // undefined
console.log(typeof nul)   // object  ←  历史遗留 bug
console.log(typeof sym)   // symbol
console.log(typeof big)   // bigint
console.log(typeof obj)   // object
```

JS 数据类型分两大类：

- **原始类型 7 种**：`number`、`string`、`boolean`、`undefined`、`null`、`symbol`、`bigint`
- **引用类型**：`object`，及其派生的 `Array`、`Function`、`Date`、`RegExp`、`Map`、`Set` 等

存储和复制行为是主要区别：原始类型直接存值，赋值/传参都是复制一份；引用类型存的是堆上对象的地址，赋值是复制地址，所以两个变量指向同一个对象，改一边会影响另一边。

`typeof null === 'object'` 的来源：JS 早期版本里值在底层用低位标签表示对象类型，而 null 被实现成全 0 指针，恰好落在了"对象"那个标签上。这个 bug 在 ECMAScript 提案里讨论过是否修复，但会破坏太多老代码，就保留了。判 null 用 `=== null` 就好。

`symbol` 和 `bigint` 是后加的两个：前者用来做对象的唯一 key（避免库之间的属性冲突），后者用来表示超过 `Number.MAX_SAFE_INTEGER`（即 2^53-1）的大整数。

### typeof 与 instanceof 的区别是什么？

```javascript
typeof 1               // 'number'
typeof 'a'             // 'string'
typeof undefined       // 'undefined'
typeof null            // 'object'        ← 不准
typeof []              // 'object'        ← 不准
typeof function(){}    // 'function'

[] instanceof Array              // true
[] instanceof Object             // true   ← 数组也是对象
new Date() instanceof Date       // true
'abc' instanceof String          // false  ← 字面量没经过包装
```

定位完全不同：

- **`typeof`** 看的是值的底层类型标签，只能区分到 7 种原始类型 + `object` + `function`。它最大的用处是判断变量"存在与否"——`typeof x === 'undefined'` 即使 x 没声明也不报错。
- **`instanceof`** 沿着原型链查找 `instanceof` 右侧构造函数的 `prototype`，能区分具体引用类型。但跨 iframe / 跨 realm 时会失败（每个 window 有独立的 Array 构造函数）。

判类型较稳的"终极方案"是 `Object.prototype.toString.call(x)`，返回 `[object Array]` / `[object Null]` / `[object Date]` 这种字符串，不受原型修改和 realm 影响：

```js
const type = v => Object.prototype.toString.call(v).slice(8, -1).toLowerCase()
type(null)      // 'null'
type([])        // 'array'
type(new Map()) // 'map'
```

### 谈谈 JavaScript 中的类型转换机制。

```javascript
Number('123')      // 123
Number('123a')     // NaN
Number('')         // 0
Number(null)       // 0     ← 注意
Number(undefined)  // NaN   ← 跟 null 不一样
Number([])         // 0
Number([1])        // 1
Number([1,2])      // NaN
Number({})         // NaN

'5' - 1            // 4    （减号触发数字转换）
'5' + 1            // '51' （加号字符串优先）
1 + '2' + 3        // '123'
'5' == 5           // true
[] == false        // true
[] == ![]          // true ← 经典坑
```

显式转换三个函数：`Number()` / `String()` / `Boolean()`，外加 `parseInt` / `parseFloat`（处理"123px"这种）。

隐式转换最爱出现在四个地方：`==` 比较、`+` 拼接、`if` / 三元等条件判断、算术运算。

**Falsy 值就六个**：`false` / `0` / `''` / `null` / `undefined` / `NaN`（再加 `0n` 总共 7 个）。除此之外都是 truthy——包括 `'0'`、`'false'`、`[]`、`{}`，这点经常考。

**对象转原始值**走 `ToPrimitive` 算法：

1. 先看有没有 `Symbol.toPrimitive` 方法，有就调
2. 否则根据上下文调 `valueOf` 或 `toString`——数字上下文优先 `valueOf`，字符串上下文优先 `toString`
3. 拿到的还是对象就报 TypeError

`[] + []` 得到 `''`，`[] + {}` 得到 `'[object Object]'`，`{} + []` 得到 `0`（这里 `{}` 被解析成块语句）——都是这套规则的副作用。

工程上的结论：能用 `===` 就别用 `==`，能显式转就别让 JS 隐式转，少问题。

### == 和 === 的区别，分别在什么情况下使用？

```javascript
console.log(1 == '1')     // true
console.log(1 === '1')    // false
console.log(null == undefined)  // true
console.log(null === undefined) // false
```


1. `==` 会先做类型转换，再比较值是否相等。
2. `===` 不会做类型转换，要求类型和值都相等。

开发中通常优先使用 `===`，因为行为更明确、可预测性更高。

`==` 主要的问题在于它的隐式转换规则复杂，容易出现非预期结果，比如：

1. `'' == 0` 为 `true`
2. `null == undefined` 为 `true`

开发时优先使用 `===`，不是因为“规范要求”，而是因为 `==` 会先做隐式类型转换，转换规则不少，读代码的人必须先推断两边会不会被转换、转换成什么再判断结果。
这会明显提高理解成本，也更容易埋下边界 bug，所以除非你明确需要利用它的转换规则，否则应优先使用 `===`。

### 深拷贝和浅拷贝的区别是什么？如何实现一个深拷贝？

```javascript
function deepClone(obj, map = new WeakMap()) {
  if (obj === null || typeof obj !== 'object') return obj
  if (map.has(obj)) return map.get(obj)
  const result = Array.isArray(obj) ? [] : {}
  map.set(obj, result)
  for (const key in obj) {
    if (Object.prototype.hasOwnProperty.call(obj, key)) {
      result[key] = deepClone(obj[key], map)
    }
  }
  return result
}
```


浅拷贝只复制对象的第一层属性，如果属性值还是对象，那么复制的仍然是引用地址。

深拷贝会递归复制对象的每一层，最终得到一份完全独立的新数据。

常见浅拷贝方式：

1. `Object.assign()`
2. 展开运算符 `...`
3. 数组的 `slice()`、`concat()`

常见深拷贝方式：

1. `JSON.parse(JSON.stringify(obj))`
2. 手写递归拷贝
3. `structuredClone()`
4. 使用第三方库，比如 `lodash.cloneDeep`

这里顺手再记两个限制即可：

1. `JSON` 方式会丢失 `undefined`、函数、`Symbol`、`Date`、`RegExp` 等特殊值。
2. 手写深拷贝要处理循环引用、原型链、特殊对象类型。

### 不使用递归如何实现一个深克隆?

递归版深拷贝的问题很直接：对象一旦嵌套太深，就可能把调用栈打爆。

不递归的写法也不神秘，本质就是把“函数一层层往下调”的过程，改成“自己维护一个待处理任务栈”。逻辑没变，只是从调用栈换成了显式数据结构。

**代码**

```javascript
function deepCloneIterative(source) {
  // 基础类型直接返回
  if (source === null || typeof source !== 'object') return source

  // 1. 先克隆"外壳",作为根
  const root = Array.isArray(source) ? [] : {}
  const map = new WeakMap()           // 解决循环引用
  map.set(source, root)

  // 2. 用栈代替递归
  const stack = [{ src: source, target: root }]

  while (stack.length) {
    const { src, target } = stack.pop()

    // 用 Reflect.ownKeys 同时拿到 string 和 symbol 键
    for (const key of Reflect.ownKeys(src)) {
      const value = src[key]

      if (value !== null && typeof value === 'object') {
        // 循环引用命中:直接指向已有克隆
        if (map.has(value)) {
          target[key] = map.get(value)
          continue
        }

        // 特殊对象快速路径
        if (value instanceof Date)   { target[key] = new Date(value.getTime()); continue }
        if (value instanceof RegExp) { target[key] = new RegExp(value.source, value.flags); continue }

        // 普通对象/数组:先建空壳,入栈待处理
        const clone = Array.isArray(value) ? [] : {}
        target[key] = clone
        map.set(value, clone)
        stack.push({ src: value, target: clone })
      } else {
        // 基础类型 + 函数(函数通常按引用拷)
        target[key] = value
      }
    }
  }

  return root
}

// 测试:超深嵌套(递归会爆栈,迭代正常)
let deep = {}
let cur = deep
for (let i = 0; i < 100000; i++) { cur.next = {}; cur = cur.next }
const cloned = deepCloneIterative(deep)
console.log('OK')

// 循环引用
const a = { name: 'a' }; a.self = a
const b = deepCloneIterative(a)
console.log(b.self === b)             // true
```

**使用建议**

1. **绝大多数业务对象用 `structuredClone(obj)`**,浏览器和 Node 17+ 内置,正确处理 Date/RegExp/Map/Set/循环引用,且采用非递归实现,**首选**。
2. 如果需要手写实现,迭代版本重点展示三点:**显式栈、`WeakMap` 解循环、特殊类型分支**。
3. 函数、Symbol、`#` 私有字段、`getter/setter`、`Proxy` 包装的对象,任何深拷贝方案都不能完美还原,业务代码应避免依赖这类拷贝行为。
4. 性能极端场景(大对象高频拷贝),与其优化深拷贝,不如改成**不可变数据 + 结构共享**(Immer / Immutable.js),从源头避免拷贝。

### JavaScript 中参数是按值传递还是按引用传递？

```javascript
function change(a, b) {
  a = 20
  b.name = 'Jerry'
}

let x = 10
let y = { name: 'Tom' }
change(x, y)
console.log(x)       // 10
console.log(y.name)  // Jerry
```


更准确的说法是：JavaScript 中函数参数一律按值传递。

区别在于传递的“值”是什么：

1. 基本类型传递的是值本身。
2. 引用类型传递的是对象地址这个值。

所以不少人会误以为对象是“按引用传递”，实际不是。对象参数在函数里如果修改对象内部属性，外部会感知到；但如果在函数内部把参数重新赋值为另一个对象，不会影响外部变量本身。

更准确的说法是：JavaScript 没有严格意义上的“按引用传递”，而是始终按值传递。
之所以不少人误以为对象是“按引用传递”，是因为对象变量里保存的这个“值”恰好是地址，函数拿到这份地址拷贝后，仍然可通过它访问并修改同一个对象。

### JavaScript 数组的常用方法有哪些？

```javascript
const arr = [1, 2, 3]
console.log(arr.map(x => x * 2))       // [2,4,6]
console.log(arr.filter(x => x > 1))    // [2,3]
console.log(arr.reduce((a, b) => a + b, 0)) // 6
arr.push(4)
console.log(arr.includes(3))           // true
```


数组常见方法可以按用途分类：

1. 增删改：`push`、`pop`、`shift`、`unshift`、`splice`
2. 查找：`find`、`findIndex`、`includes`、`indexOf`
3. 遍历：`forEach`、`map`、`filter`、`some`、`every`
4. 归并：`reduce`
5. 截取拼接：`slice`、`concat`
6. 排序反转：`sort`、`reverse`
7. 转换：`join`
8. 扁平化：`flat`

这里真正容易混淆的点，不是“这些方法都见过”，而是它们会不会返回新值、会不会改原数组。

1. `map` 会返回新数组，因为它的设计目标是数据映射
2. `forEach` 没有返回值，因为它只是为了执行副作用
3. `splice` 会修改原数组，因为它本身就是增删改操作
4. `slice` 不会修改原数组，因为它的定位是截取副本

### 什么是类数组对象？`arguments` 有什么特点？

```javascript
function demo() {
  console.log(arguments.length)
  console.log(arguments[0])
  const arr = Array.from(arguments)
  console.log(arr.map(x => x * 2))
}
demo(1, 2, 3)
```


类数组对象指的是“长得像数组，但并不是数组”的对象，通常具有：

1. 按数字索引存储元素。
2. 有 `length` 属性。
3. 不能直接使用完整的数组原型方法。

`arguments` 就是函数内部典型的类数组对象，用来获取实参列表。

它的特点大概是：

1. 不是数组。
2. 可通过下标访问参数。
3. 有 `length`。
4. 在现代开发中通常更推荐使用剩余参数 `...args` 替代。

类数组转数组，常用做法有：

1. `Array.from(arguments)`
2. `[...arguments]`
3. `Array.prototype.slice.call(arguments)`

### 说说 var、let、const 之间的区别。

```javascript
var a = 1
let b = 2
const c = 3

if (true) {
  var a = 10
  let b = 20
  console.log(a, b) // 10 20
}
console.log(a) // 10
console.log(b) // 2
```


三者都能声明变量，但差异明显。

1. `var` 存在变量提升，且没有块级作用域。
2. `let` 有块级作用域，存在暂时性死区，不允许重复声明。
3. `const` 也有块级作用域，声明后必须初始化，且不能重新赋值。

补充说明：

1. `const` 保证的是绑定地址不变，不代表对象内容绝对不可修改。
2. 在现代开发中，优先使用 `const`，需要重新赋值时再用 `let`。

### JavaScript 为什么会有变量提升?

**为什么会这样**

JS 引擎执行代码分两步:**先编译,再执行**。编译阶段会扫描当前作用域,把 `var` 声明和 `function` 声明**提前注册到词法环境**中(分配内存、给默认值),执行阶段才一行行跑代码。这种"声明在编译期处理、赋值留到执行期"的机制,在视觉上就表现为"变量好像被提到了顶部",但实际上**变量从未真的被移动**,只是引擎在编译期已经知道它存在。

**具体过程**

1. **`var` 声明**:编译期注册,默认值 `undefined`。访问它得到 `undefined`,不报错。
2. **`function` 声明**:编译期连函数体一起提升,所以提前调用也能跑。
3. **`function` 表达式**:右边的函数对象不会被提升,只有左边的 `var` 名字被提升。
4. **`let` / `const` / `class`**:也会被引擎记录("半提升"),但**进入暂时性死区(TDZ)**,声明语句执行前访问会抛 `ReferenceError`,不是 `undefined`。
5. **同名提升优先级**:函数声明 > 变量声明;后定义的同名函数会覆盖前面的。

直接对比的话可按下面理解：

1. `var` 会提升，初值是 `undefined`，提前访问不会报错
2. `function` 声明也会提升，而且连函数体一起提升，所以能直接调用
3. `function` 表达式只提升变量名，初值还是 `undefined`
4. `let` / `const` / `class` 也会提升，但会进入 TDZ，提前访问直接报 `ReferenceError`

**代码**

```javascript
console.log(a)        // undefined  —— var 被提升,值为 undefined
var a = 1

console.log(b)        // ReferenceError —— let 在 TDZ
let b = 1

foo()                 // ✅ 正常调用 —— 函数声明整体提升
function foo() { console.log('foo') }

bar()                 // TypeError: bar is not a function —— 只提升 var
var bar = function () { console.log('bar') }
```

**使用建议**

1. 全部用 `let` / `const`,变量提升和 TDZ 带来的"看似无关代码却报错"问题基本消失。
2. 函数定义优先用**函数声明**(顶层)或 `const fn = () => {}`(块内),不要混用让代码顺序难懂。
3. 讨论 TDZ 时,关键是说明"`let/const` 也提升,只是访问会抛错"——TDZ 不等于"不提升"。

### const 声明的变量值想要改变该怎么做?

**为什么会这样**

`const` 在 JS 里只锁定**变量到值的绑定**,不锁定值本身的内容。也就是说,`const obj = {}` 保证 `obj` 这个标识符不能再指向新对象,但**对象内部的属性仍然可以修改**。要改变 `const` 变量的可观察结果，只能修改其内容。

**具体过程**

1. **`const` 的限制**:不能 `=` 重新赋值;但读 / 写对象属性、调用对象方法、push 数组、Map.set 等**完全不受限制**。
2. **三层"不可变"层级**:
   - 第一层:`const`(绑定不可变,内容可变)。
   - 第二层:`Object.freeze()`(冻结第一层属性,深层仍可变)。
   - 第三层:递归深冻结 / Immutable.js / Immer(任何层级都不能改)。
3. **想"换值"的场景**:重新声明一个新 `const`,或者改用 `let`。
4. **特殊情况**:`Object.freeze` 在严格模式下改属性会抛错,非严格模式下静默失败。

**代码**

```javascript
const obj = { count: 0 }

// ❌ 不行:重新赋值给 obj 本身
obj = { count: 1 }              // TypeError: Assignment to constant variable

// ✅ 改属性可以
obj.count = 1
obj.name = 'Tom'

// ✅ 用 Object.assign 批量改
Object.assign(obj, { count: 2, name: 'Jerry' })

// ✅ 数组同理
const arr = [1, 2, 3]
arr.push(4)                     // OK,arr 还是同一个数组引用
arr[0] = 999                    // OK

// 如果需要"不可变",用 Object.freeze 或解构产生新对象
const frozen = Object.freeze({ count: 0 })
frozen.count = 1                // 严格模式抛错,非严格模式静默失败

// 函数式风格:不改原对象,生成新对象(配 let)
let state = { count: 0 }
state = { ...state, count: state.count + 1 }
```

**对比表**

如果只看可变范围，可以这么记：

1. `let obj = {}`：绑定能改，对象内容也能改
2. `const obj = {}`：绑定不能改，但对象内容还能改
3. `Object.freeze`：第一层属性不能改，但深层对象默认还能改
4. 递归深 freeze：从第一层到深层都不能改

**使用建议**

1. 默认 `const`,只有在确实需要重新赋值时才改 `let`——这是社区共识。
2. 想"语义上不可变"的常量(如配置表、枚举映射),配上 `Object.freeze` 或 `as const`(TS),让误改在开发期就报错。
3. React / Redux 这类"状态不可变"的体系下,改值要靠**生成新对象**(`...state` 或 Immer),不要去改 `const` 内部属性,否则视图不会刷新。
4. 不要用 `Object.freeze` 当深拷贝替代品,它只冻一层。

### null 和 undefined 有什么区别？

这是一道很高频的基础题。

1. `undefined`
   - 表示“未定义”
   - 变量声明了但还没赋值
   - 函数没有显式返回值时默认返回 `undefined`

2. `null`
   - 表示“空值”
   - 通常是开发者主动赋的一个空对象指针语义

常见区别：

1. `typeof undefined === 'undefined'`
2. `typeof null === 'object'`，这是历史遗留问题
3. `null == undefined` 为 `true`
4. `null === undefined` 为 `false`

之所以要把这两个值区分开，是因为它们在语义上承担的角色不同：

1. `undefined` 更多表示“系统还没给值”
2. `null` 更多表示“这里原本应该有值，但我现在明确让它为空”

这个差异会直接影响 API 设计、默认值处理和数据校验逻辑。

### Object.is 和 === 有什么区别？

大多数情况下，`Object.is` 和 `===` 很像，但它更精确。

主要区别有两个：

1. `NaN === NaN` 是 `false`
   - `Object.is(NaN, NaN)` 是 `true`

2. `+0 === -0` 是 `true`
   - `Object.is(+0, -0)` 是 `false`

所以可按下面理解：

- `===` 是严格相等
- `Object.is` 是更精确的同值比较

### 数组的 forEach、map、filter、some、every、find 有什么区别？

这类题可以看作把这些方法的返回值和用途分清楚。

1. `forEach`
   - 遍历数组
   - 没有返回值

2. `map`
   - 返回一个映射后的新数组

3. `filter`
   - 返回符合条件的新数组

4. `some`
   - 只要有一个满足条件就返回 `true`

5. `every`
   - 必须全部满足条件才返回 `true`

6. `find`
   - 返回第一个满足条件的元素
   - 找不到返回 `undefined`

### for...in 和 for...of 有什么区别？

1. `for...in`
   - 遍历可枚举属性名
   - 返回的是 `key`
   - 更适合对象

2. `for...of`
   - 遍历可迭代对象的值
   - 返回的是 `value`
   - 更适合数组、字符串、Set、Map

注意点：

1. `for...in` 不推荐遍历数组
2. `for...of` 不能直接遍历普通对象，除非对象实现了迭代器

### Object.freeze、Object.seal、Object.preventExtensions 有什么区别？

1. `Object.preventExtensions(obj)`
   - 不能新增属性
   - 但还能删除和修改

2. `Object.seal(obj)`
   - 不能新增和删除属性
   - 但还能修改已有属性值

3. `Object.freeze(obj)`
   - 不能新增、删除、修改属性

这里容易被追问的一点是：这三个方法默认都只是浅层控制。
原因是它们限制的是对象第一层属性描述符或扩展能力，而不是递归处理整个对象图，所以如果属性值本身还是对象，那个内部对象依然可以继续被修改。

## 二、作用域、原型与面向对象

### 说说你对作用域链的理解。

**为什么会这样**

JS 是**词法作用域(静态作用域)** 语言:函数能访问哪些变量,在**写代码时**就由它在源码中的物理位置决定,不是运行时谁调它就用谁的环境。当函数访问一个变量时,需要一条"由内向外的查找路径"——这条路径就是**作用域链**。它的本质还是变量查找算法,作用是把"嵌套结构"变成"线性查找表"。

**具体过程**

1. **函数定义时**就保存一个 `[[Environment]]` 内部属性,指向定义所在的词法环境。
2. **函数调用时**创建新的"执行上下文",其作用域链 = `当前词法环境 → 外层词法环境 → ... → 全局环境`。
3. **变量查找**:从链头开始,找不到就沿链向外走,直到全局;还找不到就 `ReferenceError`。
4. **作用域链不可变**:函数定义在哪,链就是哪,**和调用位置无关**(这是和 `this` 最大的区别)。
5. **闭包**:函数即使在原作用域销毁后,只要被外部引用,它的作用域链就不会被回收——这正是闭包能"记住"外层变量的原因。

**对比:作用域链 vs 原型链 vs this 绑定**

这三个东西容易混，所以可以直接对着记：

1. 作用域链在函数定义时就定下来了，用来找变量，方向是由内向外
2. 原型链在对象创建时建立，用来找属性，方向是实例往原型上走
3. `this` 要到函数调用时才决定，看的是调用方式，不看定义位置

**代码**

```javascript
const a = 1
function outer() {
  const b = 2
  function inner() {
    const c = 3
    console.log(a, b, c)   // 1 2 3:沿作用域链向外找
  }
  return inner
}

// 即使 outer() 已执行完,inner 的作用域链仍持有 b
const fn = outer()
fn()                      // 闭包让 b 不被回收

// 词法作用域 vs 动态作用域:JS 是词法的
const x = 'global'
function readX() { return x }     // x 在定义时就锁定到 global
function caller() {
  const x = 'local'
  return readX()                  // 仍然返回 'global',不是 'local'
}
console.log(caller())             // 'global'
```

**使用建议**

1. 不要写"靠层层向外找变量"的代码,可读性极差;每个变量尽量在使用它的最小作用域内声明。
2. 闭包持有作用域链 = 持有外层变量,**用完后置 `null` 或解绑**,防止意外内存占用(详见"闭包内存泄漏"一题)。
3. 大量循环里访问外层变量会有微小性能成本,把它**先用 `const local = outer` 缓存到当前作用域**,JIT 优化更好。
4. 模块化(ESM)就是天然的作用域隔离,不要用 IIFE 和 `var` 模拟旧时代的私有作用域。

### 说说你对闭包的理解。

闭包不用讲得太玄。函数离开原来的作用域以后，仍然能访问外层变量，这就是闭包。

它常见的用途主要两类：一类是把数据藏起来做封装，另一类是把某个状态记住，留给后面的调用继续用。防抖、节流、柯里化、缓存函数，都是这个思路。

闭包本身不等于内存泄漏。问题只会出在引用本来该断的时候没有断，比如把闭包挂到全局对象上、定时器没清、监听器没解绑，或者顺手把一个很大的对象也一起引用住了。

**代码**

```javascript
// ① 数据私有化
function createCounter() {
  let count = 0                  // 外部访问不到
  return {
    inc: () => ++count,
    get: () => count,
  }
}
const c = createCounter()
c.inc(); c.inc()
console.log(c.get())             // 2

// ② 状态持久化(防抖)
function debounce(fn, delay) {
  let timer = null               // 闭包记住 timer
  return (...args) => {
    clearTimeout(timer)
    timer = setTimeout(() => fn(...args), delay)
  }
}

// ③ 共享环境陷阱
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0)   // 3 3 3:三个闭包共享同一个 i
}
for (let j = 0; j < 3; j++) {
  setTimeout(() => console.log(j), 0)   // 0 1 2:每轮 let 产生新作用域
}

// ④ 真实泄漏:DOM 引用挂到全局
let leakedHandler
function bind() {
  const huge = new Array(1e6).fill('x')
  leakedHandler = () => console.log(huge.length)   // huge 永远不释放
}
```

**如何预防内存泄漏**

1. **配对清理**:`addEventListener` 必有对应 `removeEventListener`;`setInterval` 必有对应 `clearInterval`。
2. **解除引用**:闭包用完显式置 `fn = null`,或用 `WeakMap` / `WeakRef` 持有"可回收"的引用。
3. **作用域最小化**:闭包内只引用真正需要的字段,不要把整个大对象捎进去——一个不该被引用的 `event` 对象就能拖死一棵 DOM 树。
4. **组件/页面销毁时统一清理**:Vue 的 `onUnmounted`、React 的 `useEffect` 返回函数,统一收拾监听、定时器、订阅。
5. **DevTools 验证**:Chrome Memory 面板 → Heap snapshot → 搜 `Detached` 关键字,能直接看到没释放的 DOM 闭包。
6. **代码评审重点**:"这个回调谁来解绑?"——大部分泄漏死在"绑了没解"。

闭包本身不是负担，问题只会出在该清理的时候没清理。真放到工程里，重点还是监听器、定时器、全局缓存这些地方的引用管理。

### JavaScript 中执行上下文和执行栈是什么？

```javascript
function foo() {
  console.log('foo start')
  bar()
  console.log('foo end')
}

function bar() {
  console.log('bar')
}

foo()
```


执行上下文可以看成代码执行时所需的环境信息，主要包括：

1. 变量对象或词法环境
2. 作用域链
3. `this` 指向

执行栈则是管理执行上下文的栈结构。

代码执行过程通常是：

1. 全局代码先进入全局执行上下文。
2. 函数调用时创建函数执行上下文并压栈。
3. 函数执行完成后弹栈。
4. 最终回到全局上下文。

执行栈说到底就是 JavaScript 用来管理函数调用顺序的机制。

### 谈谈 this 对象的理解。

```javascript
const obj = {
  name: 'Tom',
  say() {
    console.log(this.name)
  }
}
obj.say() // Tom

const fn = obj.say
fn() // 非严格模式下通常是 undefined 或 window.name
```


`this` 不是函数定义时决定的，而是**调用时**决定的——同一个函数被怎么调用，`this` 就是什么。

四条绑定规则，优先级从低到高：

| 调用方式 | this 指向 | 示例 |
|---------|----------|------|
| 默认调用 | 非严格 `window` / `globalThis`，严格 `undefined` | `fn()` |
| 隐式调用（方法调用） | 调用它的那个对象 | `obj.fn()` |
| 显式绑定（call/apply/bind） | 传入的第一个参数 | `fn.call(obj)` |
| new 调用 | 新创建的实例 | `new Fn()` |

外加一个特例：**箭头函数没有自己的 this**，沿词法作用域往外找。`call/apply/bind` 也改不了箭头函数的 this。

容易出现的问题：

```js
const obj = {
  name: 'Tom',
  say() { console.log(this.name) },
  delay() { setTimeout(function () { console.log(this.name) }, 100) }
}

obj.say()        // 'Tom' —— 隐式绑定

const f = obj.say
f()              // undefined —— 拆下来调用，丢了上下文

obj.delay()      // undefined —— setTimeout 回调是普通函数调用，this 是 window
// 修复方式：把回调改成箭头函数，继承 delay 的 this
```

回调、定时器、事件处理器里 this 经常丢失，根源都是函数从对象上被"拆下来"调用。React 类组件里到处 `this.xxx = this.xxx.bind(this)`、Vue 模板里 `@click="handleClick"` 自动绑组件实例，都是为了解决这件事。箭头函数能解决大部分场景。

`this` 和作用域链是两件事：作用域链管"变量从哪找"（定义时确定），`this` 管"当前调用的上下文"（调用时确定）。把这两条混在一起就会越想越乱。

### JavaScript 原型、原型链是什么？有什么特点？

```javascript
function Person(name) { this.name = name }
Person.prototype.sayHi = function () { return 'hi ' + this.name }

const p = new Person('Tom')
console.log(p.sayHi())                          // hi Tom
console.log(p.__proto__ === Person.prototype)   // true
console.log(Person.prototype.__proto__ === Object.prototype) // true
console.log(Object.prototype.__proto__)         // null  ← 链的终点
```

每个对象内部有个 `[[Prototype]]` 槽，指向另一个对象——它的原型。访问属性时本对象没有就沿着这条引用往上找，直到 `Object.prototype` 顶端，再往上是 `null`。这就是原型链。

几条容易绕晕的关系：

- `Person.prototype` 是个普通对象，里面有 `constructor` 指回 `Person`。
- `p.__proto__ === Person.prototype`（实例的原型指向构造函数的 prototype）。
- `Person.__proto__ === Function.prototype`（函数本身也是对象，它的原型是 `Function.prototype`）。
- `Function.prototype.__proto__ === Object.prototype`（绕一圈最后还是回到 Object）。
- `Object.__proto__ === Function.prototype`（`Object` 也是构造函数）。
- `Function.__proto__ === Function.prototype`（自己是自己的原型，唯一的循环）。

实际工程意义：原型链是 JS 实现"方法复用"的机制——所有 `Person` 实例共享同一份 `sayHi`，不会每个实例都存一份。实例上自己的属性叫"自有属性"（`Object.hasOwn(p, 'name')` 是 true），原型上的属性通过 `in` 操作符能查到但不是自有属性。

`__proto__` 是非标准但事实上通用的访问器，标准写法是 `Object.getPrototypeOf(obj)` 和 `Object.setPrototypeOf(obj, proto)`。改原型在 V8 里很慢，应该用 `Object.create(proto)` 创建对象时就指定好。

### JavaScript 如何实现继承？各方式的优缺点是什么?

JS 没有 Java 那种传统类继承机制，所谓“继承”就是把另一个对象放到自己的原型链上。

传统资料里会把继承方式拆成不少代，实际常用结论很简单：

1. 早期写法不少，本质都在解决两件事
   - 子实例怎么拿到父类实例属性
   - 子实例怎么复用父类原型方法
2. 现在新项目直接用 `class extends`
3. 老项目如果看到各种寄生组合、借用构造函数之类写法，知道它们分别解决什么问题即可

常见写法可以按下面方式区分：

1. 原型链继承：能复用原型方法，但实例属性容易共享出问题
2. 借用构造函数：实例属性独立了，但拿不到父类原型方法
3. 组合继承：两边都补到了，但父构造函数会调用两次
4. 寄生组合继承：把组合继承的多余调用优化掉
5. `class extends`：现在常见用，语义也最清晰

最典型的几种写法：

**① 原型链继承**

```javascript
function Parent() { this.colors = ['red'] }
function Child() {}
Child.prototype = new Parent()

const a = new Child(), b = new Child()
a.colors.push('blue')
console.log(b.colors)         // ['red', 'blue']  ⚠️ 引用类型被共享
```

**② 借用构造函数**

```javascript
function Parent(name) { this.name = name; this.colors = ['red'] }
function Child(name) { Parent.call(this, name) }   // 借用

Parent.prototype.say = function () {}              // 子实例拿不到这个方法
const c = new Child('Tom')
c.say()                                            // ❌ TypeError
```

**③ 组合继承**

```javascript
function Parent(name) { this.name = name; this.colors = ['red'] }
Parent.prototype.say = function () { return this.name }

function Child(name, age) {
  Parent.call(this, name)        // 第 1 次调用 Parent —— 拿到实例属性
  this.age = age
}
Child.prototype = new Parent()   // 第 2 次调用 Parent —— 浪费
Child.prototype.constructor = Child
```

**⑥ 寄生组合继承**

```javascript
function inherit(Child, Parent) {
  Child.prototype = Object.create(Parent.prototype)   // 不调 Parent
  Child.prototype.constructor = Child
}

function Parent(name) { this.name = name }
Parent.prototype.say = function () { return this.name }

function Child(name, age) {
  Parent.call(this, name)
  this.age = age
}
inherit(Child, Parent)
```

**⑦ ES6 `class extends`**

```javascript
class Animal {
  constructor(name) { this.name = name }
  say() { return this.name }
}
class Dog extends Animal {
  constructor(name, breed) {
    super(name)                  // 必须在 this 之前调
    this.breed = breed
  }
  bark() { return 'wang' }
}
```

如果只看工程建议，可以直接记三条：

1. 新项目直接用 `class extends`
2. 老项目看到各种继承写法，先判断它是哪一类，不要急着改
3. 业务代码里大多数复用问题，更适合用组合而不是继承

### 说说 new 操作符具体干了什么？

```javascript
function myNew(Constructor, ...args) {
  const obj = Object.create(Constructor.prototype)
  const result = Constructor.apply(obj, args)
  return result !== null && (typeof result === 'object' || typeof result === 'function')
    ? result
    : obj
}
```


`new` 的执行过程大致分几步：

1. 创建一个新对象。
2. 把新对象的原型指向构造函数的 `prototype`。
3. 以这个新对象作为 `this` 执行构造函数。
4. 如果构造函数返回的是对象，就返回该对象；否则返回新创建的对象。

所以 `new` 的本质还是“创建实例对象并建立原型关联”。

手写一个简化版 `new`，核心实现可以这样写：

```javascript
function myNew(Constructor, ...args) {
  const obj = Object.create(Constructor.prototype)
  const result = Constructor.apply(obj, args)
  return result !== null && (typeof result === 'object' || typeof result === 'function')
    ? result
    : obj
}
```

示例：

```javascript
function Person(name) {
  this.name = name
}

Person.prototype.sayHi = function () {
  return 'hi ' + this.name
}

const p = myNew(Person, 'Tom')
console.log(p.name)      // Tom
console.log(p.sayHi())   // hi Tom
```

### bind、call、apply 的区别是什么？如何实现一个 bind？

```javascript
Function.prototype.myCall = function (context, ...args) {
  context = context == null ? globalThis : Object(context)
  const key = Symbol('fn')
  context[key] = this
  const result = context[key](...args)
  delete context[key]
  return result
}

Function.prototype.myApply = function (context, args = []) {
  context = context == null ? globalThis : Object(context)
  const key = Symbol('fn')
  context[key] = this
  const result = context[key](...args)
  delete context[key]
  return result
}
```


这三个方法说到底都在做一件事：改函数执行时的 `this`。

区别：

1. `call`：立即执行，参数逐个传入。
2. `apply`：立即执行，参数以数组形式传入。
3. `bind`：不立即执行，返回一个绑定了 `this` 的新函数。

如果手写 `bind`，核心要点有：

1. 返回一个新函数。
2. 新函数执行时把 `this` 绑定到指定对象。
3. 支持柯里化参数。
4. 处理作为构造函数时的 `new` 场景。

可以顺手把 `call`、`apply`、`bind` 的简化实现一起记住。

1. 手写 `call`

```javascript
Function.prototype.myCall = function (context, ...args) {
  context = context == null ? globalThis : Object(context)
  const key = Symbol('fn')
  context[key] = this
  const result = context[key](...args)
  delete context[key]
  return result
}
```

2. 手写 `apply`

```javascript
Function.prototype.myApply = function (context, args = []) {
  context = context == null ? globalThis : Object(context)
  const key = Symbol('fn')
  context[key] = this
  const result = context[key](...args)
  delete context[key]
  return result
}
```

3. 手写 `bind`

```javascript
Function.prototype.myBind = function (context, ...outerArgs) {
  const self = this

  function boundFn(...innerArgs) {
    const isNew = this instanceof boundFn
    const thisArg = isNew ? this : context
    return self.apply(thisArg, [...outerArgs, ...innerArgs])
  }

  boundFn.prototype = Object.create(self.prototype)
  return boundFn
}
```

示例：

```javascript
function greet(greeting, name) {
  return greeting + ', ' + name
}

console.log(greet.myCall(null, 'Hello', 'Tom'))           // Hello, Tom
console.log(greet.myApply(null, ['Hello', 'Jerry']))      // Hello, Jerry

const sayHello = greet.myBind(null, 'Hello')
console.log(sayHello('Jack'))                             // Hello, Jack
```

### 箭头函数和普通函数有什么区别？

箭头函数和普通函数不只是写法不同。

主要区别：

1. 箭头函数没有自己的 `this`
   - 它的 `this` 来自外层作用域

2. 不能作为构造函数
   - 不能用 `new`

3. 没有 `arguments`
   - 需要用 `rest` 参数代替

4. 没有 `prototype`

适合用在：

1. 回调函数
2. 需要固定 `this` 的场景

不适合：

1. 依赖动态 `this` 的对象方法
2. 构造函数

### 普通函数、构造函数、类三者之间是什么关系？

直接看成这样：

1. JavaScript 中函数本身就是一等公民
2. 构造函数实际也是函数，只是通常配合 `new` 使用
3. `class` 本质是基于原型继承的语法糖

通常会概括为：

> `class` 没有改变 JavaScript 基于原型的本质，只是提供了更清晰的面向对象写法。

## 三、异步、事件与浏览器交互

### 说说 JavaScript 中的事件模型。

```javascript
const box = document.getElementById('box')
box.addEventListener('click', () => console.log('capture'), true)
box.addEventListener('click', () => console.log('bubble'), false)
```


JavaScript 的事件模型通常包含三个阶段：

1. 捕获阶段
2. 目标阶段
3. 冒泡阶段

事件从顶层向目标元素传播是捕获，目标元素处理后再从目标元素向外传播是冒泡。

开发里常见的是冒泡模型，因为它便于做事件代理。

### 解释下什么是事件代理？应用场景是什么？

```javascript
document.getElementById('list').addEventListener('click', e => {
  if (e.target.matches('li')) {
    console.log('clicked', e.target.textContent)
  }
})
```


事件代理指的是不把事件直接绑定到子元素上，而是绑定到它们的父元素或祖先元素上，利用事件冒泡统一处理。

好处：

1. 减少事件绑定数量。
2. 提高性能。
3. 动态新增元素也能自动响应。

常见场景：

1. 列表点击
2. 表格操作按钮
3. 菜单项交互

### 说说浏览器的渲染原理。

浏览器拿到 HTML/CSS/JS 后，到屏幕上出现像素，大致经历下面几个阶段：

1. **解析 HTML 构建 DOM 树**
   解析器把 HTML 字符流变成 DOM 节点树。遇到 `<script>` 默认会阻塞解析（除非加 `async` / `defer`），因为脚本可能修改 DOM。

2. **解析 CSS 构建 CSSOM 树**
   CSS 是渲染阻塞资源：CSSOM 没构建完，就没法计算样式，渲染会卡住。所以 CSS 通常放 `<head>` 提前下载。

3. **合成 Render Tree（渲染树）**
   DOM + CSSOM 合并，过滤掉不可见节点（如 `display:none`、`<head>`），只保留要绘制的节点。

4. **Layout（布局/回流 Reflow）**
   计算每个节点在视口中的位置和大小（盒模型、几何信息）。

5. **Paint（绘制/重绘 Repaint）**
   把每个节点画成位图：文字、颜色、边框、阴影等。

6. **Composite（合成）**
   把不同图层（layer）按层级合成最终画面，交给 GPU 上屏。`transform`、`opacity` 这类属性可以只走合成层，不触发 Layout 和 Paint，所以动画性能最好。

要注意的几个点：

- **回流一定触发重绘，重绘不一定触发回流**。改几何属性（width、top、font-size）会回流，改颜色只会重绘。
- 频繁读写布局属性（`offsetWidth`、`scrollTop`）会引起**强制同步布局**，应批量读写。
- 渲染主线程和 JS 主线程是同一个，**JS 长任务会阻塞渲染**，这就是为什么需要事件循环和异步机制。

### JS 为什么是单线程？多线程会发生什么？

JavaScript 被设计成单线程，最主要还是因为它最初就是拿来操作浏览器 DOM 的。

如果 DOM 同时被多个线程改，浏览器就得引入锁和更复杂的同步机制，模型会一下子重不少。所以浏览器最后走的是另一条路：主线程负责脚本和 DOM，异步任务靠事件循环调度，重计算再交给 Worker。

### 什么是事件循环（Event Loop）？

JS 是单线程，但浏览器需要它同时处理点击事件、定时器、网络响应、Promise 回调……事件循环就是把这些异步事情按规则排队调度起来的机制。

核心模型：

- **调用栈（Call Stack）**：同步代码执行的地方，函数进栈出栈。
- **宿主环境**（浏览器/Node）：提供 Web API（定时器、网络、DOM 事件等），异步操作交给它去等，到时间了把回调塞进任务队列。
- **任务队列**：
  - **宏任务**：`setTimeout` / `setInterval`、I/O、`MessageChannel`、UI 事件、`script` 整体。
  - **微任务**：`Promise.then/catch/finally`、`queueMicrotask`、`MutationObserver`、Node 的 `process.nextTick`（优先级比微任务还高）。

每一轮（tick）的执行规则：

```
1. 取一个宏任务执行
2. 清空所有微任务（执行过程中新加的微任务也会被这一轮清掉）
3. 必要时浏览器渲染（requestAnimationFrame → Style → Layout → Paint）
4. 回到 1，取下一个宏任务
```

记住几个关键点：

- **微任务一次清空完**，宏任务一次只取一个——所以一个 `Promise.then` 里再 `.then` 会立刻被排进当前微任务队列继续执行；而 `setTimeout` 必须等到下一轮。
- **`script` 是宏任务**，所以页面初始整段同步代码本身就是第一个宏任务。
- **渲染时机**：理论上每帧只渲染一次（约 16.6ms），把动画放在 `requestAnimationFrame` 里能保证和渲染同步；放 `setTimeout` 里可能错过帧。
- **浏览器 vs Node**：浏览器每个宏任务后清微任务；Node 11+ 已对齐，但 Node 还有自己的阶段（timers / poll / check 等）和 `setImmediate` / `process.nextTick`。

经典题：

```js
console.log(1)
setTimeout(() => console.log(2))
Promise.resolve().then(() => console.log(3))
console.log(4)
// 输出 1 4 3 2
```

不是"Promise 比 setTimeout 先"，而是同步代码（1、4）→ 微任务（3）→ 下一轮宏任务（2）。

### 说说你对事件循环的理解。

参见上一题。补一个稍微复杂点的例子，能讲清楚就过：

```js
async function a() {
  console.log('a start')
  await b()
  console.log('a end')          // 这行在 await 之后，会被包成微任务
}
async function b() { console.log('b') }

console.log('script start')
setTimeout(() => console.log('timeout'))
a()
new Promise(r => { console.log('promise'); r() }).then(() => console.log('then'))
console.log('script end')

// script start → a start → b → promise → script end
// → a end → then → timeout
```

`await` 的语义可以粗暴理解为：暂停当前 async 函数，把后续代码当作 `.then` 回调挂到 await 表达式返回的 Promise 上——也就是丢进微任务队列。所以 `a end` 和 `then` 都在微任务里执行。

### 什么是防抖和节流？有什么区别？如何实现？

```js
// 防抖：连续触发期间一直推迟，停了 delay 才执行最后一次
function debounce(fn, delay = 300) {
  let timer = null
  return function (...args) {
    clearTimeout(timer)
    timer = setTimeout(() => fn.apply(this, args), delay)
  }
}

// 节流：固定间隔最多触发一次
function throttle(fn, delay = 300) {
  let last = 0
  return function (...args) {
    const now = Date.now()
    if (now - last >= delay) {
      last = now
      fn.apply(this, args)
    }
  }
}
```

直观区别：连按 10 秒按钮，间隔 1 秒——

- 防抖：只在最后一次按下后 1 秒才执行 1 次
- 节流：每 1 秒执行 1 次，10 秒共 10 次

适用场景反过来记最快：

- 防抖：**只关心最后一次**——搜索联想、resize、表单字段实时校验、富文本自动保存
- 节流：**要持续响应但限频**——滚动加载、鼠标拖拽、按钮防连点

进阶版还会处理两个细节：

- **立即执行选项**：第一次触发就执行，后面才走防抖/节流
- **取消方法**：组件卸载时能取消未执行的定时器

```js
function debounce(fn, delay = 300, { leading = false } = {}) {
  let timer = null
  const debounced = function (...args) {
    if (timer) clearTimeout(timer)
    if (leading && !timer) fn.apply(this, args)
    timer = setTimeout(() => { timer = null; if (!leading) fn.apply(this, args) }, delay)
  }
  debounced.cancel = () => { clearTimeout(timer); timer = null }
  return debounced
}
```

工程里别自己造，lodash 的 `debounce` / `throttle` 这两个考虑得最周全，直接用。

### 如何实现上拉加载、下拉刷新？

```javascript
window.addEventListener('scroll', () => {
  const nearBottom = window.innerHeight + window.scrollY >= document.body.offsetHeight - 100
  if (nearBottom) {
    console.log('load more')
  }
})
```


上拉加载本质还是接近列表底部时触发下一页请求；下拉刷新本质还是触发重新获取第一页数据。

常见实现方式：

1. 监听滚动位置并判断是否到底。
2. 使用 `IntersectionObserver` 监听底部哨兵元素。
3. 移动端通过触摸事件或现成组件库实现下拉刷新。

项目里，难点不在“监听滚动”本身，而在状态控制。
因为滚动事件可能在短时间内连续触发，如果不控制并发，就会重复请求同一页数据；如果不维护分页状态和“是否加载完”的标记，就可能出现重复拼接、顺序错乱或无限请求。

### setTimeout、setInterval、requestAnimationFrame 有什么区别？

三个都是定时调度，但定位完全不同。

**`setTimeout(fn, delay)`**：延迟 `delay` 毫秒后跑一次。注意 `delay` 是"最小延迟"——主线程没空时回调要排队等，HTML5 规范还规定嵌套超过 5 层的 setTimeout 最小被钳到 4ms。

**`setInterval(fn, delay)`**：每隔 `delay` 跑一次，但**不管上次有没有跑完**。如果回调本身就要 200ms，间隔 100ms 写在那里也没用，且任务会堆积。生产代码更推荐"setTimeout 递归调用"模式：

```js
function tick() {
  doWork()
  setTimeout(tick, 100)   // 上一次跑完再约下一次，永远不堆积
}
setTimeout(tick, 100)
```

**`requestAnimationFrame(fn)`**：浏览器下一次重绘**之前**调一次，节奏对齐屏幕刷新率（一般 60Hz 即 16.6ms 一次，高刷屏更密）。优势：

- 跟渲染节奏同步，动画不掉帧、不抖
- 页面切到后台时浏览器会自动暂停，省电
- 比 `setTimeout(fn, 16)` 准得多

动画就该用 RAF，定时任务用 setTimeout/setInterval。补一个：要在浏览器空闲时跑非紧急任务（数据上报、预计算）用 `requestIdleCallback`，它会等浏览器忙完一帧才调你。

### 宏任务和微任务分别有哪些？

事件循环必考。补几个容易遗漏的：

**宏任务（Macrotask / Task）**：

- `script` 整体（页面初次同步代码）
- `setTimeout` / `setInterval`
- `setImmediate`（Node）
- I/O 回调、网络请求回调
- UI 事件（click、scroll 等回调）
- `MessageChannel`、`postMessage`
- `requestAnimationFrame`（严格说是渲染步骤里的，行为类似宏任务）

**微任务（Microtask）**：

- `Promise.then / catch / finally`
- `queueMicrotask(fn)`
- `MutationObserver` 回调
- `process.nextTick`（Node 独有，优先级比 Promise 还高）
- `await` 后面的代码（本质还是包成 Promise.then）

执行规则在上一题已经讲完——一个宏任务跑完，清空所有微任务，再渲染，再下一个宏任务。

工程上几个实际用处：

- 想让回调"尽快但异步"执行用 `queueMicrotask(fn)`，比 `setTimeout(fn, 0)` 快得多
- 想让 DOM 更新后再读取尺寸，用 `MutationObserver` 或 `Promise.resolve().then(...)`
- 想等下一帧渲染再操作，用 `requestAnimationFrame`

### async/await 和 Promise 是什么关系？

`async/await` 是 Generator + Promise 的语法糖，让异步代码读起来像同步。

几个关键点：

- **`async` 函数永远返回 Promise**——直接 `return 1` 也会被包成 `Promise.resolve(1)`，`throw` 会变成 `Promise.reject(err)`。
- **`await` 后面接什么都行**——是 Promise 就等它 resolve；不是就当 `Promise.resolve(value)` 处理。
- **`await` 会暂停当前 async 函数**，但**不会阻塞主线程**——函数从这里挂起，把后续代码作为微任务排队，控制权交回事件循环。等 Promise resolve 后再恢复。
- **错误处理走 try/catch**——比 `.catch()` 直观，但一个忘了的 try 就会变成未处理的 Promise rejection。

```js
async function loadAll() {
  // 串行：3 个请求挨个等
  const a = await fetch('/a')
  const b = await fetch('/b')
  const c = await fetch('/c')
  return [a, b, c]
}

async function loadAllParallel() {
  // 并行：3 个请求一起发，一起等
  const [a, b, c] = await Promise.all([fetch('/a'), fetch('/b'), fetch('/c')])
  return [a, b, c]
}
```

`Promise.all` 配合 await 是处理并发的标准写法。如果不在乎个别失败，用 `Promise.allSettled`。

容易出现的问题：

- **for 循环里 await** 会让循环串行，每次等上一个完成。要并发请改成 `await Promise.all(arr.map(asyncFn))`。
- **forEach 里 await 不生效**——因为 forEach 的回调返回值会被忽略。要等就用 `for...of`。
- **顶层 await**：ES2022 起，原生 ES Module 模块顶层可以直接写 `await`，但会让整个模块及其依赖变成异步加载。

### 什么是函数组合？有什么应用场景？

```javascript
const compose = (...fns) => value => fns.reduceRight((acc, fn) => fn(acc), value)
const add1 = x => x + 1
const double = x => x * 2
console.log(compose(double, add1)(3)) // 8
```


函数组合指的是把多个小函数按顺序组合起来，形成一个新的函数，让数据依次流过这些函数。

它的价值在于：

1. 让复杂逻辑拆成多个单一职责函数。
2. 提高复用性和可测试性。
3. 让数据处理链路更清晰。

常见用法：

1. 表单数据预处理。
2. 中间件链路。
3. 数据格式转换流水线。
4. 函数式工具库实现。

函数组合就是把复杂逻辑拆成几个小函数，再按顺序拼起来。
这样做的价值不只是“写法优雅”，而是每个小函数都更容易测试、复用和替换，最终降低单个大函数过度膨胀带来的维护成本。

### JavaScript 中如何实现函数缓存？函数缓存有哪些应用场景？

```javascript
function memo(fn) {
  const cache = new Map()
  return function (...args) {
    const key = JSON.stringify(args)
    if (cache.has(key)) return cache.get(key)
    const result = fn.apply(this, args)
    cache.set(key, result)
    return result
  }
}
```


函数缓存也叫记忆化，思路是：相同输入直接返回上次计算结果，避免重复计算。

实现思路：

1. 用对象或 `Map` 记录参数和结果的映射。
2. 每次执行前先查缓存。
3. 命中则直接返回，未命中再计算并存储。

用在哪：

1. 复杂计算结果复用。
2. 重复接口请求的本地去重。
3. 递归优化。
4. 模板编译缓存。

### 观察者模式和发布订阅模式有什么区别？

```javascript
class Subject {
  constructor() { this.list = [] }
  add(fn) { this.list.push(fn) }
  notify(data) { this.list.forEach(fn => fn(data)) }
}
```


这两个模式都和“事件通知”有关，但耦合关系不同。

观察者模式通常是：

1. 观察者直接订阅目标对象。
2. 目标对象状态变化后直接通知观察者。

发布订阅模式通常是：

1. 发布者和订阅者之间通过事件中心或消息中心解耦。
2. 发布者不直接感知订阅者是谁。

粗略区分：观察者是“对象对对象”的依赖通知，发布订阅是“通过中介统一转发”的消息机制。

### 说说 JavaScript 数字精度丢失的问题，如何解决？

```javascript
console.log(0.1 + 0.2) // 0.30000000000000004
console.log((0.1 * 10 + 0.2 * 10) / 10) // 0.3
```


JavaScript 中 `Number` 采用 IEEE 754 双精度浮点数表示，因此某些小数计算会出现精度误差，比如 `0.1 + 0.2 !== 0.3`。

解决思路：

1. 对金额等场景转成整数运算，比如“元转分”。
2. 使用 `toFixed` 做展示层格式化，但它不一定适合做精确计算。
3. 使用高精度运算库，比如 `decimal.js`、`big.js`。
4. 超大整数可以考虑 `BigInt`，但它不处理小数。

### 大文件上传如何做断点续传？

```javascript
function createChunks(file, size = 1024 * 1024) {
  const chunks = []
  for (let i = 0; i < file.size; i += size) {
    chunks.push(file.slice(i, i + size))
  }
  return chunks
}
```


大文件断点续传基本都是围绕分片上传来做。

一般流程：

1. 前端把文件切成多个切片。
2. 每个切片携带文件标识和分片索引上传。
3. 服务端记录已上传分片。
4. 中断后重新上传时，只补传未完成分片。
5. 最后服务端合并分片。

工程上通常还会配合：

1. 文件 hash 校验
2. 秒传
3. 并发上传
4. 上传进度展示

### Web 常见的攻击方式有哪些？如何防御?

Web 安全问题可以归纳为几类常见风险：

1. XSS
2. CSRF
3. 点击劫持
4. 文件上传
5. 原型污染 / 依赖投毒

这些问题表现不同，但处理思路比较统一：

1. 输入不要轻信
2. 输出不要直接执行
3. 权限和凭证不要自动信任
4. 风险点要分层拦截

常见问题不用摊太开，记住几类高频项即可：

1. XSS：把输入当脚本执行了
2. CSRF：借浏览器自动带凭证发起伪造请求
3. 点击劫持：把用户点击引到错误位置
4. 文件上传 / SSRF / 原型污染：都属于边界校验没守住

**1. XSS**

本质还是把用户输入当成脚本执行了。

```javascript
// ❌ 危险:用户输入直接拼到 HTML
el.innerHTML = `<div>${user.name}</div>`
// 输入 <img src=x onerror=alert(document.cookie)> 就执行了

// ✅ 安全:文本接口
el.textContent = user.name

// ✅ 富文本场景用白名单库
import DOMPurify from 'dompurify'
el.innerHTML = DOMPurify.sanitize(html)
```

处理这种问题，关键是按上下文做输出转义，不要把用户输入直接塞进 `innerHTML`。再往上一层，就是 `HttpOnly` 和 CSP 这些防护。

**2. CSRF**

本质还是借浏览器自动带 Cookie 这件事，伪造用户请求。

```html
<!-- 攻击站点 -->
<img src="https://bank.com/transfer?to=hacker&amount=10000">
```

处理方式通常就是：

1. SameSite Cookie
2. CSRF Token
3. 关键操作二次验证
4. 不要让 GET 承担改数据动作

**3. 点击劫持**

```http
# 响应头加上,禁止被嵌套
X-Frame-Options: DENY
Content-Security-Policy: frame-ancestors 'none'
```

**4. 原型污染**

```javascript
// ❌ 危险的 merge 实现
function merge(a, b) {
  for (const k in b) a[k] = b[k]   // 可被 __proto__ 污染
}
const evil = JSON.parse('{"__proto__": {"isAdmin": true}}')
merge({}, evil)
console.log({}.isAdmin)            // true,所有对象都中招

// ✅ 防御
Object.create(null)                // 原型为 null,无法被污染
Object.freeze(Object.prototype)    // 冻结 Object.prototype
```

如果最后压成一组通用原则，主要是：

1. 输入校验
2. 输出转义
3. Cookie 安全属性
4. CSP / HTTPS
5. 依赖治理
6. 最小权限

### 跨域的原因及解决方案

跨域问题来自浏览器的同源策略。协议、域名、端口只要有一个不同，就不算同源。

这种问题容易误解的点有三个：

1. 同源策略只限制浏览器，不限制服务端互访
2. 不少情况下请求实际已经发出去了，只是响应不给前端 JS 读
3. 像 `img`、`script`、`link` 这类标签天然就能跨域加载资源

真正高频的方案也不多：

1. 生产环境一般就是 CORS 或反向代理
2. 开发环境一般靠 dev server proxy
3. `postMessage` 解决的是窗口 / iframe 通信，不是普通接口跨域
4. JSONP 现在更多只是历史知识点

生产环境最主流的还是 CORS 和反向代理。开发环境则基本靠 dev server proxy。

**方案 1：CORS**

```nginx
# 服务端响应头
Access-Control-Allow-Origin: https://a.com           # 允许的来源(不要写 * 配 cookie)
Access-Control-Allow-Credentials: true               # 允许带 cookie
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400                        # 预检结果缓存
```

前端如果要带 cookie，还要额外加上：

```javascript
// 默认不带 cookie,跨域要带必须设
fetch('/api', { credentials: 'include' })
axios.defaults.withCredentials = true
```

**方案 2：Nginx 反向代理**

这种方案的思路更直接：把跨域问题消化在网关层，浏览器看到的始终是同源。

```nginx
server {
  listen 80;
  server_name a.com;
  location /api/ {
    proxy_pass http://backend.b.com:3000/;     # 浏览器认为还在 a.com
  }
}
```

详见 `14-Git与DevOps面试题.md` 的 Nginx 配置题。

**方案 3：开发态 dev-server proxy**

```javascript
// vite.config.js
export default {
  server: {
    proxy: {
      '/api': {
        target: 'http://backend:3000',
        changeOrigin: true,
        rewrite: p => p.replace(/^\/api/, ''),
      },
    },
  },
}
```

这个方案只在开发服务器里有效，上线后还是要交给 Nginx 或 CORS。

**方案 4：JSONP**

```javascript
function jsonp(url, callbackName) {
  window[callbackName] = data => console.log(data)
  const s = document.createElement('script')
  s.src = `${url}?callback=${callbackName}`
  document.body.appendChild(s)
}
// 服务端返回:callbackName({...})
```

它只支持 GET，也有明显安全和维护问题，现在更多是老系统遗留知识点。

**方案 5：postMessage**

主页面与 iframe / 弹窗 / Worker 之间通信。

```javascript
// 父页面
const iframe = document.querySelector('iframe')
iframe.contentWindow.postMessage({ type: 'hi' }, 'https://child.com')

// iframe 内
window.addEventListener('message', e => {
  if (e.origin !== 'https://parent.com') return     // 必须校验来源
  console.log(e.data)
})
```

这里容易出问题的地方，是接收方一定要校验 `origin`，否则消息链路本身就可能变成安全问题。

压短一点就是：

1. 生产优先代理或正规 CORS
2. 开发用 proxy 模拟真实环境
3. 别再把 JSONP 当常规方案

### 什么是 JSONP？它的原理和局限性是什么？

```javascript
function jsonp(url, callbackName) {
  const script = document.createElement('script')
  script.src = `${url}?callback=${callbackName}`
  document.body.appendChild(script)
}
```


JSONP 是前端早期处理跨域的一种办法，本质就是借用了 `script` 标签不受同源策略限制这个特性。

原理大致是：

1. 前端动态创建 `script` 标签。
2. 请求服务端接口。
3. 服务端返回一段函数调用代码。
4. 浏览器加载脚本后直接执行，这样可以把数据传回前端。

它的局限性明显：

1. 只支持 GET 请求。
2. 安全性较差。
3. 现代项目中基本被 CORS 替代。

### XSS 和 CSRF 的区别是什么？

这是前端安全中的高频对比题。

**XSS：**

1. 本质还是恶意脚本被注入页面并执行
2. 攻击目标通常是用户浏览器环境
3. 常见防御：
   - 输入过滤
   - 输出转义
   - CSP
   - HttpOnly Cookie

**CSRF：**

1. 本质还是诱导用户在已登录状态下发起非本人意愿请求
2. 重点利用的是浏览器自动带 Cookie
3. 常见防御：
   - CSRF Token
   - SameSite Cookie
   - Referer / Origin 校验

这两个攻击经常被混淆，因为它们都和“用户已登录”场景相关，但攻击路径完全不同：

1. XSS 的关键是页面里混入了恶意脚本，攻击者拿到了在你页面上下文里执行代码的能力
2. CSRF 的关键是浏览器会自动带上身份凭证，攻击者利用这一点诱导用户替自己发请求

所以一个重点防“脚本执行”，一个重点防“请求伪造”。

### JavaScript 中常见的垃圾回收机制有哪些？

这里一般不用展开到虚拟机源码级别，但基本思路要清楚。

常见垃圾回收机制有：

1. **引用计数**
   - 记录对象被引用次数
   - 引用数为 0 时回收
   - 缺点是难处理循环引用

2. **标记清除**
   - 现代 JavaScript 引擎主流方案
   - 从根对象出发标记可达对象
   - 没被标记的对象会被回收

3. **分代回收**
   - 新生代和老生代分开处理
   - 因为大多数对象生命周期很短

之所以现代引擎普遍采用“标记清除 + 分代回收”，是因为真实程序里的对象生命周期分布很不均匀：
大量临时对象创建后很快就失效，少量对象会长期存活。
按代处理能显著减少每次 GC 扫描的范围，这样可以在回收效率和停顿时间之间取得更好的平衡。

### ES6 中扩展运算符和 rest 参数怎么理解？有什么应用场景？

```javascript
const arr1 = [1, 2]
const arr2 = [...arr1, 3]
function sum(...nums) {
  return nums.reduce((a, b) => a + b, 0)
}
```


两者都写作 `...`，但使用位置和语义不同。

1. 扩展运算符用于“展开”，比如把数组、可迭代对象拆开。
2. rest 参数用于“收集”，把多个参数收集成一个数组。

常用场景：

1. 数组合并。
2. 参数收集。
3. 浅拷贝。
4. 把类数组转为数组。
5. 对象属性展开。

这里容易混淆，是因为扩展运算符和 rest 参数都写成 `...`，但它们解决的是两个方向相反的问题。

1. 扩展运算符是在“把一个整体拆开”
2. rest 参数是在“把多个零散值收集起来”

如果不把这个“展开”和“收集”的方向区别说清楚，代码一多就容易理解错。

### ES6 中解构赋值怎么理解？有什么应用场景？

### ES6 中 Iterator 和 `for...of` 怎么理解？

```javascript
const arr = [1, 2, 3]
for (const item of arr) {
  console.log(item)
}
```


Iterator 即迭代器，是一种统一的遍历接口。

思路是：

1. 通过 `next()` 按顺序返回数据。
2. 每次返回 `{ value, done }`。

`for...of` 则是基于 Iterator 协议进行遍历的语法。

常见可迭代对象有：

1. Array
2. String
3. Map
4. Set
5. Generator 返回值

这两者的差异不是“语法不同”那么简单，而是底层遍历对象不同：

1. `for...in` 走的是对象可枚举属性枚举逻辑，所以更适合看 key
2. `for...of` 走的是迭代器协议，所以更适合拿 value

数组更接近一个有序值集合，所以遍历数组时 `for...of` 通常比 `for...in` 更符合语义。

### ES6 中新增的 Set、Map 两种数据结构怎么理解？

```javascript

const map = new Map([['name', 'Tom']])
console.log([...set])
console.log(map.get('name'))
```


1. `Set` 类似“值的集合”，成员唯一，常用于数组去重。
2. `Map` 是“键值对集合”，键可以是任意类型。

相较于对象：

1. `Map` 的键更灵活。
2. `Map` 更适合频繁增删查。
3. `Set` 更适合处理去重和集合运算。

### WeakMap 和 WeakSet 怎么理解？和 Map、Set 有什么区别？

```javascript
const wm = new WeakMap()
const obj = {}
wm.set(obj, 'data')
console.log(wm.get(obj))
```


WeakMap 和 WeakSet 可以看成“弱引用版本”的 Map 和 Set。

主要区别：

1. WeakMap 的键只能是对象。
2. WeakSet 的成员只能是对象。
3. 它们持有的是弱引用，不会阻止对象被垃圾回收。
4. 一般不能像普通 Map/Set 那样遍历。

常见用法：

1. 给对象附加元数据。
2. 缓存对象相关信息，又不希望影响垃圾回收。
3. 避免因缓存导致内存泄漏。

### 你是怎么理解 ES6 中 Promise 的？使用场景是什么？

```javascript
const p = new Promise(resolve => {
  setTimeout(() => resolve('ok'), 1000)
})
p.then(console.log)
```


Promise 是异步编程的一种解决方案，用来表示一个未来才会完成的操作结果。

它有三种状态：

1. `pending`
2. `fulfilled`
3. `rejected`

特点大概是：

1. 状态一旦改变就不可逆。
2. 支持链式调用。
3. 可以统一处理异步成功和失败。

常见用法：

1. 网络请求
2. 异步流程编排
3. 多个异步任务汇总，如 `Promise.all`

### Promise.all、Promise.race、Promise.allSettled 有什么区别？

```javascript
Promise.all([Promise.resolve(1), Promise.resolve(2)]).then(console.log)
Promise.race([
  new Promise(r => setTimeout(() => r('fast'), 100)),
  new Promise(r => setTimeout(() => r('slow'), 200))
]).then(console.log)
Promise.allSettled([Promise.resolve(1), Promise.reject('err')]).then(console.log)
```


这几个都是 Promise 的静态方法，但聚合策略不同。

1. `Promise.all`：全部成功才成功，一个失败就整体失败。
2. `Promise.race`：谁先结束就采用谁的结果。
3. `Promise.allSettled`：等待所有 Promise 都结束，并返回每个结果状态。

常用场景：

1. `all`：多个接口都成功后再继续。
2. `race`：超时控制、抢最快结果。
3. `allSettled`：批量请求后统一查看成功和失败情况。

### 怎么理解 ES6 中 Generator？使用场景是什么？

```javascript
function* gen() {
  yield 1
  yield 2
  return 3
}
const g = gen()
console.log(g.next())
console.log(g.next())
console.log(g.next())
```


Generator 是一种可暂停、可恢复执行的函数，通过 `function*` 定义，通过 `yield` 暂停。

它的特点大概是：

1. 可以分步返回结果。
2. 可以手动控制执行过程。
3. 返回一个迭代器对象。

用在哪：

1. 自定义迭代逻辑
2. 异步流程控制的早期方案
3. 配合 `co` 等库组织异步代码

现在业务里，不少异步控制场景已被 `async/await` 替代，但理解 Generator 对理解迭代器和异步演进过程重要。

### 你是怎么理解 ES6 中 Proxy 的？使用场景是什么？

```javascript
const obj = {}
const proxy = new Proxy(obj, {
  get(target, key) {
    console.log('get', key)
    return target[key]
  },
  set(target, key, value) {
    console.log('set', key, value)
    target[key] = value
    return true
  }
})
```


`Proxy` 用于拦截并自定义对象的基本操作，比如读取、赋值、删除、函数调用等。

适合用在：

1. 数据响应式
2. 权限控制
3. 数据校验
4. 日志埋点
5. 代理默认值

这里经常会联系到框架应用，因为 `Proxy` 不只是语法层面的“代理对象”，它真正有用的地方在于可以拦截读取、赋值、删除等基础操作。
像 Vue3 这样的响应式系统，正是利用这种拦截能力在数据读写时做依赖收集和更新派发，所以它比单纯讲概念更能体现 `Proxy` 的工程价值。

### Reflect 是什么？它和 Proxy 有什么关系？

```javascript
const obj = { name: 'Tom' }
console.log(Reflect.get(obj, 'name'))
Reflect.set(obj, 'age', 18)
console.log(Reflect.ownKeys(obj))
```


Reflect 可以看成把对象操作的一些底层能力，统一放到一个内置对象上提供。

常见方法有：

1. `Reflect.get`
2. `Reflect.set`
3. `Reflect.has`
4. `Reflect.deleteProperty`
5. `Reflect.ownKeys`

它和 Proxy 的关系很紧密：

1. Proxy 负责拦截。
2. Reflect 负责把默认行为优雅地转发出去。

所以不少 Proxy handler 内部都会配合 Reflect 使用，这样语义更清晰，也更接近默认对象行为。

### 你是怎么理解 ES6 中 Module 的？使用场景是什么？

```javascript
// a.js
export const name = 'Tom'

// b.js
import { name } from './a.js'
console.log(name)
```


ES Module 是 JavaScript 官方模块化方案，核心能力是把代码拆分成独立模块，并通过 `import` / `export` 组织依赖。

它的特点大概是：

1. 静态分析友好
2. 支持 Tree Shaking
3. 作用域隔离
4. 模块依赖关系清晰

使用场景：

1. 前端工程化开发
2. 代码拆分和复用
3. 构建工具优化打包结果

### 你是怎么理解 ES6 中 Decorator 的？使用场景是什么？

```javascript

  descriptor.writable = false
  return descriptor
}
```


Decorator 说到底就是给类、方法、属性再包一层增强逻辑。

它适合做：

1. 日志埋点
2. 权限校验
3. 元数据注入
4. AOP 风格扩展

不过要说明：Decorator 在 JavaScript 里经历过多次提案变化，实际使用更多见于 TypeScript、Angular、NestJS 等生态中。

### ES6 中 class 怎么理解？和构造函数有什么区别？

`class` 归根结底还是基于原型继承，只是外面包了一层更像传统面向对象语言的语法。

它带来的好处：

1. 写法更接近传统面向对象语言
2. 继承表达更清晰
3. 方法组织更直观

和构造函数的区别主要在于：

1. `class` 必须用 `new` 调用
2. 类中的方法默认定义在原型上
3. 类声明存在暂时性死区，不像函数声明那样完全提升可用

可以记成：

> `class` 没有改变 JavaScript 基于原型的本质，只是提供了更清晰的面向对象语法。

### ES6 中模板字符串有什么优势？

模板字符串使用反引号定义，主要优势有：

1. 支持字符串插值 `${expr}`
2. 支持多行字符串
3. 可读性更好
4. 可以配合标签模板做更高级处理

常见场景：

1. 拼接 HTML 字符串
2. 生成日志文本
3. 多行文本模板

### 可选链 ?. 和空值合并 ?? 怎么理解？

这是现代 JavaScript 很高频的新语法。

1. `?.`
   - 用于安全访问可能不存在的属性
   - 如果前面的值是 `null` 或 `undefined`，直接返回 `undefined`

2. `??`
   - 只有左侧是 `null` 或 `undefined` 时，才返回右侧默认值

和 `||` 的区别：

1. `||` 会把 `0`、`''`、`false` 也当成假值
2. `??` 只处理空值

例如：

```javascript
const count = 0
console.log(count || 10) // 10
console.log(count ?? 10) // 0
```

### `Object.defineProperty` 和 `Proxy` 的区别是什么？

`Object.defineProperty` 更像“拦截某个已有属性”，而 `Proxy` 更像“代理整个对象行为”。

核心差异：

1. `defineProperty` 需要逐个属性处理
2. `Proxy` 可以统一拦截读、写、删、枚举等操作
3. `Proxy` 对新增属性和数组下标变化这类场景更自然

所以不少现代响应式系统更倾向 `Proxy`。

### 什么是属性描述符（Property Descriptor）？

属性描述符是 JavaScript 描述对象属性行为的一组元信息。

常见字段有：

1. `value`
2. `writable`
3. `enumerable`
4. `configurable`
5. `get`
6. `set`

它重要的原因在于：对象属性不只有“有没有这个值”，还有“能不能改、能不能删、能不能枚举、是不是访问器属性”这些行为层定义。

### 原型上的属性和实例自己的属性有什么区别？

实例自己的属性直接存在对象本身上；原型属性存在于它的原型对象上，实例通过原型链访问到它。

这两者的区别会影响：

1. `hasOwnProperty`
2. 枚举行为
3. 读写覆盖逻辑
4. 内存复用方式

### 1. 简易防抖

```javascript
function debounce(fn, delay = 300) {
  let timer = null;
  return function (...args) {
    clearTimeout(timer)
    timer = setTimeout(() => fn.apply(this, args), delay)
  }
}
```

### 2. 简易节流

```javascript
function throttle(fn, delay = 300) {
  let last = 0
  return function (...args) {
    const now = Date.now()
    if (now - last >= delay) {
      last = now
      fn.apply(this, args)
    }
  }
}
```

### 3. 手写深拷贝

```javascript
function deepClone(obj, map = new WeakMap()) {
  if (obj === null || typeof obj !== 'object') return obj
  if (map.has(obj)) return map.get(obj)
  const result = Array.isArray(obj) ? [] : {}
  map.set(obj, result)
  for (const key in obj) {
    if (Object.prototype.hasOwnProperty.call(obj, key)) {
      result[key] = deepClone(obj[key], map)
    }
  }
  return result
}

/**
  1. 先拷贝外壳
  2. 再一层层处理内部属性
  3. 遇到对象就继续展开
  4. 遇到基础值就直接复制
  5. 用 Map 防止重复拷贝和循环引用
*/
function deepCloneIterative(source) {
    if (source === null || typeof source !== 'object') return source
    const root = Array.isArray(source) ? [] : {}
    const map = new Map()
    map.set(source, root)

    const stack = [
      { source, target: root }
    ]
    while (stack.length) {
      const { source, target } = stack.pop()
      for (const key in source) {
        if (!Object.prototype.hasOwnProperty.call(source, key)) continue
        const value = source[key]
        if (value !== null && typeof value === 'object') {
          if (map.has(value)) {
            target[key] = map.get(value)
          } else {
            const clone = Array.isArray(value) ? [] : {}
            target[key] = clone
            map.set(value, clone)
            stack.push({ source: value, target: clone })
          }
        } else {
          target[key] = value
        }
      }
    }

    return root
  }
```

### 4. 手写发布订阅

```javascript
class EventBus {
  constructor() {
    this.events = {}
  }
  on(name, fn) {
    ;(this.events[name] ||= []).push(fn)
  }
  emit(name, ...args) {
    ;(this.events[name] || []).forEach(fn => fn(...args))
  }
  off(name, fn) {
    this.events[name] = (this.events[name] || []).filter(item => item !== fn)
  }
}
```
