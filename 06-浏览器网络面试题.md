## 一、网络基础与分层模型

### 什么是 HTTP？HTTP 和 HTTPS 的区别是什么？

HTTP 是超文本传输协议，用于浏览器和服务器之间传输数据。

HTTP 和 HTTPS 的主要区别：

1. HTTP 明文传输，HTTPS 在 HTTP 基础上加入 TLS/SSL 加密。
2. HTTP 默认端口是 80，HTTPS 默认端口是 443。
3. HTTPS 能提供加密、身份认证和数据完整性校验。
4. HTTPS 部署需要证书，成本和握手开销更高。

一句话总结：HTTPS 是更安全的 HTTP，而不是新的应用层协议替代品。

### 如何理解 OSI 七层模型？

OSI 七层模型是一个理论分层模型，用来说明网络通信中各层职责。

从上到下分别是：

1. 应用层
2. 表示层
3. 会话层
4. 传输层
5. 网络层
6. 数据链路层
7. 物理层

面试中常见说法是：实际工程里更常参考 TCP/IP 模型，但 OSI 七层有助于理解各层职责边界。

### 如何理解 TCP/IP 协议族？

TCP/IP 更接近真实互联网实现，通常可分为五层：

1. 应用层
2. 传输层
3. 网络层
4. 数据链路层
5. 物理层

它不是单个协议，而是一组协议的集合。

比如：

1. 应用层有 HTTP、DNS、WebSocket
2. 传输层有 TCP、UDP
3. 网络层有 IP

### 如何理解 UDP 和 TCP？区别是什么？应用场景有哪些？

TCP 和 UDP 都属于传输层协议。

TCP 的特点：

1. 面向连接
2. 可靠传输
3. 有序
4. 有流量控制和拥塞控制
5. 开销更大

UDP 的特点：

1. 无连接
2. 不保证可靠
3. 开销小
4. 传输快
5. 更适合实时场景

应用场景：

1. TCP：网页请求、文件传输、邮件
2. UDP：音视频通话、直播、游戏、DNS 查询

### TCP 为什么需要三次握手和四次挥手？

三次握手的目的是让通信双方都确认“自己能发、也能收”，同时同步初始序列号。

为什么不是两次：两次无法充分确认双方收发能力，也可能导致历史连接请求误建立。

四次挥手的原因是 TCP 是全双工通信：

1. 一方发送 FIN，只代表自己不再发送数据。
2. 对方仍然可能还有数据没发完，所以需要单独确认和关闭。

所以关闭连接通常需要双方各发送一次关闭和确认。

### TCP 的可靠传输机制有哪些？

TCP 能保证可靠传输，主要依赖：

1. 序列号和确认应答
2. 超时重传
3. 校验和
4. 流量控制
5. 拥塞控制
6. 有序重组

面试时可以概括为：TCP 的可靠不是靠单一机制，而是一整套传输控制体系共同保证的。

### TCP 的流量控制和拥塞控制分别是什么？

两者目标不同：

1. 流量控制解决“接收方来不及处理”的问题。
2. 拥塞控制解决“网络整体太拥堵”的问题。

流量控制主要依赖滑动窗口。

拥塞控制常见机制包括：

1. 慢启动
2. 拥塞避免
3. 快重传
4. 快恢复

### TCP 粘包是怎么回事，如何处理？为什么 UDP 不会粘包？

TCP 是面向字节流的协议，没有消息边界，所以接收方读到的数据可能把多次发送的数据合并，或者一次发送的数据拆开，这就是粘包和拆包问题。

常见处理方式：

1. 固定消息长度
2. 在消息头中携带长度字段
3. 使用特殊分隔符

UDP 不会粘包，是因为 UDP 是面向报文的，每个数据包天然有边界。

## 二、HTTP 协议与请求响应

### 说说 HTTP1.0、HTTP1.1、HTTP2 的区别。

HTTP1.0、1.1、2 的主要区别可以从传输效率看：

1. HTTP1.0 默认短连接，请求完成后连接关闭。
2. HTTP1.1 支持长连接、管道化、更多缓存控制。
3. HTTP2 引入二进制分帧、多路复用、头部压缩、服务器推送。

常见总结：

1. HTTP1.1 解决了一部分连接复用问题。
2. HTTP2 重点解决队头阻塞和冗余头部问题。

### 说一下 GET 和 POST 的区别。

常见区别有：

1. 语义不同：GET 用于获取资源，POST 常用于提交数据。
2. 参数位置不同：GET 常放在 URL，POST 常放在请求体。
3. GET 更容易被缓存和收藏。
4. GET 有长度限制通常是浏览器或服务器限制，不是协议强制。
5. POST 不一定更安全，若不用 HTTPS 也可能被窃听。

面试时要强调：两者最本质的区别是语义和幂等性，而不是“一个安全一个不安全”。

### GET 方法 URL 长度限制的原因是什么？

严格来说，HTTP 协议本身并没有强制规定 GET 的 URL 必须多短。

所谓长度限制，更多来自这些层面：

1. 浏览器实现限制。
2. 服务器或网关配置限制。
3. 历史兼容性因素。
4. URL 本身不适合承载过大数据。

所以这道题的关键点是：URL 长度限制通常不是 HTTP 协议硬性规定，而是客户端、服务端和中间层共同形成的现实约束。

### POST 和 PUT 的区别是什么？

两者都可以用于提交数据，但语义不同：

1. POST 更常用于创建资源或触发动作。
2. PUT 更强调对指定资源做整体更新。
3. PUT 通常被认为是幂等的，重复执行结果一致。
4. POST 通常不是幂等的。

### HTTP 常见的请求头有哪些？作用是什么？

常见请求头有：

1. `Host`：目标主机
2. `User-Agent`：客户端信息
3. `Accept`：可接收的数据类型
4. `Content-Type`：请求体类型
5. `Authorization`：认证信息
6. `Cookie`：会话信息
7. `Referer`：来源页面
8. `Origin`：请求源
9. `Cache-Control`：缓存控制

面试里不要只背名字，要知道它们和认证、缓存、跨域、安全的关系。

### Cookie 和 Set-Cookie 有什么区别？

两者都和会话管理有关，但方向不同：

1. `Set-Cookie` 是服务端通过响应头下发 Cookie。
2. `Cookie` 是浏览器后续请求时通过请求头携带 Cookie。

可以理解为：

1. 服务端“发 Cookie”用 `Set-Cookie`
2. 浏览器“带 Cookie”用 `Cookie`

### HTTP 常见的状态码有哪些？适用场景是什么？

常见状态码可以按类别记忆：

1. `2xx`：成功，如 `200`、`204`、`206`
2. `3xx`：重定向，如 `301`、`302`、`303`、`304`、`307`
3. `4xx`：客户端错误，如 `400`、`401`、`403`、`404`、`405`
4. `5xx`：服务端错误，如 `500`、`502`、`503`、`504`

面试高频点：

1. `401` 表示未认证。
2. `403` 表示已认证但无权限。
3. `304` 表示资源未修改，常用于协商缓存。

### 307、303、302 的区别是什么？

它们都属于重定向，但对请求方法的处理不同。

1. `302` 在历史实现中可能把 POST 变成 GET，行为不够稳定。
2. `303` 明确要求客户端使用 GET 去请求新地址。
3. `307` 明确要求保持原请求方法和请求体不变。

所以如果需要“临时重定向且保持方法不变”，应优先考虑 `307`。

### HTTP 请求报文和响应报文是什么样的？

HTTP 请求报文通常包括：

1. 请求行：方法、URL、协议版本
2. 请求头
3. 空行
4. 请求体

HTTP 响应报文通常包括：

1. 状态行：协议版本、状态码、状态描述
2. 响应头
3. 空行
4. 响应体

这道题核心是理解 HTTP 是一种“文本格式的请求-响应协议”。

### HTTP 常见的请求方法有哪些？OPTIONS 的使用场景是什么？

常见方法包括：

1. GET
2. POST
3. PUT
4. DELETE
5. PATCH
6. OPTIONS
7. HEAD

`OPTIONS` 常见场景：

1. 询问服务器支持哪些方法
2. CORS 预检请求

前端面试里，`OPTIONS` 最常和跨域预检联系在一起。

### 对 keep-alive 的理解是什么？

`keep-alive` 指保持 TCP 连接复用，避免每次请求都重新建立连接。

它的价值：

1. 减少握手开销
2. 降低延迟
3. 提升多资源加载效率

HTTP1.1 中长连接是默认行为，`keep-alive` 思想对性能优化很重要。

### HTTP2 的头部压缩算法是怎样的？

HTTP2 使用 HPACK 算法对请求头和响应头进行压缩。

核心思路：

1. 建立静态表和动态表
2. 重复字段不再完整传输
3. 通过索引和霍夫曼编码减少体积

面试里知道“HPACK + 静态表/动态表 + 减少重复头部传输”就够了。

### 说一下 HTTP3。

HTTP3 基于 QUIC，而 QUIC 基于 UDP。

它的核心改进：

1. 避免 TCP 层队头阻塞
2. 更快建立连接
3. 更适合弱网和移动网络
4. 把传输层和安全能力整合得更紧密

一句话总结：HTTP3 是在 HTTP2 之上继续优化传输层体验，而不只是继续堆协议特性。

### HTTP 协议的优点和缺点是什么？

优点：

1. 简单，易于理解和实现。
2. 灵活，扩展性强。
3. 基于请求-响应模型，应用广泛。
4. 与浏览器和 Web 生态深度结合。

缺点：

1. 无状态，需要额外机制维持会话。
2. 明文 HTTP 不安全。
3. 早期版本存在队头阻塞、头部冗余等问题。
4. 需要配合缓存、压缩、长连接等机制才能获得更好性能。

### HTTP 协议的性能怎么样？

HTTP 的性能不能只看协议名字，而要结合版本、网络环境和优化手段一起看。

影响性能的关键点包括：

1. 连接是否可复用。
2. 头部是否冗余。
3. 是否存在队头阻塞。
4. 是否启用了压缩、缓存、CDN、长连接等优化。

一般可以这样总结：

1. HTTP1.1 比 HTTP1.0 明显更适合现代 Web。
2. HTTP2 在多路复用和头部压缩上提升很大。
3. HTTP3 继续针对弱网和传输层问题做优化。

### URL 有哪些组成部分？

一个完整 URL 一般包括：

1. 协议
2. 域名
3. 端口
4. 路径
5. 查询参数
6. 哈希值

例如：`https://example.com:443/user?id=1#info`

### 端口号的作用是什么？

端口号用于区分一台主机上不同的网络服务。

简单理解：IP 用来找到机器，端口用来找到机器上的具体应用。

## 三、HTTPS、TLS 与安全通信

### 为什么说 HTTPS 比 HTTP 安全？HTTPS 是如何保证安全的？

HTTPS 比 HTTP 安全，主要因为它提供了三种能力：

1. 加密：防止传输内容被窃听。
2. 身份认证：确认访问的是正确服务器。
3. 完整性校验：防止数据被篡改。

它的安全基础来自 TLS，包括对称加密、非对称加密、摘要算法和数字证书。

### TLS/SSL 的工作原理是什么？

TLS 的核心目标是让通信双方安全协商出一个会话密钥。

大致过程：

1. 客户端发起握手，告知支持的加密套件。
2. 服务端返回证书和协商参数。
3. 客户端验证证书合法性。
4. 双方协商出会话密钥。
5. 后续通信主要使用对称加密。

面试时可以强调：非对称加密主要用于安全协商密钥，对称加密用于后续高效传输。

### 数字证书是什么？

数字证书本质上是由权威 CA 机构签发的一份身份证明，用来证明“这个公钥确实属于这个网站”。

它通常包含：

1. 证书持有者信息
2. 公钥
3. 签发机构
4. 有效期
5. 数字签名

客户端通过验证证书链和签名，判断服务器身份是否可信。

### HTTPS 通信握手过程是什么？

简化后的握手过程可以这样回答：

1. 客户端发起 HTTPS 请求。
2. 服务端返回证书和支持的加密信息。
3. 客户端验证证书。
4. 客户端生成随机信息并安全传给服务端。
5. 双方生成会话密钥。
6. 后续使用对称加密通信。

### 什么是中间人攻击？如何防范？

中间人攻击是指攻击者插入客户端和服务器之间，窃听、篡改或伪造通信内容。

常见防范方式：

1. 使用 HTTPS。
2. 正确验证证书。
3. 避免忽略浏览器证书告警。
4. 使用 HSTS。
5. 在高安全场景下做证书绑定等增强措施。

## 四、DNS、CDN 与请求路径

### DNS 协议是什么？说说 DNS 完整的查询过程。

DNS 是域名解析协议，作用是把域名转换成 IP 地址。

常见查询过程：

1. 浏览器先查本地缓存。
2. 操作系统缓存。
3. 本地域名服务器。
4. 根域名服务器。
5. 顶级域服务器。
6. 权威域名服务器。
7. 返回结果并缓存。

### DNS 为什么同时使用 TCP 和 UDP？

DNS 查询通常使用 UDP，因为报文小、速度快。

但在以下场景会使用 TCP：

1. 响应内容过大
2. 区域传送
3. 某些特殊可靠性要求场景

### 递归查询和迭代查询的区别是什么？

1. 递归查询：客户端把解析任务交给 DNS 服务器，由它继续向下查询并返回最终结果。
2. 迭代查询：DNS 服务器每次只返回下一步该问谁，查询方自己继续问。

### 如何理解 CDN？说说实现原理。

CDN 是内容分发网络，本质是把静态资源缓存到离用户更近的边缘节点，降低访问延迟。

核心原理：

1. 用户访问域名时，DNS 或调度系统把请求导向合适节点。
2. 节点命中缓存则直接返回。
3. 未命中则回源拉取资源并缓存。

适用场景：

1. 图片、视频、脚本、样式文件加速
2. 大流量静态资源分发

### 说说地址栏输入 URL 敲下回车后发生了什么。

这是高频综合题，可以按顺序回答：

1. 浏览器解析 URL。
2. 查询缓存和 DNS 解析。
3. 建立 TCP 连接，HTTPS 还要做 TLS 握手。
4. 浏览器发送 HTTP 请求。
5. 服务器处理并返回响应。
6. 浏览器解析 HTML，构建 DOM。
7. 解析 CSS，构建 CSSOM。
8. 合成渲染树并布局、绘制。
9. 遇到 JS 时按加载和执行规则处理。
10. 页面完成渲染，后续可能继续触发异步加载。

## 五、浏览器缓存与存储

### 对浏览器缓存机制的理解是什么？

浏览器缓存的核心目的是减少重复请求、降低延迟、节省带宽。

常见缓存机制分两类：

1. 强缓存
2. 协商缓存

浏览器会先判断强缓存是否命中，未命中再发请求走协商缓存。

### 强缓存和协商缓存的区别是什么？

1. 强缓存命中时，浏览器直接使用本地缓存，不发请求。
2. 协商缓存会发请求到服务器，由服务器判断资源是否变更。

常见字段：

1. 强缓存：`Expires`、`Cache-Control`
2. 协商缓存：`Last-Modified`、`If-Modified-Since`、`ETag`、`If-None-Match`

### 与缓存相关的 HTTP 请求头有哪些？

常见缓存相关头包括：

1. `Cache-Control`
2. `Expires`
3. `ETag`
4. `If-None-Match`
5. `Last-Modified`
6. `If-Modified-Since`
7. `Pragma`

其中现代项目中更常用的是 `Cache-Control` 和 `ETag`。

### 为什么需要浏览器缓存？

原因主要有：

1. 提升页面打开速度
2. 减少服务器压力
3. 节省带宽
4. 提升用户体验

### 点击刷新按钮、按 F5、按 Ctrl+F5、地址栏回车有什么区别？

大致可以这样理解：

1. 地址栏回车：通常会优先按正常缓存策略处理。
2. F5 或普通刷新：可能跳过部分强缓存，但仍可能走协商缓存。
3. Ctrl+F5：更偏强制刷新，尽量重新从服务器获取资源。

不同浏览器实现会有差异，但面试里这样回答通常足够。

### 浏览器本地存储方式有哪些？使用场景是什么？

常见方式有：

1. Cookie
2. LocalStorage
3. SessionStorage
4. IndexedDB

使用场景：

1. Cookie：会话标识、登录态
2. LocalStorage：长期保存少量配置
3. SessionStorage：会话级临时数据
4. IndexedDB：大规模结构化数据离线存储

### Cookie 有哪些字段？作用分别是什么？

常见字段包括：

1. `name` / `value`
2. `Expires` / `Max-Age`
3. `Domain`
4. `Path`
5. `Secure`
6. `HttpOnly`
7. `SameSite`

它们主要用于控制生命周期、作用范围和安全策略。

### Cookie、LocalStorage、SessionStorage 的区别是什么？

主要区别：

1. 生命周期不同。
2. 容量不同。
3. Cookie 会自动随请求发送，另外两者不会。
4. 安全性和用途不同。

### IndexedDB 有哪些特点？

IndexedDB 是浏览器提供的本地数据库，特点是：

1. 容量较大
2. 支持结构化数据
3. 异步操作
4. 支持索引和事务
5. 适合离线应用和大量本地数据

## 六、浏览器原理与渲染机制

### 对浏览器的理解是什么？

浏览器本质上是一个把网络资源转换成可交互页面的客户端程序。

它主要承担：

1. 网络请求
2. 资源解析
3. 页面渲染
4. 脚本执行
5. 安全隔离
6. 存储与缓存管理

### 对浏览器内核的理解是什么？常见浏览器内核有哪些？

通常说的浏览器内核，主要包括渲染引擎和 JS 引擎等核心部分。

常见渲染引擎：

1. Blink
2. WebKit
3. Gecko

面试时常见说法：

1. Chrome、Edge 多使用 Blink
2. Safari 使用 WebKit
3. Firefox 使用 Gecko

### 浏览器的主要组成部分有哪些？

可以概括为：

1. 用户界面
2. 浏览器引擎
3. 渲染引擎
4. 网络模块
5. JavaScript 引擎
6. UI 后端
7. 数据存储模块

### 浏览器的渲染过程是什么？

简化流程：

1. 解析 HTML，生成 DOM 树。
2. 解析 CSS，生成 CSSOM 树。
3. 合成渲染树。
4. 布局，计算元素几何信息。
5. 绘制成像素。
6. 分层和合成显示到屏幕。

### 渲染过程中遇到 JS 文件如何处理？

默认情况下，浏览器解析 HTML 遇到普通 `script` 会暂停 HTML 解析，先下载并执行脚本，再继续解析。

因为脚本可能会修改 DOM 结构，浏览器必须保证执行顺序正确。

所以性能优化中常见做法是：

1. 把脚本放到底部
2. 使用 `defer`
3. 使用 `async`

### 什么是文档的预解析？

预解析是浏览器在主解析线程执行阻塞任务时，提前扫描文档中后续可能需要加载的资源，比如脚本、样式、图片，尽早发起下载。

它的作用是提高资源加载并行度。

### CSS 会阻塞文档解析或渲染吗？

严格来说，CSS 一般不会像同步脚本那样直接阻塞 HTML 文档解析，但它会阻塞渲染，并且可能间接影响后续脚本执行。

因为脚本执行可能依赖最终样式信息，所以浏览器会谨慎协调 CSS 和 JS 的加载执行顺序。

### 如何优化关键渲染路径？

常见做法：

1. 减少关键资源体积
2. 内联关键 CSS
3. 延迟加载非关键资源
4. 减少阻塞脚本
5. 压缩、缓存、CDN 加速
6. 优化图片和字体加载

### 什么情况会阻塞渲染？

常见情况有：

1. 大体积 CSS 未完成下载和解析
2. 阻塞型同步脚本执行
3. 大量主线程计算
4. 频繁回流重绘

## 七、同源策略、跨域与代理

### 什么是同源策略？

同源策略是浏览器的安全机制，要求协议、域名、端口三者都相同才算同源。

它的核心目的，是防止一个站点随意读取另一个站点的敏感数据。

### 如何解决跨域问题？

常见方案有：

1. CORS
2. JSONP
3. 反向代理，如 Nginx
4. Node 中间件代理
5. `postMessage`
6. WebSocket

实际项目里最常见的是：开发阶段走代理，生产环境由后端或网关正确配置 CORS。

### CORS 是什么？简单请求和预检请求是什么？

CORS 是跨域资源共享机制，允许服务器通过响应头声明哪些跨域请求可以被浏览器放行。

1. 简单请求：满足特定方法和请求头限制的请求，可直接发出。
2. 预检请求：浏览器先发一个 `OPTIONS` 请求，询问服务器是否允许真正请求。

### 为什么有时候 POST 会发送两次请求？

很多人看到“POST 发两次”其实是：

1. 第一次是浏览器自动发出的 `OPTIONS` 预检请求。
2. 第二次才是真正的 POST 请求。

这通常和 CORS、非简单请求头、`Content-Type` 等条件有关。

### CORS 中 Cookie 相关问题要注意什么？

如果跨域请求需要携带 Cookie：

1. 前端要设置 `withCredentials` 或对应选项。
2. 服务端 `Access-Control-Allow-Origin` 不能写 `*`。
3. 服务端要显式返回 `Access-Control-Allow-Credentials: true`。
4. Cookie 还要考虑 `SameSite` 策略。

### JSONP 的原理是什么？优缺点是什么？

JSONP 利用了 `script` 标签不受同源策略限制的特点，通过动态插入脚本标签获取数据。

优点：

1. 实现简单
2. 兼容老环境

缺点：

1. 只支持 GET
2. 安全性差
3. 现代项目中基本被 CORS 替代

### postMessage 的使用场景是什么？

`postMessage` 主要用于跨窗口、跨 iframe、跨源页面之间通信。

常见场景：

1. iframe 和父页面通信
2. 多窗口通信
3. 跨域页面传递消息

### 正向代理和反向代理的区别是什么？

1. 正向代理代理的是客户端。
2. 反向代理代理的是服务器。

前端更常接触的是反向代理，比如 Nginx 做跨域转发、负载均衡、静态资源服务。

### Nginx 的概念及工作原理是什么？

Nginx 是高性能 Web 服务器和反向代理服务器。

常见用途：

1. 反向代理
2. 负载均衡
3. 静态资源托管
4. HTTPS 终止
5. 网关转发

### 如何实现浏览器内多个标签页之间的通信？

常见方式：

1. `localStorage` 事件
2. `BroadcastChannel`
3. `SharedWorker`
4. Service Worker
5. 同源下共享后端连接或轮询状态

## 八、浏览器事件、线程与垃圾回收

### 进程和线程的概念及区别是什么？

1. 进程是资源分配的基本单位。
2. 线程是 CPU 调度的基本单位。
3. 一个进程可以包含多个线程。
4. 线程共享进程资源，切换成本通常比进程低。

### 浏览器渲染进程的线程有哪些？

常见可以概括为：

1. GUI 渲染线程
2. JS 引擎线程
3. 事件触发线程
4. 定时器线程
5. 异步网络线程

面试时要知道：GUI 线程和 JS 线程互斥，因此长时间 JS 执行会阻塞页面渲染。

### 对 Service Worker 的理解是什么？

Service Worker 是浏览器后台运行的脚本，可以拦截请求、管理缓存、支持离线能力和消息推送。

常见场景：

1. 离线缓存
2. PWA
3. 资源预缓存
4. 后台消息推送

### 事件是什么？事件模型是什么？

事件是浏览器中用户或系统触发的交互信号，比如点击、输入、加载完成。

事件模型通常指：

1. 捕获阶段
2. 目标阶段
3. 冒泡阶段

### 如何阻止事件冒泡？

常见方式是：

1. `event.stopPropagation()`
2. 某些场景下也可能结合 `stopImmediatePropagation()`

### 对事件委托的理解是什么？使用场景有哪些？

事件委托就是把子元素事件统一委托给父元素处理，依赖事件冒泡。

适用场景：

1. 列表项点击
2. 动态内容操作
3. 大量子节点的统一事件处理

### 同步和异步的区别是什么？

1. 同步任务按顺序执行，前一个不结束，后一个不能开始。
2. 异步任务不会阻塞当前主流程，完成后再通过回调或任务队列处理结果。

### 对事件循环的理解是什么？宏任务和微任务有哪些？

浏览器的事件循环负责在主线程上调度同步任务和异步任务。

常见分类：

1. 宏任务：`setTimeout`、`setInterval`、I/O、UI 事件
2. 微任务：`Promise.then`、`MutationObserver`、`queueMicrotask`

执行顺序通常是：执行一个宏任务后，清空当前微任务队列，再进入下一个宏任务。

### 什么是执行栈？

执行栈是管理函数调用顺序的栈结构。

1. 函数调用时入栈。
2. 执行完成后出栈。
3. 栈过深可能导致栈溢出。

### Node 中的 Event Loop 和浏览器中的有什么区别？

两者核心思想一样，都是事件循环，但实现细节不同。

1. 浏览器更偏向 UI 渲染和 DOM 事件协调。
2. Node 更偏向 I/O 阶段调度。
3. Node 中 `process.nextTick` 的优先级通常很高，需要特别注意。

### V8 的垃圾回收机制是怎样的？

V8 采用分代垃圾回收思想，大致分为：

1. 新生代
2. 老生代

常见回收策略：

1. 新生代多用复制算法
2. 老生代多用标记清除、标记整理

核心目标是：在尽量少影响主线程的前提下回收无用对象。

### 哪些操作会造成内存泄漏？

常见情况有：

1. 意外的全局变量
2. 闭包引用未释放
3. 定时器未清除
4. 事件监听未移除
5. 脱离文档的 DOM 仍被引用
6. 缓存无限增长

## 九、浏览器安全

### 什么是 XSS 攻击？如何防御？

XSS 是跨站脚本攻击，本质是攻击者把恶意脚本注入页面，并在用户浏览器中执行。

常见类型：

1. 存储型 XSS
2. 反射型 XSS
3. DOM 型 XSS

常见防御方式：

1. 输入校验和输出转义
2. 不信任用户输入拼接 HTML
3. 使用 CSP
4. 避免滥用 `innerHTML`

### 什么是 CSRF 攻击？如何防御？

CSRF 是跨站请求伪造，攻击者诱导用户在已登录状态下对目标站点发起非本人意愿的请求。

防御方式：

1. CSRF Token
2. SameSite Cookie
3. 校验 Referer / Origin
4. 对敏感操作二次验证

### 网络劫持有哪些？如何防范？

常见网络劫持包括：

1. DNS 劫持
2. HTTP 劫持
3. 中间人攻击

防范方式：

1. HTTPS
2. HSTS
3. 正确的 DNS 安全策略
4. 重要资源完整性校验
5. 避免连接不可信网络环境

### 有哪些可能引起前端安全的问题？

常见前端安全问题包括：

1. XSS
2. CSRF
3. 点击劫持
4. 文件上传漏洞
5. 敏感信息泄露
6. 第三方脚本风险
7. 不安全的本地存储使用

面试里要强调：前端安全从来不是单点问题，而是输入、传输、存储、渲染多个环节共同治理。
