## 一、微信小程序基础认知

### 说说你对微信小程序的理解？优缺点是什么？

微信小程序是一种运行在微信生态内的轻量级应用形态，不需要用户下载安装即可使用。

它的特点：

1. 依托微信生态，触达成本低。
2. 具备接近原生应用的体验。
3. 有自己独立的运行环境和开发体系。
4. 支持登录、支付、分享、地图等微信原生能力。

优点：

1. 获取用户成本低，使用门槛低。
2. 开发和发布效率较高。
3. 性能通常比普通 H5 更稳定。
4. 更容易接入微信能力，如支付、登录、订阅消息。

缺点：

1. 受限于微信平台规则和能力边界。
2. 生态封闭，跨平台复用有限。
3. 页面和包体积、接口权限等有平台限制。
4. 调试、兼容和审核流程与普通 Web 不同。

### 说说微信小程序的实现原理。

微信小程序不是直接运行在浏览器里的普通网页，而是由微信客户端提供运行容器。

可以从两个层面理解：

1. 逻辑层：运行 JavaScript，负责数据、状态、接口调用。
2. 视图层：负责 WXML/WXSS 渲染。

核心机制：

1. 逻辑层和视图层是分开的。
2. 两层之间通过数据通信驱动界面更新。
3. 微信客户端提供大量原生能力和桥接接口。

一句话总结：小程序本质上是“微信容器 + 双线程模型 + 平台能力桥接”。

### 为什么小程序里访问不到 DOM 和 window？

**根本原因**

小程序采用**逻辑层与视图层物理隔离的双线程架构**。逻辑层跑业务 JS,但它跑在一个**没有浏览器渲染引擎的纯 JS 运行时**里(iOS 的 JavaScriptCore、Android 的 V8、开发者工具的 Chrome 内核),这个运行时根本没有挂载 `window`、`document`、`navigator` 这些 BOM/DOM 对象,不是微信"屏蔽"了,而是宿主**从来就没提供过**。

**具体机制**

1. **双线程隔离**:逻辑层(JS 引擎)和视图层(WebView / 自研 Skyline 渲染层)是两个独立线程,互相看不见对方的内存。逻辑层即使想操作 DOM,也无法跨线程触达视图层的元素。
2. **没有渲染引擎**:浏览器里 `window` / `document` 是渲染引擎挂上去的全局对象,小程序逻辑层用的是裸 JS 引擎(JSCore/V8),只有 ECMAScript 标准 API,所以 `setTimeout` 这类纯语言 API 能用,但 `window.location`、`document.querySelector` 不存在。
3. **通信靠 JSBridge**:逻辑层改数据 → 通过 JSBridge 序列化传到视图层 → 视图层基于新数据重渲染。这条链路上不允许直接传 DOM 引用,因为跨线程根本没法传。
4. **安全 + 一致性需要**:如果允许直接操纵 DOM,审核机制就被绕过(可在运行时改页面结构和文案);同时 iOS/Android/PC 渲染层不一致,封死 DOM 才能保证多端表现统一。

**对比表**

| 全局对象 | 浏览器 | 小程序逻辑层 | 替代方案 |
|---------|-------|-------------|---------|
| `window` | ✅ | ❌ | `getApp()`、`wx.getSystemInfo()` |
| `document` | ✅ | ❌ | `wx.createSelectorQuery()` |
| `localStorage` | ✅ | ❌ | `wx.setStorageSync()` |
| `XMLHttpRequest` / `fetch` | ✅ | ❌ | `wx.request()` |
| `cookie` | ✅ | ❌ | 手动管理 token |
| `setTimeout` / `Promise` | ✅ | ✅ | 同浏览器 |
| `console` | ✅ | ✅ | 同浏览器 |

**代码:用 SelectorQuery 替代 querySelector**

```javascript
// ❌ 浏览器写法,小程序里会报 document is not defined
const el = document.querySelector('.box')
const rect = el.getBoundingClientRect()

// ✅ 小程序写法:通过 SelectorQuery 拿节点信息(异步)
wx.createSelectorQuery()
  .select('.box')
  .boundingClientRect(rect => {
    console.log(rect.width, rect.height)
  })
  .exec()

// ✅ 数据驱动视图,不操作节点
this.setData({ visible: true })   // 视图层自己根据数据重渲染
```

**实操建议**

1. 选三方库前先确认它**不依赖 DOM/BOM**,像 `lodash`、`dayjs` 这类纯 JS 库可用,`jQuery`、`echarts`(浏览器版)、直接读 `window` 的库不能用。
2. 需要节点尺寸/位置时用 `wx.createSelectorQuery()`,但它是**异步**的,设计时要把这层异步性算进去,避免在 `onLoad` 里同步取节点。
3. 业务里不要写 `if (typeof window !== 'undefined')` 的兼容判断,小程序里 `window` 永远不存在,直接用平台 API 更清晰。
4. 真要在小程序里用浏览器生态的库(地图、富文本编辑器等),用 `<web-view>` 嵌一个 H5 页,在 H5 里随便操作 DOM,再用 `postMessage` 和小程序通信。
5. 用 Skyline 渲染引擎的页面也只是改了视图层的渲染方式,**逻辑层依然没有 DOM**,不要被"小程序也支持自定义渲染"的说法误导。

## 二、生命周期与路由机制

### 说说微信小程序的生命周期函数有哪些？

小程序生命周期通常分三类：

1. 应用生命周期。
2. 页面生命周期。
3. 组件生命周期。

应用生命周期常见有：

1. `onLaunch`
2. `onShow`
3. `onHide`
4. `onError`

页面生命周期常见有：

1. `onLoad`
2. `onShow`
3. `onReady`
4. `onHide`
5. `onUnload`

组件生命周期通常围绕组件创建、挂载、更新和销毁展开。

### `pageLifetimes` 是什么？适合什么场景？

`pageLifetimes` 是小程序自定义组件中用来感知所属页面生命周期变化的一组能力。

常见场景：

1. 组件需要感知页面显示和隐藏。
2. 页面切换时组件要做数据刷新或暂停逻辑。
3. 组件内部要跟随页面生命周期处理埋点、轮询、动画等行为。

它的价值在于：组件不仅能感知自身生命周期，还能感知页面级变化。

### 说说微信小程序生命周期的执行过程。

常见理解方式是：

1. 小程序首次启动时，应用级 `onLaunch` 只会执行一次。
2. 小程序进入前台时会触发应用级 `onShow`。
3. 页面进入时通常依次经历 `onLoad`、`onShow`、`onReady`。
4. 页面离开时会触发 `onHide` 或 `onUnload`。
5. 小程序退到后台会触发应用级 `onHide`。

面试时要注意区分：

1. `onLoad` 更偏初始化参数。
2. `onShow` 可能会重复触发。
3. `onReady` 更偏首屏渲染完成。

### 说说微信小程序中路由跳转的方式有哪些？区别是什么？

常见路由 API 有：

1. `wx.navigateTo`
2. `wx.redirectTo`
3. `wx.switchTab`
4. `wx.navigateBack`
5. `wx.reLaunch`

区别大致是：

1. `navigateTo`：保留当前页，跳转到非 tabBar 页面。
2. `redirectTo`：关闭当前页，跳转到新页面。
3. `switchTab`：跳转到 tabBar 页面。
4. `navigateBack`：返回上一页或多级页面。
5. `reLaunch`：关闭所有页面后重新打开一个页面。

面试时可以补一句：小程序页面栈有深度限制，所以不能无限制 `navigateTo`。

## 三、登录、支付与发布流程

### 说说微信小程序的登录流程。

小程序登录的核心思路是：前端先向微信拿临时凭证，再由服务端换取用户身份信息。

常见流程：

1. 小程序调用 `wx.login()` 获取 `code`。
2. 前端把 `code` 传给后端。
3. 后端调用微信接口换取 `openid`、`session_key` 等信息。
4. 后端结合自身业务系统建立登录态。
5. 返回自有 token 或 session 给前端。

面试时一定要强调：真正的身份体系通常仍由业务后端维护，而不是只依赖微信返回的数据。

### `wx.checkSession` 是做什么的？使用时要注意什么？

`wx.checkSession` 用来检查当前小程序登录态是否仍然有效，也就是之前拿到的 `session_key` 是否还在有效期内。

它常见的使用方式是：

1. 先调用 `wx.checkSession`。
2. 如果有效，则继续使用当前登录态。
3. 如果失效，再重新调用 `wx.login` 走登录流程。

要注意的是：

1. 它只能判断微信侧会话是否有效。
2. 业务系统自己的 token 是否过期，还要单独校验。

### 说说微信小程序的支付流程。

常见支付流程：

1. 用户发起下单。
2. 小程序把订单信息传给后端。
3. 后端调用微信支付下单接口，获取支付参数。
4. 后端把支付参数返回给小程序。
5. 小程序调用 `wx.requestPayment` 发起支付。
6. 支付完成后，后端再以异步通知为准更新订单状态。

面试时要强调：订单状态不能只相信前端支付回调，最终要以后端收到的支付通知为准。

### 说说微信小程序的发布流程。

常见流程：

1. 本地开发和调试。
2. 上传代码到微信后台。
3. 提交体验版测试。
4. 完成审核资料和配置。
5. 提交微信审核。
6. 审核通过后发布上线。

一般团队还会配合：

1. 版本管理。
2. 环境区分。
3. 灰度验证。
4. 回滚预案。

## 四、小程序性能优化

### 说说提高微信小程序应用速度的手段有哪些？

小程序性能优化通常从启动、请求、渲染、包体几个方向入手。

常见手段：

1. 控制主包大小。
2. 使用分包和独立分包。
3. 分包预下载和异步化。
4. 精简首屏数据。
5. 提前请求首屏所需接口。
6. 合理使用缓存。
7. 避免无用自定义组件和无效渲染。
8. 使用骨架屏优化感知速度。
9. 减少启动阶段同步 API 调用。

### 小程序性能问题通常从哪些维度定位？

常见可以从三个维度拆解：

1. 启动慢。
2. 请求慢。
3. 交互慢。

进一步排查时可以关注：

1. 包体积是否过大。
2. 首屏接口是否串行阻塞。
3. setData 是否过大、过频繁。
4. 页面节点和组件是否过重。
5. 是否存在低效渲染或重复请求。

### 小程序启动优化可以从哪些方面入手？

启动优化重点在减少初始化阶段的工作量。

常见做法：

1. 使用独立分包，降低主包压力。
2. 静态资源走 CDN。
3. 过滤未使用组件和无效依赖。
4. 减少启动阶段同步逻辑。
5. 将非首屏逻辑延后执行。
6. 使用本地缓存承接部分初始化数据。

### 小程序请求优化可以从哪些方面入手？

请求优化重点在减少等待时间和重复请求。

常见做法：

1. 数据预拉取。
2. 提前拉取首屏所需数据。
3. 对可复用数据做本地缓存。
4. 合并接口或减少串行请求。
5. 做失败重试和降级处理。

### 小程序交互优化可以从哪些方面入手？

交互优化重点是让页面更流畅、反馈更及时。

常见做法：

1. 分步渲染，优先展示核心内容。
2. 控制 `setData` 的频率和体积。
3. 避免一次性渲染过多节点。
4. 对长列表做虚拟化或分页加载。
5. 业务请求做防抖、节流和兜底。

### `setData` 的原理是什么？为什么它会影响性能？

`setData` 是小程序里把逻辑层数据同步到视图层的核心方法。

由于小程序逻辑层和视图层是分开的，数据更新并不是直接操作 DOM，而是通过桥接通信把变更传到视图层。

它影响性能的原因主要有：

1. 数据量过大会增加通信成本。
2. 调用过于频繁会导致视图层更新压力变大。
3. 一次传递过深、过大的对象会增加序列化和更新成本。

优化时通常要注意：

1. 只传必要字段。
2. 控制调用频率。
3. 避免在短时间内大量重复 `setData`。

### 骨架屏、分包预下载、数据缓存分别解决什么问题？

1. 骨架屏：提升首屏感知速度，减少用户对白屏的焦虑。
2. 分包预下载：降低后续页面切换等待时间。
3. 数据缓存：减少重复请求，缩短初始化和回显耗时。

## 五、H5 秒开与移动端性能优化

### 什么是 H5 秒开方案？

H5 秒开不是单一技术，而是一套围绕“让页面尽快可见、可用”的综合优化方案。

它通常包含：

1. 首屏资源优先加载。
2. 包体与静态资源优化。
3. 路由和组件懒加载。
4. 数据预取和缓存。
5. 骨架屏和占位渲染。
6. 长任务和长列表优化。

一句话总结：H5 秒开的本质是提升“首屏可用时间”和“感知速度”。

### H5 秒开的性能指标通常看哪些？

常见会关注：

1. 首屏时间。
2. 白屏时间。
3. 最大内容渲染时间。
4. 接口响应时间。
5. 交互可用时间。
6. 页面渲染耗时。

面试时不一定要背具体指标名，但要能表达“加载、渲染、交互、请求”这几个维度。

### H5 秒开怎么做包体积优化？

常见做法：

1. 路由懒加载。
2. 组件按需加载。
3. Tree Shaking。
4. 分包或拆包。
5. 压缩 JS/CSS。
6. 去除无用依赖。

### H5 秒开怎么做首屏渲染优化？

常见做法：

1. 精简首屏渲染数据。
2. 提前请求关键接口。
3. 缓存首屏接口结果。
4. 骨架屏。
5. 避免首屏加载不必要组件。
6. 把非首屏逻辑延迟执行。

### H5 秒开怎么做请求优化？

常见方式：

1. 数据预拉取。
2. 缓存可复用接口。
3. 避免串行请求。
4. 使用 CDN 和就近调度。
5. 对低优先级请求延后加载。

### H5 秒开怎么做交互优化？

常见方式：

1. 分步渲染。
2. 长列表虚拟滚动。
3. 用 Web Worker 处理重计算。
4. 用 `requestAnimationFrame` 处理动画。
5. 对滚动、输入等高频事件做节流防抖。

### 路由懒加载、组件懒加载、骨架屏、虚拟滚动分别适合什么场景？

1. 路由懒加载：适合页面多、首屏只需少数路由的项目。
2. 组件懒加载：适合大组件、低频组件、弹窗类组件。
3. 骨架屏：适合内容结构明确但数据未返回的首屏页面。
4. 虚拟滚动：适合超长列表、消息流、表格页。

### 小程序和 H5 在性能优化思路上有哪些共同点和差异？

共同点：

1. 都关注首屏速度。
2. 都要控制资源体积。
3. 都要减少无效请求和重复渲染。
4. 都需要缓存、懒加载、骨架屏等手段。

差异点：

1. 小程序受平台容器和包体限制更明显。
2. H5 更依赖浏览器网络、渲染和资源加载策略。
3. 小程序常见优化点会聚焦 `setData`、分包、微信容器能力。
4. H5 常见优化点会聚焦首屏资源、脚本加载、浏览器渲染链路。

## 六、WebView 与跨端协作

### 小程序中的 WebView 是什么？适合什么场景？

小程序中的 WebView 通常指通过 `web-view` 组件承载一个 H5 页面。

适合场景：

1. 已有 H5 页面需要快速接入小程序。
2. 某些内容页不值得完全重写成原生小程序页面。
3. 活动页、营销页、资讯页等内容型页面。

但要注意：

1. WebView 页面体验通常不如原生小程序页面一致。
2. 与小程序原生能力的通信和权限管理要额外处理。
3. 路由、返回、登录态同步要设计清楚。

### 小程序、H5、Hybrid/WebView 在跨端协作中怎么选？

可以从“体验、复用、能力、成本”四个维度去选：

1. 小程序适合微信生态内的业务闭环。
2. H5 适合快速分发和跨平台内容触达。
3. Hybrid/WebView 适合复用已有 Web 资产并接入部分原生能力。

面试时可以总结为：不是技术谁先进就选谁，而是看业务场景、平台边界和团队投入产出比。

## 附：常见代码实现

### 1. 小程序登录示例

```javascript
wx.login({
  success(res) {
    if (res.code) {
      wx.request({
        url: '/api/login',
        method: 'POST',
        data: { code: res.code }
      })
    }
  }
})
```

### 2. 小程序支付示例

```javascript
wx.requestPayment({
  timeStamp: '',
  nonceStr: '',
  package: '',
  signType: 'RSA',
  paySign: '',
  success() {},
  fail() {}
})
```

### 3. H5 骨架屏占位

```html
<div class="skeleton-card"></div>
```

```css
.skeleton-card {

  background: linear-gradient(90deg, #eee 25%, #f5f5f5 37%, #eee 63%);

  animation: loading 1.4s ease infinite;

@keyframes loading {
  0% { background-position: 100% 50%; }
  100% { background-position: 0 50%; }

```
