## 一、微信小程序基础认知

### 说说你对微信小程序的理解？优缺点是什么？

小程序就是寄宿在微信里的轻量应用，"用完即走"是它最初的产品定位——不用下载安装，扫码或搜索就能用，关掉就走。微信提供了一个运行容器，小程序代码在里面跑，能拿到微信的登录、支付、分享、订阅消息这些原生能力。

跟 H5、Native 各自的位置：

| 维度 | 小程序 | H5 | Native App |
|------|-------|-----|-----------|
| 分发 | 微信内扫码 / 搜索 | 任意 URL | 应用商店 |
| 安装 | 不需要 | 不需要 | 需要 |
| 体验 | 接近原生 | 取决于浏览器 | 最好 |
| 开发成本 | 中（微信生态绑死） | 低 | 高 |
| 能力 | 微信生态能力（支付、分享） | Web API | 系统全部能力 |
| 审核 | 严，平台规则限制多 | 无 | 严 |

优势：

- 微信 10 亿 + 用户基础，触达成本极低
- 启动快、安装便捷，转化率比 H5 高
- 原生支付 / 登录 / 分享，业务接入快
- 性能比纯 H5 稳定（双线程架构 + 原生渲染）

劣势：

- 生态封闭：只能在微信里跑，到 App、PC 都不能直接用
- 包体积 / API 受限：单包 2MB，多包总共 20MB，超过后需要分包
- 审核流程：发版要走平台审核，迭代节奏被卡
- 调试体验比浏览器差，开发者工具有自己的怪癖

跨小程序平台问题更复杂：微信、支付宝、抖音、百度、QQ、京东各家小程序语法相近但 API 不通。所以才有 Taro / uni-app / mpvue 这些跨端框架。

### 说说微信小程序的实现原理。

小程序关键的设计就是**双线程架构**：逻辑层（JS）和视图层（渲染）跑在两个独立线程里，靠 JSBridge 通信。

```
┌─────────────────┐       ┌────────────────────────┐
│  逻辑层 Service │ ←───→ │  视图层 View (WebView) │
│  (JSCore/V8)    │ Bridge │  (一个页面一个 WebView) │
│                 │       │                        │
│  - 业务 JS      │       │  - WXML 渲染            │
│  - 数据 / setData│       │  - WXSS 样式            │
│  - wx.* API     │       │  - 用户事件捕获          │
└─────────────────┘       └────────────────────────┘
```

为什么这么设计：

1. **安全**：逻辑层拿不到 DOM，无法运行时改页面结构，平台审核过的内容能保持不变
2. **性能**：JS 计算和页面渲染互不阻塞，业务再忙也不卡渲染
3. **一致性**：渲染层是 WebView（或新的 Skyline 自研引擎），多端一套表现

代价就是逻辑层和视图层之间一切都靠 setData，通信成本不低——这也是 `setData` 性能问题的根源（数据要序列化、跨线程、视图层 diff、重新渲染）。

### 说说微信小程序的生命周期函数有哪些？

参考下题更详细的版本。简记：

- **App**：`onLaunch` → `onShow` → `onHide` → `onError` / `onPageNotFound`
- **Page**：`onLoad` → `onShow` → `onReady` → `onHide` → `onUnload`，外加 `onPullDownRefresh`、`onReachBottom`、`onShareAppMessage`、`onPageScroll`
- **Component**：`created` → `attached` → `ready` → `moved` → `detached`，加 `pageLifetimes`（监听所在页面的 show/hide）和 `lifetimes`（组件自身生命周期，3.x 推荐写在这里面）

跟 Vue / React 对照记容易：

| 小程序 | Vue | React |
|--------|-----|-------|
| onLoad（页面级，只触发一次）| created | constructor / 函数组件首次执行 |
| onReady（DOM 渲染完成）| mounted | useLayoutEffect / componentDidMount |
| onShow / onHide（页面切换）| activated / deactivated | （需要自己监听）|
| onUnload | beforeUnmount | useEffect 清理函数 / componentWillUnmount |

### 为什么小程序里访问不到 DOM 和 window？

小程序采用**逻辑层与视图层物理隔离的双线程架构**。逻辑层跑业务 JS,但它跑在一个**没有浏览器渲染引擎的纯 JS 运行时**里(iOS 的 JavaScriptCore、Android 的 V8、开发者工具的 Chrome 内核),这个运行时根本没有挂载 `window`、`document`、`navigator` 这些 BOM/DOM 对象,不是微信"屏蔽"了,而是宿主**从来就没提供过**。

具体原因有这几条：

1. **双线程隔离**:逻辑层(JS 引擎)和视图层(WebView / 自研 Skyline 渲染层)是两个独立线程,互相看不见对方的内存。逻辑层即使想操作 DOM,也无法跨线程触达视图层的元素。
2. **没有渲染引擎**:浏览器里 `window` / `document` 是渲染引擎挂上去的全局对象,小程序逻辑层用的是裸 JS 引擎(JSCore/V8),只有 ECMAScript 标准 API,所以 `setTimeout` 这类纯语言 API 能用,但 `window.location`、`document.querySelector` 不存在。
3. **通信靠 JSBridge**:逻辑层改数据 → 通过 JSBridge 序列化传到视图层 → 视图层基于新数据重渲染。这条链路上不允许直接传 DOM 引用,因为跨线程根本没法传。
4. **安全 + 一致性需要**:如果允许直接操纵 DOM,审核机制就被绕过(可在运行时改页面结构和文案);同时 iOS/Android/PC 渲染层不一致,封死 DOM 才能保证多端表现统一。

各个全局对象在小程序里的对应方案：

| 全局对象 | 浏览器 | 小程序逻辑层 | 替代方案 |
|---------|-------|-------------|---------|
| `window` | ✅ | ❌ | `getApp()`、`wx.getSystemInfo()` |
| `document` | ✅ | ❌ | `wx.createSelectorQuery()` |
| `localStorage` | ✅ | ❌ | `wx.setStorageSync()` |
| `XMLHttpRequest` / `fetch` | ✅ | ❌ | `wx.request()` |
| `cookie` | ✅ | ❌ | 手动管理 token |
| `setTimeout` / `Promise` | ✅ | ✅ | 同浏览器 |
| `console` | ✅ | ✅ | 同浏览器 |

具体代码层面用 SelectorQuery 替代 querySelector：

```javascript
// ❌ 浏览器写法,小程序里会报 document is not defined
const el = document.querySelector('.box')
const rect = el.getBoundingClientRect()

// ✅ 小程序写法:通过 SelectorQuery 拿节点信息(异步)
wx.createSelectorQuery()
  .select('.box')
  .boundingClientRect(rect => {
    console.log(rect.width, rect.height)
  })
  .exec()

// ✅ 数据驱动视图,不操作节点
this.setData({ visible: true })   // 视图层自己根据数据重渲染
```

几个要注意的地方：

1. 选三方库前先确认它**不依赖 DOM/BOM**,像 `lodash`、`dayjs` 这类纯 JS 库可用,`jQuery`、`echarts`(浏览器版)、直接读 `window` 的库不能用。
2. 需要节点尺寸/位置时用 `wx.createSelectorQuery()`,但它是**异步**的,设计时要把这层异步性算进去,避免在 `onLoad` 里同步取节点。
3. 业务里不要写 `if (typeof window !== 'undefined')` 的兼容判断,小程序里 `window` 永远不存在,直接用平台 API 更清晰。
4. 如果需要在小程序里用浏览器生态的库(地图、富文本编辑器等),用 `<web-view>` 嵌一个 H5 页,在 H5 里随便操作 DOM,再用 `postMessage` 和小程序通信。
5. 用 Skyline 渲染引擎的页面也只是改了视图层的渲染方式,**逻辑层依然没有 DOM**,不要被"小程序也支持自定义渲染"的说法误导。

## 二、生命周期与路由机制

### 说说微信小程序的生命周期函数有哪些？

小程序生命周期通常分三类：

1. 应用生命周期。
2. 页面生命周期。
3. 组件生命周期。

应用生命周期常见有：

1. `onLaunch`
2. `onShow`
3. `onHide`
4. `onError`

页面生命周期常见有：

1. `onLoad`
2. `onShow`
3. `onReady`
4. `onHide`
5. `onUnload`

组件生命周期通常围绕组件创建、挂载、更新和销毁展开。

### `pageLifetimes` 是什么？适合什么场景？

`pageLifetimes` 是小程序自定义组件中用来感知所属页面生命周期变化的一组能力。

比较常见的场景有：

1. 组件需要感知页面显示和隐藏。
2. 页面切换时组件要做数据刷新或暂停逻辑。
3. 组件内部要跟随页面生命周期处理埋点、轮询、动画等行为。

它的价值在于：组件不仅能感知自身生命周期，还能感知页面级变化。

### 说说微信小程序生命周期的执行过程。

常见理解方式是：

1. 小程序首次启动时，应用级 `onLaunch` 只会执行一次。
2. 小程序进入前台时会触发应用级 `onShow`。
3. 页面进入时通常依次经历 `onLoad`、`onShow`、`onReady`。
4. 页面离开时会触发 `onHide` 或 `onUnload`。
5. 小程序退到后台会触发应用级 `onHide`。

这里容易混淆的是三个时机语义不同：

1. `onLoad` 只在页面创建时拿一次参数，所以它更适合做“基于入参的初始化”
2. `onShow` 只要页面重新进入前台就可能再次触发，所以它更适合做“恢复展示时的数据刷新”
3. `onReady` 更偏第一次渲染完成后的时机，因此常用于拿节点尺寸、启动只需执行一次的 UI 逻辑

### 说说微信小程序中路由跳转的方式有哪些？区别是什么？

常见路由 API 有：

1. `wx.navigateTo`
2. `wx.redirectTo`
3. `wx.switchTab`
4. `wx.navigateBack`
5. `wx.reLaunch`

区别大致是：

1. `navigateTo`：保留当前页，跳转到非 tabBar 页面。
2. `redirectTo`：关闭当前页，跳转到新页面。
3. `switchTab`：跳转到 tabBar 页面。
4. `navigateBack`：返回上一页或多级页面。
5. `reLaunch`：关闭所有页面后重新打开一个页面。

这里不能只背几个 API 名字，因为路由方式的选择和页面栈管理直接相关。
`navigateTo` 会保留当前页并压栈，所以如果连续进入不少页面，栈深会不断增长；而 `redirectTo`、`reLaunch`、`switchTab` 实际上都在控制页面栈结构。
也正因为小程序页面栈有深度限制，所以工程上不能无节制地使用 `navigateTo`。

## 三、登录、支付与发布流程

### 说说微信小程序的登录流程。

小程序登录一般是先由前端向微信拿临时凭证，再交给服务端换取用户身份信息。

排查流程：

1. 小程序调用 `wx.login()` 获取 `code`。
2. 前端把 `code` 传给后端。
3. 后端调用微信接口换取 `openid`、`session_key` 等信息。
4. 后端结合自身业务系统建立登录态。
5. 返回自有 token 或 session 给前端。

这里需要说明的是：微信只负责帮你确认“这个用户在微信生态里的身份线索”，并不直接等于你业务系统里的登录态。
后端还必须基于 `openid`、`session_key` 等信息建立自己的用户体系、签发自己的 token / session，并决定权限、过期、续签和风控策略。

### `wx.checkSession` 是做什么的？使用时要注意什么？

`wx.checkSession` 用来检查当前小程序登录态是否仍然有效，也就是之前拿到的 `session_key` 是否还在有效期内。

它常见的使用方式是：

1. 先调用 `wx.checkSession`。
2. 如果有效，则继续使用当前登录态。
3. 如果失效，再重新调用 `wx.login` 走登录流程。

要注意的是：

1. 它只能判断微信侧会话是否有效。
2. 业务系统自己的 token 是否过期，还要单独校验。

### 说说微信小程序的支付流程。

常见支付流程：

1. 用户发起下单。
2. 小程序把订单信息传给后端。
3. 后端调用微信支付下单接口，获取支付参数。
4. 后端把支付参数返回给小程序。
5. 小程序调用 `wx.requestPayment` 发起支付。
6. 支付完成后，后端再以异步通知为准更新订单状态。

原因在于前端支付回调只能证明“用户侧表面上支付流程结束了”，但它并不能保证支付结果已经被服务端可靠确认。
前端可能被关闭、回调可能被篡改、网络可能中断，所以真正可靠的订单状态确认一定要以后端收到的微信异步支付通知为准。

### 说说微信小程序的发布流程。

排查流程：

1. 本地开发和调试。
2. 上传代码到微信后台。
3. 提交体验版测试。
4. 完成审核资料和配置。
5. 提交微信审核。
6. 审核通过后发布上线。

一般团队还会配合：

1. 版本管理。
2. 环境区分。
3. 灰度验证。
4. 回滚预案。

## 四、小程序性能优化

### 说说提高微信小程序应用速度的手段有哪些？

小程序性能优化通常从启动、请求、渲染、包体几个方向入手。

常见手段：

1. 控制主包大小。
2. 使用分包和独立分包。
3. 分包预下载和异步化。
4. 精简首屏数据。
5. 提前请求首屏所需接口。
6. 适当使用缓存。
7. 避免无用自定义组件和无效渲染。
8. 使用骨架屏优化感知速度。
9. 减少启动阶段同步 API 调用。

### 小程序性能问题通常从哪些维度定位？

常见可以从三个维度拆解：

1. 启动慢。
2. 请求慢。
3. 交互慢。

进一步排查时可以关注：

1. 包体积是否过大。
2. 首屏接口是否串行阻塞。
3. setData 是否过大、过频繁。
4. 页面节点和组件是否过重。
5. 是否存在低效渲染或重复请求。

### 小程序启动优化可以从哪些方面入手？

启动优化重点在减少初始化阶段的工作量。

比较常见的做法有：

1. 使用独立分包，降低主包压力。
2. 静态资源走 CDN。
3. 过滤未使用组件和无效依赖。
4. 减少启动阶段同步逻辑。
5. 将非首屏逻辑延后执行。
6. 使用本地缓存承接部分初始化数据。

### 小程序请求优化可以从哪些方面入手？

请求优化重点在减少等待时间和重复请求。

比较常见的做法有：

1. 数据预拉取。
2. 提前拉取首屏所需数据。
3. 对可复用数据做本地缓存。
4. 合并接口或减少串行请求。
5. 做失败重试和降级处理。

### 小程序交互优化可以从哪些方面入手？

交互优化重点是让页面更流畅、反馈更及时。

比较常见的做法有：

1. 分步渲染，优先展示核心内容。
2. 控制 `setData` 的频率和体积。
3. 避免一次性渲染过多节点。
4. 对长列表做虚拟化或分页加载。
5. 业务请求做防抖、节流和兜底。

### `setData` 的原理是什么？为什么它会影响性能？

`setData` 是小程序里把逻辑层数据同步到视图层的核心方法。

由于小程序逻辑层和视图层是分开的，数据更新并不是直接操作 DOM，而是通过桥接通信把变更传到视图层。

它影响性能的主要原因：

1. 数据量过大会增加通信成本。
2. 调用过于频繁会导致视图层更新压力变大。
3. 一次传递过深、过大的对象会增加序列化和更新成本。

优化时通常要注意：

1. 只传必要字段。
2. 控制调用频率。
3. 避免在短时间内大量重复 `setData`。

### 骨架屏、分包预下载、数据缓存分别解决什么问题？

1. 骨架屏：提升首屏感知速度，减少用户对白屏的焦虑。
2. 分包预下载：降低后续页面切换等待时间。
3. 数据缓存：减少重复请求，缩短初始化和回显耗时。

### 分包和独立分包分别解决什么问题？

小程序的分包策略，主要就是在解决“主包不能过大”和“不是所有页面都值得首包加载”这两个问题。

1. **普通分包**
   - 把非首屏页面、低频页面拆出去
   - 让主包更小，启动更快

2. **独立分包**
   - 某个分包可以相对独立运行
   - 用户从这个分包页面进入时，不必强依赖先下载主包全部内容

为什么这重要：

1. 小程序启动体验对主包体积很敏感
2. 不少业务页面天然有明显的访问冷热分层
3. 独立分包特别适合活动页、营销页、低耦合业务模块

所以分包不只是“拆文件”，它更多是在做访问路径和资源优先级治理。

## 五、H5 秒开与移动端性能优化

### 什么是 H5 秒开方案？

H5 秒开不是单一技术，而是一套围绕“让页面尽快可见、可用”的综合优化方案。

它通常包含：

1. 首屏资源优先加载。
2. 包体与静态资源优化。
3. 路由和组件懒加载。
4. 数据预取和缓存。
5. 骨架屏和占位渲染。
6. 长任务和长列表优化。

H5 秒开会被单独拿出来讨论，原因很现实：用户不会关心具体优化动作，主要感知两点：

1. 页面多久能看到主要内容
2. 页面看起来是否“立刻可用”

所以秒开的核心不是某一项技术，而是围绕“首屏可见”和“感知速度”做整套优化组合。

### H5 秒开的性能指标通常看哪些？

常见会关注：

1. 首屏时间。
2. 白屏时间。
3. 最大内容渲染时间。
4. 接口响应时间。
5. 交互可用时间。
6. 页面渲染耗时。

这里不能只停留在背几个指标缩写，因为秒开优化本质是在拆解用户等待链路。

1. 加载维度看资源什么时候到
2. 渲染维度看首屏内容什么时候画出来
3. 交互维度看用户什么时候可以点得动
4. 请求维度看关键数据什么时候返回

能从这四个维度解释问题，比单纯背指标名更能体现你真的理解性能优化。

### H5 秒开怎么做包体积优化？

比较常见的做法有：

1. 路由懒加载。
2. 组件按需加载。
3. Tree Shaking。
4. 分包或拆包。
5. 压缩 JS/CSS。
6. 去除无用依赖。

### H5 秒开怎么做首屏渲染优化？

比较常见的做法有：

1. 精简首屏渲染数据。
2. 提前请求关键接口。
3. 缓存首屏接口结果。
4. 骨架屏。
5. 避免首屏加载不必要组件。
6. 把非首屏逻辑延迟执行。

### H5 秒开怎么做请求优化？

比较常见的方式有：

1. 数据预拉取。
2. 缓存可复用接口。
3. 避免串行请求。
4. 使用 CDN 和就近调度。
5. 对低优先级请求延后加载。

### H5 秒开怎么做交互优化？

比较常见的方式有：

1. 分步渲染。
2. 长列表虚拟滚动。
3. 用 Web Worker 处理重计算。
4. 用 `requestAnimationFrame` 处理动画。
5. 对滚动、输入等高频事件做节流防抖。

### 路由懒加载、组件懒加载、骨架屏、虚拟滚动分别适合什么场景？

1. 路由懒加载：适合页面多、首屏只需少数路由的项目。
2. 组件懒加载：适合大组件、低频组件、弹窗类组件。
3. 骨架屏：适合内容结构明确但数据未返回的首屏页面。
4. 虚拟滚动：适合超长列表、消息流、表格页。

### 小程序和 H5 在性能优化思路上有哪些共同点和差异？

共同点：

1. 都关注首屏速度。
2. 都要控制资源体积。
3. 都要减少无效请求和重复渲染。
4. 都需要缓存、懒加载、骨架屏等手段。

差异点：

1. 小程序受平台容器和包体限制更明显。
2. H5 更依赖浏览器网络、渲染和资源加载策略。
3. 小程序常见优化点会聚焦 `setData`、分包、微信容器能力。
4. H5 常见优化点会聚焦首屏资源、脚本加载、浏览器渲染链路。

## 六、WebView 与跨端协作

### 小程序中的 WebView 是什么？适合什么场景？

小程序中的 WebView 通常指通过 `web-view` 组件承载一个 H5 页面。

适合用在：

1. 已有 H5 页面需要快速接入小程序。
2. 某些内容页不值得完全重写成原生小程序页面。
3. 活动页、营销页、资讯页等内容型页面。

但要注意：

1. WebView 页面体验通常不如原生小程序页面一致。
2. 与小程序原生能力的通信和权限管理要额外处理。
3. 路由、返回、登录态同步要设计清楚。

### 小程序和 WebView / H5 之间如何通信？

这是跨端协作里很高频的一题。

常见思路是：

1. 小程序向 WebView 传初始参数
2. H5 页面通过 URL 参数或注入数据拿到上下文
3. H5 再通过 `postMessage` 向小程序回传消息
4. 小程序在特定时机接收并处理这些消息

为什么通信要特别小心？

1. 小程序和 H5 不在同一个运行环境里
2. 两边生命周期不同
3. 消息到达时机不一定和页面展示时机一致

所以工程上通常要处理：

1. 消息协议格式统一
2. 消息幂等
3. 页面销毁后的消息失效
4. 登录态和来源校验

### 小程序和 H5 的登录态通常如何协同？

这题容易出错的地方是：小程序和 H5 往往不是同一个宿主环境，所以登录态不能简单照搬浏览器 Cookie 思路。

常见协同方式：

1. 小程序先完成自身登录
2. 后端生成一次性登录凭证或短时 token
3. 打开 H5 时通过安全方式把凭证带过去
4. H5 再向业务后端换取自己的会话状态

为什么不建议直接把长期 token 明文拼在 URL 上？

1. URL 容易被记录和泄露
2. 页面跳转链路里可见范围太大
3. 安全审计和失效控制都不理想

所以更稳妥的做法通常是：

1. 短时票据
2. 一次性交换
3. 后端重新签发 H5 会话

### 小程序、H5、Hybrid/WebView 在跨端协作中怎么选？

可以从”体验、复用、能力、成本”四个维度去选：

1. 小程序适合微信生态内的业务闭环。
2. H5 适合快速分发和跨平台内容触达。
3. Hybrid/WebView 适合复用已有 Web 资产并接入部分原生能力。

跨端方案不能只看“谁更先进”，最后还是在做约束匹配。

1. 业务是不是强依赖微信生态能力
2. 页面是不是主要靠 SEO 或外链传播
3. 团队是偏前端还是偏客户端
4. 是否需要多端复用到支付宝、抖音、H5

所以最终选型看的不是单个技术点，而是业务里、平台边界和团队投入产出比。

### 小程序里为什么经常强调减少 `setData` 频率和数据量？

因为 `setData` 不是普通对象赋值，它背后涉及逻辑层到视图层的数据通信。

如果数据量大、调用频繁，就会带来：

1. 序列化成本
2. 线程通信成本
3. 视图层重复更新成本

### 为什么小程序和 H5 的分享实现思路经常不同？

因为小程序分享能力通常是宿主平台提供的原生入口，而 H5 分享更多依赖浏览器、WebView 或接入对应平台 JS SDK。

也就是说：

1. 分享入口时机不同
2. 可配置内容边界不同
3. 是否能直接调起平台分享面板也不同

### 为什么小程序审核会反过来影响技术实现？

因为小程序不是纯技术运行环境，它同时是一个强平台治理环境。

有些实现即使技术上可行，如果触发了：

1. 页面内容规范问题
2. 登录、支付、跳转链路问题
3. 诱导行为、外链边界问题

最后仍然会在审核和发布环节被卡住。

```javascript
class WxAPIWrapper {
  // 检查微信版本
  static checkVersion(requiredVersion) {
    const systemInfo = wx.getSystemInfoSync()
    const currentVersion = systemInfo.version
    return this.compareVersion(currentVersion, requiredVersion) >= 0
  }

  // 版本号比较
  static compareVersion(v1, v2) {
    const arr1 = v1.split('.')
    const arr2 = v2.split('.')
    const len = Math.max(arr1.length, arr2.length)

    for (let i = 0; i < len; i++) {
      const num1 = parseInt(arr1[i] || 0)
      const num2 = parseInt(arr2[i] || 0)
      if (num1 > num2) return 1
      if (num1 < num2) return -1
    }
    return 0
  }

  // 安全调用 API
  static safeCall(apiName, options = {}, fallback) {
    if (wx.canIUse(apiName)) {
      return wx[apiName](options)
    } else {
      console.warn(`API ${apiName} 不可用，使用降级方案`)
      return fallback && fallback()
    }
  }

  // 请求权限（兼容旧版本）
  static requestAuth(scope) {
    return new Promise((resolve, reject) => {
      if (wx.canIUse('authorize')) {
        wx.authorize({
          scope: scope,
          success: resolve,
          fail: reject
        })
      } else {
        // 旧版本直接调用对应 API
        resolve()
      }
    })
  }
}
```

**3. 小程序分享图片不显示**

**问题描述：**

使用 `onShareAppMessage` 自定义分享时，设置的图片不显示或显示为默认图片。

**常见原因：**

1. 图片路径不正确
2. 使用了网络图片但未配置域名白名单
3. 图片尺寸不符合要求（推荐 5:4）
4. 图片格式不支持

**解决方案：**

```javascript
Page({
  onShareAppMessage(options) {
    // 方案1: 使用本地图片（较稳定）
    return {
      title: '分享标题',
      // ✅ 使用本地图片
      imageUrl: '/images/share.jpg',
      path: '/pages/index/index'
    }

    // 方案2: 使用 CDN 图片（需要配置域名白名单）
    // return {
    //   title: '分享标题',
    //   imageUrl: 'https://cdn.example.com/share.jpg',
    //   path: '/pages/index/index'
    // }

    // ❌ 错误：使用相对路径
    // imageUrl: './images/share.jpg'

    // ❌ 错误：使用未配置的域名
    // imageUrl: 'https://unknown.com/share.jpg'
  },

  // 方案3: 动态生成分享图片
  async generateShareImage() {
    return new Promise((resolve) => {
      const query = wx.createSelectorQuery()
      query.select('#shareCanvas')
        .fields({ node: true, size: true })
        .exec((res) => {
          const canvas = res[0].node
          const ctx = canvas.getContext('2d')

          // 设置画布尺寸（推荐 500x400，比例 5:4）
          canvas.width = 500
          canvas.height = 400

          // 绘制分享图片内容
          ctx.fillStyle = '#fff'
          ctx.fillRect(0, 0, 500, 400)

          // 绘制文字、图片等
          ctx.fillStyle = '#000'
          ctx.font = '20px sans-serif'
          ctx.fillText('分享标题', 20, 50)

            canvas: canvas,
            success: (res) => {

  async onShareAppMessage() {
```

3. 支持 JPG、PNG 格式
4. 本地图片路径必须以 `/` 开头
5. 网络图片必须配置在 `downloadFile` 合法域名中

**4. 小程序 canvas 绘图模糊**

**问题描述：**

在高 DPR（设备像素比）屏幕上，canvas 绘制的内容模糊不清。

**原因：**

Canvas 的逻辑像素和物理像素不匹配，需要根据设备像素比进行缩放。

**解决方案：**

```javascript
Page({
  data: {
    canvasWidth: 0,
    canvasHeight: 0
  },

  onReady() {
    this.initCanvas()
  },

  initCanvas() {
    // 获取设备像素比
    const systemInfo = wx.getSystemInfoSync()
    const dpr = systemInfo.pixelRatio

    const query = wx.createSelectorQuery()
    query.select('#myCanvas')
      .fields({ node: true, size: true })
      .exec((res) => {
        const canvas = res[0].node
        const ctx = canvas.getContext('2d')

        // 获取 canvas 的 CSS 尺寸
        const cssWidth = res[0].width
        const cssHeight = res[0].height

        // 设置 canvas 的实际渲染尺寸（物理像素）
        canvas.width = cssWidth * dpr
        canvas.height = cssHeight * dpr

        // 缩放绘图上下文，使逻辑像素和物理像素对应
        ctx.scale(dpr, dpr)

        // 保存尺寸信息
        this.setData({
          canvasWidth: cssWidth,
          canvasHeight: cssHeight
        })

        // 开始绘制（使用 CSS 尺寸）
        this.drawContent(ctx, cssWidth, cssHeight)
      })
  },

  drawContent(ctx, width, height) {
    // 绘制背景
    ctx.fillStyle = '#fff'
    ctx.fillRect(0, 0, width, height)

    // 绘制文字（使用逻辑像素）
    ctx.fillStyle = '#000'
    ctx.font = '16px sans-serif' // 注意：这里使用逻辑像素
    ctx.fillText('清晰的文字', 20, 50)

    // 绘制图片
    const img = canvas.createImage()
    img.onload = () => {
      ctx.drawImage(img, 0, 0, width, height)
    }
    img.src = '/images/bg.jpg'
  },

  // 导出高清图片
  exportImage() {
    const query = wx.createSelectorQuery()
    query.select('#myCanvas')
      .fields({ node: true })
      .exec((res) => {
        const canvas = res[0].node

        wx.canvasToTempFilePath({
          canvas: canvas,
          // 导出高清图片
          destWidth: canvas.width,
          destHeight: canvas.height,
          success: (res) => {
            console.log('导出成功:', res.tempFilePath)
            // 预览图片
            wx.previewImage({
              urls: [res.tempFilePath]
            })
          }
        })
      })
  }
})
```

```xml
<!-- WXML -->
<canvas
  type="2d"
  id="myCanvas"
  style="width: 300px; height: 200px;"
></canvas>
<button bindtap="exportImage">导出图片</button>
```

**核心步骤：**

1. 使用新版 Canvas 2D API（`type="2d"`）
2. 根据设备像素比设置 canvas 实际尺寸
3. 使用 `ctx.scale()` 缩放绘图上下文
4. 绘制时使用逻辑像素（CSS 尺寸）
5. 导出时指定实际尺寸

**旧版 Canvas 的处理方式：**

```javascript
    ctx.fillRect(0, 0, width * dpr, height * dpr)

    ctx.fillStyle = '#000'
    ctx.font = `${16 * dpr}px sans-serif`
    ctx.fillText('文字', 20 * dpr, 50 * dpr)

    ctx.draw()
  }
})
```

```javascript
        icon: 'none',

    console.error('[ErrorHandler]', error)
```

## 八、小程序与 H5 进阶实战

### 微信 Skyline 渲染引擎是什么？什么场景能开？

Skyline 是微信团队自研的新渲染引擎，跟传统 WebView 渲染走的不是一条路。核心动作就一个：**把渲染层从 WebView 切到原生 UI 框架**——iOS 用自研框架、Android 也是自研框架，不再走浏览器内核。

为什么要做？传统小程序的视图层是 WebView，性能瓶颈在「逻辑层 ↔ 视图层」的 JS Bridge 通信。高频交互（拖拽、滚动联动）每帧都要跨线程，掉帧是常态。Skyline 把渲染搬到原生，逻辑层和视图层之间用 worklet（可以在 UI 线程跑的轻量 JS）通信，性能直接接近原生 App。

**收益是真的大**（基础库 3.0+ 实测）：

- 列表滚动从 50fps 提到 60fps 几乎不掉帧
- 启动时间能减少 30-50%（不用初始化 WebView）
- 动画用 worklet 跑在 UI 线程，复杂交互也更流畅

**但限制也多**：

- **不是所有组件都支持**——`<map>`、`<canvas>`、`<video>` 这些原生组件在 Skyline 下兼容方式不一样，部分场景不能用
- **CSS 子集**：Skyline 只支持 CSS 的一个子集，flex 布局 OK，grid 不支持，position: sticky 在某些版本有 bug
- **WXSS 写法要适配**：渲染层不是浏览器，某些选择器（`:nth-child`、属性选择器）行为可能不一致
- **第三方组件库支持参差**——Vant Weapp、TDesign 等都在适配，但有些组件还得 fallback 到 WebView 渲染

**怎么开启**：

```json
// app.json
{
  "renderer": "skyline",
  "rendererOptions": {
    "skyline": {
      "defaultDisplayBlock": true,
      "disableABTest": true
    }
  },
  "componentFramework": "glass-easel"
}
```

**渐进策略**：单页面级别开启，逐页迁移。

```json
// page.json
{ "renderer": "skyline" }   // 只这个页面用 Skyline
```

回到结论:Skyline 是「逻辑层不变、渲染层切到原生」的方案，性能提升明显但组件/CSS 子集有限，目前生产可用但要逐页验证。

### 鸿蒙原子化服务 / HarmonyOS Next 对前端的影响

鸿蒙 Next（HarmonyOS 5）24 年底正式发布，最大的变化是「去 AOSP」——完全脱离 Android。这对前端工程师不是远在天边的事。

**ArkTS 跟 TS 的关系**：

- ArkTS 是华为基于 TS 扩展的语言，加了 `@Component`、`@State`、`@Builder` 这套装饰器
- 语法上 90% 跟 TS 一致，但去掉了 TS 的一些「动态特性」（`any`、`Object` 索引访问等），强制更严格的类型
- 类似 Angular + Vue 的混合体——声明式 UI + 装饰器 + 响应式状态

**ArkUI 跟小程序 / 跨端框架的关系**：

- ArkUI 是鸿蒙的原生 UI 框架，类似 SwiftUI / Jetpack Compose
- 跟微信小程序无关——不是同一生态
- 跟 React Native / Flutter 是竞品

**前端转鸿蒙的路径**：

1. **直接学 ArkTS**——华为推「鸿蒙应用工程师」岗位，重新学一遍 ArkUI
2. **用 Taro / uni-app 跨端到鸿蒙**——这俩都在做鸿蒙适配，能复用现有代码
3. **用 React Native 鸿蒙版**（华为出的官方移植）——RN 代码可以跑在鸿蒙上
4. **Web 应用 + 鸿蒙浏览器**——纯 H5 也能跑，但不算「原生体验」

**「原子化服务」是什么**：鸿蒙特有的概念，类似「比小程序更轻量的 mini App」——免安装、可分享、能在系统级入口（负一屏、智慧搜索）直达。开发上类似 ArkUI，但要遵守原子化服务的设计规范。

**前端从业者的实际选择**：

- 业务有鸿蒙渠道需求 → 学 ArkTS，但工作量等同于学一门新框架
- 想跨端覆盖鸿蒙 → 看 Taro / uni-app / RN 鸿蒙适配的进度
- 一般业务 → 短期不用动，鸿蒙生态还在早期，多数公司还没出鸿蒙原生 App

### 小程序首屏白屏的常见原因和定位方法

线上反馈「打开就白屏」是高频投诉。要快速定位先分类：

**类型 1：代码包还没下完就开始渲染**

主包过大（接近 2MB 上限）时，弱网下载几秒，期间用户看到白屏。

定位：开发者工具的 Network 面板看代码包下载耗时；线上看「启动监控」里的「代码包下载阶段」时长。

解决：

- 主包瘦身——把非首屏页面拆到分包
- 启用「按需注入」（`lazyCodeLoading: requiredComponents`）——只加载本页用到的组件
- 分包预下载（`preloadRule`）——主包加载完后台静默预下载常用分包

```json
// app.json
{
  "lazyCodeLoading": "requiredComponents",
  "preloadRule": {
    "pages/index/index": {
      "network": "all",
      "packages": ["cart", "user"]
    }
  }
}
```

**类型 2：onLoad 同步阻塞 setData**

启动后 `onLoad` 里同步算了一批数据再 `setData`，主线程被堵。

```js
// ❌ 同步遍历 + setData 5000 项
onLoad() {
  const list = bigArray.map(complexTransform)   // 同步阻塞
  this.setData({ list })                         // 一次性 5000 项
}

// ✅ 先 setData 关键骨架，剩余异步分批
onLoad() {
  this.setData({ skeleton: true })
  setTimeout(() => {
    this.setData({ list: bigArray.slice(0, 20) })
  }, 0)
}
```

**类型 3：首屏请求 waterfall**

`onLoad` 里串行调几个接口，加起来 2-3 秒。

定位：开发者工具 Network 面板看接口瀑布图。

解决：

- 并发请求（Promise.all）
- 关键接口直接走 prefetch（小程序的 `wx.request` 在打开页面前就发）
- 不阻塞首屏的接口拖到 `onReady` 或更晚

**类型 4：JS 报错导致 render 失败**

页面注册时报错（比如 import 了不存在的模块）、`data` 初始值导致 wxml 渲染 crash。

定位：开发者工具 Console 面板找红色错误；真机调试用 vConsole 或微信开发工具远程调试。

**类型 5：自定义组件初始化慢**

页面引入了大量自定义组件（每个组件都要解析、注册），首屏渲染前要先把组件树初始化完。

解决：

- 启用「组件按需注入」
- 首屏只用必要组件，复杂组件用 `<page-container>` 延后渲染

**Trace 工具**：微信开发者工具的「Performance」面板能记录完整启动 trace——从「`Page` 注册」到「首次渲染完成」每一帧都有，定位瓶颈最精准。

### 微信分包预加载策略实战

分包不只是「文件分组」，预加载策略才是关键。

**默认行为**：用户访问分包页面时才下载该分包。第一次访问会有「下载 → 解析 → 渲染」的延迟。

**`preloadRule` 配置**：

```json
{
  "preloadRule": {
    "pages/index/index": {
      "network": "all",
      "packages": ["activity", "user"]
    },
    "pages/cart/cart": {
      "network": "wifi",
      "packages": ["__APP__"]
    }
  }
}
```

- `network`: `all`（任何网络）或 `wifi`（只在 WiFi 下预加载，省流量）
- `packages`: 数组，指定预下载哪些分包；`__APP__` 是个特殊值，表示「除当前分包外所有分包」

**实战策略**：

按业务热度分层：

```json
{
  "subpackages": [
    {
      "root": "cart",
      "pages": ["index"]
    },
    {
      "root": "activity",
      "pages": ["promo", "coupon"],
      "independent": true   // 独立分包,可以不依赖主包
    },
    {
      "root": "rarely-used",
      "pages": ["help", "about"]
    }
  ],
  "preloadRule": {
    "pages/index/index": {
      "network": "all",
      "packages": ["cart"]    // 首页加载完立刻预下载购物车（高频）
    },
    "pages/cart/index": {
      "network": "wifi",
      "packages": ["activity"]  // 购物车页 WiFi 下预下载活动（中频）
    }
    // help/about 不预下载——用户进了再说
  }
}
```

**独立分包是另一回事**：

独立分包不依赖主包就能运行。适合「从外部链接直接运行特定页面」的场景——比如朋友圈分享的活动页，用户点开后只要下载活动分包就能看，不用先下主包。

```json
{
  "subpackages": [
    {
      "root": "promo",
      "pages": ["index"],
      "independent": true
    }
  ]
}
```

代价：独立分包不能用主包里的 utils、组件、Vuex 状态——它是真独立。如果共享代码多，独立分包不合适。

**性能监控**：「微信小程序数据助手」后台有「页面性能」一栏，能看到每个分包的下载耗时、运行错误率，按这个数据调整预加载策略。

### H5 秒开方案的具体改造路径

一个真实 case：电商落地页从 LCP 5.2s 优化到 1.4s，每一步改了什么。

**起点：LCP 5.2s 的常见症状**

- 首屏 HTML 1.2MB（含大量内联 JSON）
- JS bundle 2.8MB（含 Antd 全量、moment）
- LCP 元素是主图（800KB JPG）
- HTML 加载完到首次绘制等了 1.5s（白屏）

**改造步骤 1：路由懒加载 + 按需引入**

```js
// ❌ 之前:全量 import
import { Button, Modal, Form, Input, ... } from 'antd'

// ✅ 之后:Babel/Vite 按需
import { Button } from 'antd/es/button'

// 路由懒加载
const Detail = lazy(() => import('./Detail'))
```

效果：JS bundle 2.8MB → 800KB（首屏）

**改造步骤 2：换 moment → dayjs**

`moment` 800KB（含 locales）→ `dayjs` 7KB，API 几乎一致。这一步省 700KB。

**改造步骤 3：图片优化全套**

```html
<!-- 主图：WebP + 响应式 + 占位 -->
<img
  src="https://cdn.example.com/hero.jpg?w=750"
  srcset="
    https://cdn.example.com/hero.jpg?w=375 375w,
    https://cdn.example.com/hero.jpg?w=750 750w,
    https://cdn.example.com/hero.jpg?w=1500 1500w
  "
  sizes="100vw"
  loading="eager"
  fetchpriority="high"
  width="750" height="400"
/>

<!-- 非首屏图片：lazy + WebP fallback -->
<picture>
  <source srcset="img.webp" type="image/webp">
  <img src="img.jpg" loading="lazy" width="200" height="200">
</picture>
```

CDN 用 `?w=750` 这种参数动态裁剪，主图 800KB → 80KB（WebP）。

**改造步骤 4：关键 CSS 内联**

把首屏需要的 CSS 抽出来内联到 `<head>`，避免「等 CSS 文件下载 → 才能渲染」：

```html
<head>
  <style>
    /* critical CSS:首屏需要的最小集合 */
    body { margin: 0; font: 14px/1.5 system-ui; }
    .hero { width: 100%; aspect-ratio: 750/400; }
    /* ... */
  </style>
  <link rel="preload" href="/main.css" as="style" onload="this.rel='stylesheet'">
</head>
```

工具：`critical` / `penthouse` 自动抽取首屏 CSS。

**改造步骤 5：preconnect / dns-prefetch**

```html
<head>
  <link rel="preconnect" href="https://cdn.example.com">
  <link rel="preconnect" href="https://api.example.com">
  <link rel="dns-prefetch" href="https://stat.example.com">
</head>
```

提前建立 TCP/TLS 连接，省下首次请求时的 200-500ms。

**改造步骤 6：SSR / 预渲染**

- 内容型页面（详情页、文章）→ SSR（Next.js / Nuxt）或 SSG
- 营销页 → 用 `prerender-spa-plugin` / Vercel 预渲染，把首屏 HTML 直接写成静态文件
- 动态部分用 React Server Components / Astro 的 Islands

效果：白屏从 1.5s → 200ms

**改造步骤 7：字体子集化**

```css
/* 只加载首屏需要的字符子集 */
@font-face {
  font-family: 'CustomFont';
  src: url('/fonts/subset.woff2') format('woff2');
  font-display: swap;
  unicode-range: U+0020-007F;
}
```

工具：`fonttools` 命令行抽子集，2MB 字体能砍到 100KB。

**改造步骤 8：HTTP/2 + Brotli 压缩**

服务端配 Brotli 比 gzip 再省 15-20%。HTTP/2 多路复用避免连接数瓶颈。

**最终结果**：

- LCP：5.2s → 1.4s
- FCP：3.1s → 0.6s
- TTI：6.5s → 2.1s

秒开不是单个技术点，是「网络、资源、渲染、字体」每一层都分别优化。单个优化收益 10-15%，10 个优化叠加就是 70-80%。

### web-view 嵌入 H5 的安全设计

小程序里的 `<web-view>` 是个常被滥用的组件，安全设计稍不注意就出大问题。

**域名白名单**：

```jsonc
// 小程序后台 → 开发管理 → 业务域名
{
  "web-view": [
    "https://example.com",
    "https://*.example.com"
  ]
}
```

`<web-view>` 只能加载白名单里的域名。改这个配置要重新发版，不能动态加，所以白名单要谨慎规划——别把所有可能的子域名都加进去。

**登录态传递的错误做法**：

```html
<!-- ❌ 把 token 明文拼在 URL 里 -->
<web-view src="https://h5.example.com/page?token=abc123xyz"></web-view>
```

为什么不行：

- URL 会被 `Referer` 头传给第三方资源（图片、JS）
- 浏览器历史、错误日志、CDN 日志都会记录
- 用户截图分享会泄露

**正确做法：短时 + 一次性 code**：

```js
// 小程序里
const { code } = await wx.cloud.callFunction({
  name: 'generateH5Code',
  data: { userId: this.userId }
})
// code 是一个 5 分钟有效的一次性凭证

// 跳转 web-view
this.setData({
  webviewUrl: `https://h5.example.com/page?code=${code}`
})

// H5 页面里
async function exchangeToken(code) {
  const res = await fetch('/api/h5/exchange', {
    method: 'POST',
    body: JSON.stringify({ code })
  })
  const { token } = await res.json()
  // 把 token 存 sessionStorage,session 维度而非永久
  sessionStorage.setItem('h5Token', token)
}
```

服务端处理 `exchange` 时：

- 校验 code 是否存在、是否被用过、是否过期
- 用了就立刻 invalidate
- 签发独立的 H5 token（跟小程序 token 隔离）

**postMessage 通信的限制**：

小程序的 `web-view.bindmessage` 不是实时的——只有用户**手动触发**（点击「分享」、点击「返回」等动作）才会回调。所以 H5 → 小程序 实时通信没法做，反过来用 url 参数。

```html
<!-- H5 → 小程序:只能在特定时机触发 -->
<script>
wx.miniProgram.postMessage({
  data: { type: 'h5-ready' }
})
// 但小程序只在分享、复制链接、payment 等场景才能收到
</script>
```

**安全检查清单**：

- 白名单域名定期清理无用项
- token 用短时 + 一次性 code 换取，不直接传
- H5 内部接口都做身份校验
- 小程序里的 `web-view` 隐藏起来时要 unmount，避免后台请求
- 用户登出时同步清理 H5 sessionStorage

### H5 异常监控的链路

线上不出错是不可能的，关键是「出了能知道、能定位」。

**第一层：JS 异常捕获**

```js
// 同步错误
window.addEventListener('error', (event) => {
  report({
    type: 'js_error',
    message: event.message,
    filename: event.filename,
    lineno: event.lineno,
    colno: event.colno,
    stack: event.error?.stack,
  })
}, true)

// Promise 异常
window.addEventListener('unhandledrejection', (event) => {
  report({
    type: 'promise_error',
    reason: event.reason?.message || String(event.reason),
    stack: event.reason?.stack,
  })
})
```

**第二层：资源加载失败**

```js
window.addEventListener('error', (event) => {
  const target = event.target
  if (target && (target.tagName === 'IMG' || target.tagName === 'SCRIPT' || target.tagName === 'LINK')) {
    report({
      type: 'resource_error',
      tagName: target.tagName,
      src: target.src || target.href,
    })
  }
}, true)   // ⚠️ 必须用 capture 阶段,resource error 不冒泡
```

**第三层：接口错误**

```js
// fetch 拦截
const originalFetch = window.fetch
window.fetch = async (...args) => {
  const start = Date.now()
  try {
    const res = await originalFetch(...args)
    if (!res.ok) {
      report({
        type: 'api_error',
        url: args[0],
        status: res.status,
        duration: Date.now() - start,
      })
    }
    return res
  } catch (err) {
    report({
      type: 'api_error',
      url: args[0],
      message: err.message,
      duration: Date.now() - start,
    })
    throw err
  }
}
```

**第四层：白屏 / 性能问题**

```js
// 用 PerformanceObserver 监听 LCP / CLS / INP
new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.startTime > 4000) {
      report({ type: 'lcp_slow', value: entry.startTime })
    }
  }
}).observe({ type: 'largest-contentful-paint', buffered: true })
```

**Source Map 上传与还原**：

生产环境不能暴露 source map（被反编译），但又要能定位错误。方案：

- 构建时把 source map 单独上传到错误监控平台（Sentry / 自建）
- 不上传到 CDN
- 监控平台收到错误堆栈时用 source map 还原行号

```js
// vite.config.ts
export default {
  build: {
    sourcemap: 'hidden',   // 生成但不写 sourceMappingURL
  }
}

// CI 里
sentry-cli sourcemaps upload --release=v1.2.3 dist/
```

**采样策略**：

错误量大的项目全量上报会把后端打爆。常见策略：

- 致命错误（页面崩溃、白屏）：100% 上报
- JS 异常：10-30% 采样
- 接口错误：按业务重要性分级
- 用户主动反馈触发的：100% 上报（带完整上下文）

```js
function report(data) {
  // 基础采样
  if (data.type === 'js_error' && Math.random() > 0.3) return

  // 去重(同样的错误 5 分钟内只报一次)
  const key = `${data.type}_${data.message}`
  if (recentReported.has(key)) return
  recentReported.set(key, Date.now())

  navigator.sendBeacon('/api/log', JSON.stringify(data))
}
```

**上报时机**：

- 不要立刻上报——攒一批用 `requestIdleCallback` 空闲时发
- 页面离开时用 `sendBeacon`（即使页面关闭也能发完）
- visibilitychange = hidden 时强制 flush

```js
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'hidden') {
    flushQueue()   // 用 sendBeacon
  }
})
```

**聚合与告警**：

后端收到上报后：

- 按 message + stack hash 聚合，避免单一错误刷屏
- 设阈值告警（同一错误 1 分钟内超 100 次 → 钉钉/飞书报警）
- 关联用户、设备、版本，便于复现

### 小程序登录态续期的落地处理

常见的小程序登录流程是「`wx.login` 拿 code → 后端换 session_key → 给前端发个 token → token 存 storage」。流程本身没问题，但生产里要处理的细节远不止这些。

**问题一：session_key 失效了怎么办？**

session_key 不是永久有效——用户长时间不打开小程序、或者在其他设备登录会失效。后端用这个 key 调微信接口（比如解密手机号）会报错。

正确做法：

```js
// 前端在「关键操作前」先 check
async function ensureValidSession() {
  try {
    await wx.checkSession()
    return true   // 仍有效
  } catch {
    // 失效了,重新 login
    const { code } = await wx.login()
    await api.refreshSession({ code })
    return true
  }
}

// 业务调用
async function getPhoneNumber(e) {
  await ensureValidSession()
  return api.decodePhone(e.detail.encryptedData, e.detail.iv)
}
```

`wx.checkSession` 不会发网络请求，本地检查 session 是否过期，便宜。但它**只能告诉你「session_key 还有效」，不等于「用户业务登录态还有效」**——业务侧 token 该过期还是过期。

**问题二：业务 token 怎么续期？**

业界两种主流模式：

**短 access token + 长 refresh token**：

```js
// 后端登录接口返回
{
  accessToken: '...',   // 1 小时
  refreshToken: '...',  // 30 天
}

// 前端拦截器:401 时自动用 refreshToken 换新 accessToken
async function request(url, opts) {
  const res = await wx.request({ ...opts, url, header: { Authorization: `Bearer ${accessToken}` }})
  if (res.statusCode === 401) {
    const newToken = await refreshAccessToken()
    return wx.request({ ...opts, url, header: { Authorization: `Bearer ${newToken}` }})
  }
  return res
}
```

但**小程序场景下还有更简单的选项**——直接用 `wx.login` 重发 code：

```js
async function refreshAccessToken() {
  const { code } = await wx.login()
  const { accessToken } = await api.exchangeToken({ code })
  return accessToken
}
```

因为微信本身就是身份提供方，重新走 login 拿新 code 几乎没成本。不需要 refreshToken 那套复杂机制。这个差异是小程序生态特有的便利。

**问题三：并发请求都 401 怎么办？**

页面同时发 5 个请求，token 已经过期。5 个请求都 401，5 个都触发 refresh，发了 5 次 wx.login。这不仅浪费，还可能出现 race condition（多个 refresh 互相覆盖最新 token）。

解决：refresh 加单例锁。

```js
let refreshPromise = null

async function refreshAccessToken() {
  if (refreshPromise) return refreshPromise   // 已有 refresh 在跑,等它
  refreshPromise = doRefresh().finally(() => {
    refreshPromise = null
  })
  return refreshPromise
}
```

**问题四：跨场景的登录态共享**

小程序有「分享卡片」、「公众号文章跳小程序」、「扫码进入」多种入口。不同入口的登录态怎么衔接？

一般做法：

- 每个入口都走 `wx.login` 拿 code（这一步不会打断用户操作）
- 后端通过 `openid` 关联到同一个业务账号
- 业务侧的 token 跟入口无关，只跟 openid 绑定

第一次登录后用户已经授权过手机号、用户信息，下次只要 `wx.login` 静默续上即可，不用每次都弹授权。

### 小程序云开发到底要不要用？

云开发是腾讯给小程序提供的 BaaS（后端即服务）——免运维、按用量付费、跟小程序生态深度集成。出来好几年了，社区评价两极。要不要用看几个因素。

**云开发能干什么**：

- 云函数（Node 函数即服务）
- 云数据库（MongoDB 风格的 NoSQL）
- 云存储（CDN + 对象存储）
- 云调用（不用配 access_token 直接调微信开放接口）

```js
// 直接在小程序里调用云函数
wx.cloud.callFunction({
  name: 'getOrder',
  data: { orderId: '123' },
  success: res => console.log(res.result),
})

// 直接操作云数据库
const db = wx.cloud.database()
db.collection('orders').where({ status: 'paid' }).get()
```

**适合用云开发的场景**：

- **创业期 / 个人开发者**：省运维成本，0 配置上线
- **小程序为主业务**、不需要 PC 端 / App 端共用后端
- **微信生态紧密绑定**：要发模板消息、要调微信开放接口、要做小程序内购
- **流量可控**：日 PV < 10 万

**不适合的场景**：

- **多端共用后端**：云开发的能力主要绑定小程序，PC / App 复用成本较高
- **流量大**：按调用次数 + 存储计费，规模上去后成本可能高于自建服务器
- **复杂业务**：云数据库的查询能力弱于 MySQL，复杂 join 维护成本较高
- **要做数据合规**：云开发数据存腾讯云，金融、医疗等合规敏感业务需要谨慎评估

**云函数的限制**：

- **冷启动**：长时间没调用的云函数第一次调用要 1-2 秒
- **超时限制**：单次执行最长 60 秒
- **本地调试**：云函数本地跑要 cloudbase-cli，体验不如自建 Node 项目直接
- **依赖管理**：每个云函数有独立的 node_modules，包体积膨胀快

选型建议：

- demo / 小工具 / 个人项目 → 直接用，维护成本低
- 中型项目（流量适中、要快速迭代）→ 部分功能用云函数 + 部分自建
- 大型项目 → 自建 Node 后端 + Nginx + MySQL/MongoDB，云开发只用「云调用」这一个能力来调微信接口

### 小程序订阅消息从订阅到送达的链路

订阅消息是替代旧版「模板消息」的方案，权限模型更严格、用户感知更明确。完整链路涉及前端、后端和微信平台三方。

**第一步：申请模板**

后台 → 订阅消息 → 公共模板库挑一个、或者申请自定义模板。每个模板有 `tmplId`（模板 ID）和参数字段。

**第二步：用户授权（前端）**

订阅消息分两种：

- **一次性订阅**：用户授权一次只能发一次。用户点了「立即报名」之后可以给他发「报名成功通知」，再次发送需要重新授权
- **长期订阅**：限定特定行业（公共服务、金融），普通业务申请不到

```js
wx.requestSubscribeMessage({
  tmplIds: ['xxxxxx'],
  success(res) {
    if (res['xxxxxx'] === 'accept') {
      // 用户授权了,通知后端记录一次「可发送配额」
      api.recordSubscription({ tmplId: 'xxxxxx' })
    }
  }
})
```

这里有个硬约束：必须是用户主动点击触发，不能在 `onLoad` 自动弹（微信会拒绝）。订阅弹框 24 小时内只能弹一次同样的模板。

**第三步：后端发送（云函数 / 自建服务）**

```js
// 自建服务用 axios
async function sendSubscribeMessage(openid, tmplId, data) {
  const accessToken = await getAccessToken()   // 用 appid + secret 换

  await axios.post(
    `https://api.weixin.qq.com/cgi-bin/message/subscribe/send?access_token=${accessToken}`,
    {
      touser: openid,
      template_id: tmplId,
      page: 'pages/index/index',
      data: {
        thing1: { value: '订单已发货' },
        time2: { value: '2026-05-12 14:30' },
        thing3: { value: '韵达快递 12345' },
      }
    }
  )
}

// 云开发成本更低:云调用,不用自己管 accessToken
const result = await cloud.openapi.subscribeMessage.send({
  touser: 'OPENID',
  templateId: 'xxxxxx',
  data: { ... },
  page: 'pages/order/detail',
})
```

**第四步：发送时机管理**

业务里通常不是「立刻发」，而是「等触发条件满足再发」：

- 用户报名 → 授权订阅 → 后端记录「这个 openid 有一次报名成功通知配额」
- 数据库存储待发送任务（openid、tmplId、触发条件、过期时间）
- 触发条件满足（订单到货、活动开始前 1 小时）→ 队列发送 → 消耗配额

**常见问题**：

- **配额一次性消费**：一次授权 = 一次发送权限，发出去就没了
- **模板字段格式严格**：`thing.value` 不能超 20 字、`time.value` 必须是日期格式，错了直接发送失败
- **AccessToken 限频**：2 小时有效，要本地缓存。多实例服务还要分布式锁防止重复刷新
- **错误码 47003**：模板审核不通过、参数格式不对，看微信文档对照修

一条建议:

订阅消息把发送任务做成「队列 + 定时检查」的模式，不要在业务逻辑里直接发——发送失败要重试、配额过期要清理，独立任务队列管理起来清晰。

### 小程序怎么向鸿蒙生态迁移？

鸿蒙 Next 24 年发布后，越来越多公司被要求「适配鸿蒙」。问题是小程序的代码能直接运行在鸿蒙上吗？答案分几种情况。

**情况一：纯 H5 / web-view 渲染的小程序**

较简单——鸿蒙浏览器（基于 ArkWeb）支持完整的 Web 标准，H5 直接运行就能跑。如果你的小程序大量内容是 web-view 嵌 H5，那 H5 部分迁移成本几乎是 0。

**情况二：原生小程序代码**

微信小程序的 WXML / WXSS / JS 是微信生态私有的，鸿蒙不能直接运行。要么：

- **走鸿蒙的「快应用」**：鸿蒙生态里类小程序的轻量应用，但 API 跟微信小程序不通，要重写
- **走鸿蒙原生 App**：用 ArkTS + ArkUI 写，跟小程序代码差异明显的范式

**情况三：用跨端框架的项目（Taro / uni-app）**

Taro 4.x 和 uni-app 都在做鸿蒙输出。原理是把跨端框架的产物编译成鸿蒙能跑的形式：

- Taro 鸿蒙端：编译成 ArkTS + ArkUI 代码（开发中，2025 年逐步成熟）
- uni-app 鸿蒙端：DCloud 跟华为合作的「uni-app x → ArkTS」编译路径

这两条路相对可行——一套代码出微信小程序 + 鸿蒙快应用 + 鸿蒙原生。但都还在快速迭代，生态完整度不如微信端。

**实战路径**：

如果业务必须上鸿蒙：

- **轻量营销页 / 内容页**：纯 H5，鸿蒙浏览器打开即可
- **需要类小程序体验**：用 Taro 4 / uni-app 跨端编译到鸿蒙快应用，享受跨端复用
- **核心 App 体验**：原生 ArkTS 写，跟小程序代码无复用

鸿蒙生态短期内**不是「再造一个微信小程序」**，而是华为想在自己的设备上重建一个独立的应用生态。原有的微信小程序代码不能直接复用，但跨端框架在搭桥。

### 小程序的自定义 tabBar 怎么做？什么时候才该做？

小程序的 tabBar 默认配置是 `app.json` 里写死的——图标、文字、跳转路径全是静态的。要做更复杂的效果（中间凸起按钮、根据用户身份动态显示不同 tab、tab 上加红点动画）就得自定义。

**配置自定义 tabBar**：

```json
// app.json
{
  "tabBar": {
    "custom": true,
    "color": "#999",
    "selectedColor": "#1890ff",
    "list": [
      { "pagePath": "pages/home/home", "text": "首页" },
      { "pagePath": "pages/cart/cart", "text": "购物车" },
      { "pagePath": "pages/me/me", "text": "我的" }
    ]
  }
}
```

然后在项目根目录建 `custom-tab-bar/` 文件夹，里面有 `index.js / wxml / wxss`：

```js
// custom-tab-bar/index.js
Component({
  data: {
    selected: 0,
    list: [
      { pagePath: '/pages/home/home', icon: '/img/home.png', text: '首页' },
      // ...
    ]
  },
  methods: {
    switchTab(e) {
      const url = e.currentTarget.dataset.path
      wx.switchTab({ url })
      this.setData({ selected: e.currentTarget.dataset.index })
    }
  }
})
```

```xml
<!-- custom-tab-bar/index.wxml -->
<view class="tab-bar">
  <view
    wx:for="{{list}}" wx:key="index"
    class="tab-bar-item {{selected === index ? 'active' : ''}}"
    data-path="{{item.pagePath}}" data-index="{{index}}"
    bindtap="switchTab"
  >
    <image src="{{item.icon}}" />
    <text>{{item.text}}</text>
  </view>
</view>
```

**最需要注意的问题：tabBar 实例不共享**

自定义 tabBar 在每个 tab 页面都有**独立的组件实例**。如果你在首页 setData 把当前 tab 设成 0，切到购物车 tab 时它自己的实例 selected 还是默认值。

```js
// 每个页面的 onShow 都要同步一遍
Page({
  onShow() {
    if (typeof this.getTabBar === 'function' && this.getTabBar()) {
      this.getTabBar().setData({
        selected: 1   // 当前页对应的 tab 索引
      })
    }
  }
})
```

**适合自定义 tabBar 的场景**：

- 设计师非要中间一个大圆按钮、tab 上要加动画
- 不同身份用户看到不同 tab（B 端用户看 5 个、C 端看 3 个）
- tab 上要动态加角标 / 红点 / 数字

**不建议用的场景**：

- 单纯换个颜色、改个图标——直接 app.json 配置即可
- 简单页面切换——原生 tabBar 性能更好、维护更轻

性能上自定义 tabBar 比原生重不少——切 tab 时整个组件要重新 render，原生 tabBar 是宿主直接切换。中低端机上体感明显。

### 小程序里的离线 / 资源预加载方案

跟 Web 的 Service Worker 不一样，小程序没有 SW 机制，但有自己的资源缓存策略。

**默认的代码包缓存**：

小程序代码包下载后会缓存在用户设备里，下次启动直接读缓存（不需要重新下载）。微信会自动管理这套缓存，开发者基本不用操心。

但有个限制：**代码包更新走「冷启动 → 后台下载新版本 → 下次启动用新版」**。意味着用户点开小程序看到的可能是上次的旧版本。要立刻强制更新：

```js
// app.js
App({
  onLaunch() {
    const updateManager = wx.getUpdateManager()
    updateManager.onUpdateReady(() => {
      wx.showModal({
        title: '更新提示',
        content: '新版本已经准备好，是否重启应用？',
        success(res) {
          if (res.confirm) {
            updateManager.applyUpdate()
          }
        }
      })
    })
  }
})
```

**静态资源预下载**：

图片、字体、视频这些静态资源不打进代码包（也不应该——容易超过 2MB 上限），从 CDN 走。预下载有两种方式：

**1. 分包预下载**（前面讲过）：把资源放在分包里，配 `preloadRule` 提前加载。

**2. 用 `wx.preloadAssets`（基础库 3.6+）**：

```js
wx.preloadAssets({
  data: [
    { type: 'image', src: 'https://cdn.example.com/banner.jpg' },
    { type: 'video', src: 'https://cdn.example.com/intro.mp4' },
  ],
  success: () => console.log('预下载完成')
})
```

下载完资源进微信的 CDN 缓存，业务里 `<image src="...">` 时直接命中缓存。

**3. wx.downloadFile + 本地路径**：

```js
// 提前下载并缓存到本地
const { tempFilePath } = await wx.downloadFile({
  url: 'https://cdn.example.com/big-image.jpg'
})
// 存到 storage
wx.setStorageSync('cachedImage', tempFilePath)

// 使用时
const path = wx.getStorageSync('cachedImage')
// <image src="{{ path }}" />
```

注意 `tempFilePath` 是「会被微信回收的临时路径」——用户清缓存或某些时机会失效，业务里要做兜底（命中失败再走网络）。

几条实践经验：

- 首页大 banner 一定要走「分包预下载 + 预缓存」组合，否则首屏白图
- 视频流 / 长图详情页:用 `wx.preloadAssets` 提前下载
- 不要把图片 base64 进 wxml/wxss——会让代码包暴涨

### H5 完整的页面生命周期：visibilitychange / pageshow / unload

Web 页面的生命周期事件比想象的多。理解每个事件触发时机能解决不少「切后台后状态丢失」「页面回退数据不刷新」这种问题。

**完整的事件链**：

```
用户打开页面
  ↓
DOMContentLoaded   (DOM 解析完)
  ↓
load               (所有资源加载完)
  ↓
pageshow (persisted: false)
  ↓
[用户使用]
  ↓
切到后台:
  visibilitychange (hidden)
  ↓
切回前台:
  visibilitychange (visible)
  ↓
关闭 / 跳走:
  pagehide / unload
```

**`visibilitychange`**:页面可见性变化(切 tab、最小化、息屏)。这是最该监听的事件:

```js
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'hidden') {
    // 切后台:停止动画、暂停视频、上报埋点
    pauseAnimations()
    sendPendingAnalytics()
  } else {
    // 回前台:刷新数据、恢复播放
    refreshData()
  }
})
```

**`pageshow` / `pagehide`**:用户「点返回」回到这个页面时,会触发 `pageshow`,但不一定重新执行所有 JS——浏览器有 **BFCache(Back-Forward Cache)**,把页面整个状态(DOM + JS 内存)冻结起来,返回时秒级恢复。

```js
window.addEventListener('pageshow', (event) => {
  if (event.persisted) {
    // 从 BFCache 恢复:页面状态都还在,但数据可能过期
    refreshData()
  }
})
```

BFCache 能优化用户体验,但业务中有时需要「禁用 BFCache 强制刷新」——比如用户在订单页支付后回到列表页,需要看到最新状态而不是缓存。

```js
// 禁用当前页面的 BFCache
window.addEventListener('unload', () => {})   // 只要绑了 unload 就不进 BFCache(Safari 之外)

// 更标准的做法:Cache-Control: no-store(服务端控制)
```

**`beforeunload`**:用户尝试离开页面前。常见用法是「表单未保存提示」:

```js
window.addEventListener('beforeunload', (e) => {
  if (hasUnsavedChanges) {
    e.preventDefault()
    e.returnValue = ''   // 现代浏览器忽略自定义文案,但要写
  }
})
```

注意:`beforeunload` 在移动端基本不可靠——iOS Safari 几乎不支持,Android 表现各异。**不要把核心保存逻辑放 beforeunload 里**。

**`unload`**:页面真的要销毁了。这个事件**几乎已经不适合作为可靠方案**——它会阻止 BFCache,且移动端为了省电经常不触发。如果想在页面关闭时上报数据,用:

```js
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'hidden') {
    navigator.sendBeacon('/api/analytics', JSON.stringify(data))
  }
})
```

`navigator.sendBeacon` 即使页面已经关闭也能尽量保证请求发出。这是 24 年起更常见的做法。

**实战:正确的上报时机**:

```js
// ❌ 老做法:unload 上报
window.addEventListener('unload', () => {
  fetch('/api/log', { method: 'POST', body: '...' })
  // 移动端可能根本不触发
})

// ✅ 新做法:visibilitychange = hidden 时上报
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'hidden') {
    navigator.sendBeacon('/api/log', '...')
  }
})
```

切后台、关闭页面、刷新、跳转离开等场景都能覆盖。

### 小程序的「按需注入」「用时注入」具体是什么？

小程序启动时默认会把所有页面 / 组件的 JS 都注入到一个 ENV，启动越慢。微信后期加了两个优化策略。

**按需注入**（`lazyCodeLoading: 'requiredComponents'`）：

```json
// app.json
{
  "lazyCodeLoading": "requiredComponents"
}
```

启动时不再注入所有页面的 JS，**只注入当前打开页面用到的组件**。其他页面进入时才注入。

效果：

- 启动时 JS 解析耗时减少 30-50%（项目越大效果越明显）
- 内存占用减少
- 第一次进入某个页面时多 100-300ms 的解析时间（用户能感知但可接受）

适合：**所有中大型小程序**。基础库 2.11+ 支持，副作用很小，应该默认开。

**用时注入**：

按需注入的「极致版」——某些组件不仅延迟注入，第一次用到时才注入：

```json
// page.json
{
  "usingComponents": {
    "rare-feature": {
      "occurs": "tap"   // 用户点击触发某交互时才注入
    }
  }
}
```

适合：很重的组件（图表、富文本编辑器）但是用户大概率不会用到。

**用例对比**：

- 首页：默认就要展示全部内容 → 按需注入够
- 详情页有「评论区」：用户滚到下面才看 → 评论区组件可以「用时注入」
- 设置页里的「密码修改」：用户大概率不用 → 用时注入

实战中按需注入是普适的、强烈推荐；用时注入是「针对很重的低频组件」的优化，普通业务用不上。

**怎么验证**：开发者工具 → 详情面板 → 体积分析 → 「ENV 注入耗时」一栏能看到优化前后差异。

### 微信小程序「插件」机制是做什么的？

插件使用频率不高，但跨多个小程序复用业务时插件是唯一干净的方案。

**插件**：一种特殊的微信账号，里面装了一组页面 / 组件 / API，可以被其他小程序「引用」，运行时跟主小程序绑定。

**典型应用场景**：

- **第三方服务集成**：腾讯地图、智能客服 SDK、视频播放器等以插件形式提供，小程序引用一行配置就接入
- **多小程序共享业务**：一个公司有 5 个小程序都要「用户协议」「客服弹窗」，做成插件后每个小程序就改一行配置
- **品牌方 + 加盟商**：品牌方做插件（商品列表、下单流程），加盟商小程序引用插件 + 加自己的品牌色

**怎么用**：

```json
// app.json
{
  "plugins": {
    "my-plugin": {
      "version": "1.0.0",
      "provider": "wxabc123"
    }
  }
}

// page.json
{
  "usingComponents": {
    "hello-component": "plugin://my-plugin/hello-component"
  }
}
```

```js
// 引用插件 API
const plugin = requirePlugin('my-plugin')
plugin.sayHello()
```

**插件的限制**：

- 不能直接发起 wx.request（要走插件功能页或者主小程序代理）
- 没有自己的 appid，所有数据走宿主小程序的统计
- 不能调用某些敏感 API（支付、登录等）
- 不能引用别的插件

这些限制是「安全」考虑——插件作者不能利用宿主小程序的能力做坏事。

**跨小程序复用业务**：

真实场景：主小程序、加盟商小程序、第三方合作小程序都要共享「商品下单流程」。做成插件后：

- 商品下单流程在插件里维护
- 三个小程序各自的 app.json 引用插件
- 加盟商可以传 props 控制品牌色、跳转链接
- 主小程序更新插件版本，加盟商主动升级（不强制）

比「每个小程序复制一份代码」维护成本更低——bug 修一处所有地方生效。

**实战注意**：

- 插件版本管理：插件作者发布新版本，主小程序需要主动升级 `package.json` 里的版本号
- 调试：插件开发用「插件项目」，主小程序里调试要先把插件加进项目设置
- 上线流程：插件作者审核通过 → 主小程序使用 → 主小程序也要重新提审（因为插件版本可能变化）

### H5 唤起 App：Universal Link / App Links 详解

H5 链接里点一下能跳到 App 对应页面，这个能力分 iOS 和 Android，技术细节不一样。

**iOS Universal Link**：

苹果 9.0+ 起推荐的方案。原理：

1. App 在 Xcode 里配 `Associated Domains`（如 `applinks:example.com`）
2. 在 `https://example.com/.well-known/apple-app-site-association` 放一个 JSON 配置：

```json
{
  "applinks": {
    "apps": [],
    "details": [
      {
        "appID": "TEAMID.com.example.myapp",
        "paths": ["/products/*", "/orders/*"]
      }
    ]
  }
}
```

3. App 首次启动时，系统自动拉取这个 JSON，记住「example.com/products/* 这种链接由我处理」
4. 用户在 H5 / 短信 / 邮件里点链接 `https://example.com/products/123`，**如果装了 App 直接跳 App，没装就在浏览器打开**

优势：

- **平滑降级**：没装 App 时就当普通链接打开，体验自然
- **安全**：JSON 配置走 HTTPS，App 跟域名绑定，无法被伪造

问题：

- **AASA 文件必须 HTTPS + 正确 Content-Type（application/json）**
- **苹果有缓存**：改了 AASA 不会立刻生效，可能等几小时甚至重装 App
- **从 Safari 地址栏直接输入或粘贴链接不会唤起 App**——必须从其他 App / 网页点击跳转

**Android App Links**：

Google M（6.0）起的方案，跟 Universal Link 类似：

1. App 在 `AndroidManifest.xml` 里配 `intent-filter` + `android:autoVerify="true"`
2. 在 `https://example.com/.well-known/assetlinks.json` 放配置：

```json
[{
  "relation": ["delegate_permission/common.handle_all_urls"],
  "target": {
    "namespace": "android_app",
    "package_name": "com.example.myapp",
    "sha256_cert_fingerprints": ["XX:XX:..."]
  }
}]
```

3. 系统验证「这个 App 真的代表这个域名」（防止恶意 App 抢链接）
4. 用户点链接直接跳 App

跟 iOS 一样的「装了跳 App、没装走浏览器」的体验。

**URL Scheme（老方案，仍在用）**：

```
myapp://product/123
```

老方案，问题是「没装 App 时浏览器会卡几秒、报错」。所以现在通常作为降级方案——先 universal link，唤起失败再 fallback 到 url scheme + 提示用户下载。

**实战唤起代码**：

```js
function openApp(deeplink, fallback) {
  const start = Date.now()
  const ua = navigator.userAgent

  if (/iphone|ipad/i.test(ua)) {
    // iOS:直接 location.href = universal link
    window.location.href = deeplink

    setTimeout(() => {
      if (Date.now() - start < 2000) {
        // 没唤起 App,跳商店
        window.location.href = fallback || 'https://apps.apple.com/...'
      }
    }, 1500)
  } else {
    // Android:可以用 intent:// 或者 universal link
    window.location.href = deeplink

    setTimeout(() => {
      if (!document.hidden) {
        window.location.href = fallback || 'https://play.google.com/...'
      }
    }, 1500)
  }
}
```

**微信 / 抖音 / 浏览器限制**：

第三方 App 内的 WebView 经常禁掉 universal link（微信屏蔽了几乎所有外部唤起）。这种场景只能：

- 引导用户「右上角 → 在浏览器中打开」
- 用「中间页 + 用户主动点击下载」绕

国内 App 互相屏蔽是常态，没法彻底解决——能做的是兜底体验好一些（清晰的提示、友好的引导）。
