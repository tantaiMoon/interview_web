# 性能优化面试题

## 一、性能优化总览

### 说说你对前端性能优化的理解。

前端性能优化的核心目标，不只是“跑得更快”，而是从用户视角提升加载速度、交互流畅度和稳定性。

通常可以从几个层面理解：

1. 加载性能：资源更快到达浏览器。
2. 渲染性能：页面更快完成首屏展示。
3. 运行性能：交互过程中不卡顿。
4. 网络性能：减少请求数、请求体积和延迟。
5. 稳定性优化：降低内存泄漏、长任务和异常对体验的影响。

面试时建议按“网络传输、资源体积、渲染执行、运行时内存”四个维度组织答案。

### 白屏首页应该如何优化？

首屏白屏问题本质上是“首屏可见内容出现得太晚”。

常见优化手段：

1. 路由懒加载，避免主包过大。
2. 组件按需加载。
3. 合理使用 Tree Shaking，减少无效代码。
4. 使用骨架屏提升感知速度。
5. 长列表使用虚拟滚动。
6. 重计算任务交给 Web Worker。
7. 使用 `defer`、`async`、`module` 优化脚本加载。
8. 对图片、字体、样式做首屏优先级控制。
9. SSR 或预渲染优化首屏可见内容。

回答这类题时，最好强调：白屏优化要同时关注“真实速度”和“感知速度”。

## 二、网络与资源加载优化

### 什么是 CDN？它的作用、原理和使用场景是什么？

CDN 是内容分发网络，本质上是把静态资源缓存到离用户更近的边缘节点，以减少访问延迟。

它的作用：

1. 加快静态资源访问速度。
2. 降低源站压力。
3. 提高可用性和抗并发能力。

核心原理：

1. 用户访问资源时，调度系统将请求分配到合适节点。
2. 命中缓存则直接返回。
3. 未命中则回源获取并缓存。

常见使用场景：

1. 图片、视频、脚本、样式文件分发。
2. 下载资源加速。
3. 高频访问的静态资源托管。

### 什么是懒加载？有什么特点？实现原理是什么？

懒加载指的是“资源在真正需要时再加载”，而不是页面初始化时一次性全部加载。

它的特点：

1. 减少首屏资源体积。
2. 降低初始化请求压力。
3. 提升首屏加载速度。

常见实现原理：

1. 监听滚动位置判断元素是否进入可视区。
2. 使用 `IntersectionObserver`。
3. 路由层或组件层按需动态加载。

典型场景：

1. 图片懒加载。
2. 路由懒加载。
3. 组件懒加载。

### 懒加载与预加载的区别是什么？

两者都是资源加载优化策略，但目标不同。

1. 懒加载强调“延后加载”，节省首屏资源。
2. 预加载强调“提前加载”，为未来可能使用的资源做准备。

一句话区分：懒加载是“用到再加载”，预加载是“马上可能用到，先准备好”。

### JS 的常见加载方式有哪些？区别是什么？

常见加载方式包括：

1. 普通同步加载
2. `async`
3. `defer`
4. `type="module"`
5. `preload`
6. `prefetch`

区别大致可以这样理解：

1. 同步脚本会阻塞 HTML 解析。
2. `async` 异步下载，下载完立即执行，顺序不保证。
3. `defer` 异步下载，HTML 解析完后按顺序执行。
4. `module` 天然支持模块化和延后执行。
5. `preload` 是告诉浏览器当前页面很快就需要某资源。
6. `prefetch` 是告诉浏览器未来页面可能需要某资源。

### 如何用 Web Worker 优化长任务？

Web Worker 的作用是把耗时计算放到独立线程执行，避免阻塞主线程。

适用场景：

1. 大量数据计算。
2. 文件解析。
3. 图像处理。
4. 长时间运行任务。

面试时要补一句：Web Worker 能减少卡顿，但它和主线程通信也有成本，不适合把所有逻辑都丢进去。

## 三、渲染与运行时优化

### 回流和重绘是什么？触发条件有哪些？

1. 回流是浏览器重新计算元素几何信息和布局。
2. 重绘是元素外观变化后重新绘制，但不一定重新布局。

常见触发回流的场景：

1. 元素尺寸变化。
2. 元素位置变化。
3. 增删 DOM。
4. 读取会触发布局计算的属性。
5. 窗口尺寸变化。

回流一定会引发重绘，重绘不一定会引发回流。

### 如何避免回流和重绘？

常见优化手段：

1. 批量修改 DOM，而不是频繁逐项修改。
2. 避免频繁读取布局信息和写样式交替出现。
3. 对复杂节点做离线更新后再统一插入。
4. 动画优先使用 `transform` 和 `opacity`。
5. 减少表格布局和复杂层级联动。
6. 使用节流控制高频更新。

### 如何优化动画？

动画优化的核心是减少主线程压力和布局计算。

常见做法：

1. 优先使用 `transform` 和 `opacity`。
2. 尽量使用 `requestAnimationFrame`。
3. 避免在动画过程中频繁触发回流。
4. 控制动画元素数量和复杂度。
5. 必要时使用硬件加速，但不要滥用。

### documentFragment 是什么？和直接操作 DOM 的区别是什么？

`documentFragment` 是一个轻量级文档片段容器，可以先在内存中组织一批 DOM 节点，再一次性插入页面。

它和直接操作 DOM 的区别在于：

1. 直接逐个插入 DOM 可能触发多次渲染开销。
2. `documentFragment` 可以先离线构建，再统一插入，减少性能损耗。

适合场景：

1. 大批量列表节点创建。
2. 一次性插入多个元素。

### 什么是长列表虚拟滚动？适用场景是什么？

虚拟滚动的核心思想是：只渲染可视区域内和附近的少量节点，而不是把整个长列表全部渲染出来。

适用场景：

1. 数据量非常大的列表。
2. 表格、消息流、日志流。
3. 后台管理系统中的长数据页面。

它的价值是减少 DOM 数量、降低渲染开销、提升滚动流畅度。

### 节流和防抖是什么？有什么区别？

两者都是高频事件优化手段。

1. 防抖：连续触发时，只在停止触发一段时间后执行一次。
2. 节流：连续触发时，按固定时间间隔执行。

典型场景：

1. 防抖：搜索输入、表单校验。
2. 节流：滚动事件、拖拽事件、窗口缩放。

## 四、图片与媒体优化

### 如何对项目中的图片进行优化？

图片优化通常从体积、格式、尺寸、加载策略几个方面入手。

常见做法：

1. 选择合适格式。
2. 压缩图片体积。
3. 按需裁剪和输出多尺寸版本。
4. 图片懒加载。
5. 使用 CDN 分发。
6. 高清屏使用 `srcset` 或多倍率资源。
7. 小图标视情况使用 SVG 或图标字体。
8. 避免无意义地把大图转成 base64。

### 常见的图片格式有哪些？使用场景是什么？

常见格式和场景：

1. `JPEG/JPG`：照片类图片，体积较小，不支持透明。
2. `PNG`：需要透明背景、质量要求较高的图片。
3. `GIF`：简单动画，但色彩有限。
4. `WebP`：兼顾压缩率和质量，现代 Web 常用。
5. `SVG`：矢量图，适合图标、Logo、线性图形。
6. `AVIF`：压缩率更高，但要考虑兼容性。

面试时不要只背格式名，要说清楚“是否支持透明、是否适合照片、是否适合图标、兼容性如何”。

### 图片转 base64 有什么优缺点？

优点：

1. 减少小图标的额外请求。
2. 可直接内嵌在 CSS 或 HTML 中。

缺点：

1. 体积通常会比原图更大。
2. 不利于浏览器缓存复用。
3. 不适合大图片。

所以 base64 更适合非常小的资源，而不是通用图片优化手段。

## 五、构建与打包优化

### 如何提高 Webpack 的打包速度？

常见思路有：

1. 缩小 Loader 处理范围。
2. 使用缓存。
3. 多进程处理，如 `thread-loader`。
4. 减少不必要的插件。
5. 优化依赖体积和模块数量。
6. 升级构建工具链和 Node 版本。
7. 合理利用持久化缓存。

像 `HappyPack`、`DllPlugin` 这类方案有历史价值，但现代项目里更多会用更适合当前生态的缓存和并行方案。

### 如何减少 Webpack 打包体积？

常见方法：

1. 路由和组件按需加载。
2. Tree Shaking。
3. Scope Hoisting。
4. 减少三方依赖体积。
5. 组件库按需引入。
6. 抽离公共代码。
7. 使用 `externals` 排除不需要打包的依赖。
8. 压缩 JS、CSS、图片资源。

### 如何用 Webpack 来优化前端性能？

Webpack 不只是打包工具，也直接影响前端性能。

常见优化点：

1. 代码分割，减少首屏包体积。
2. 懒加载和异步 chunk。
3. 提前压缩资源。
4. 配合 Gzip/Brotli 输出压缩资源。
5. 生成长期缓存友好的文件名。
6. 去除无用代码。
7. 优化 source map 策略。

### externals、组件库按需引入、Gzip、Tree Shaking 分别解决什么问题？

1. `externals`：把某些依赖排除出打包结果，减少包体积。
2. 组件库按需引入：避免把整个组件库全量打入项目。
3. Gzip：减少网络传输体积。
4. Tree Shaking：移除未使用代码。

### Tree Shaking 为什么并不是万能的？

因为 Tree Shaking 生效有前提：

1. 依赖 ES Module 静态结构。
2. 代码要尽量避免副作用干扰。
3. 某些 CommonJS 包、动态引用、带副作用模块不容易被准确摇掉。

所以它能减少一部分无用代码，但不能代替整体依赖治理。

## 六、内存与稳定性优化

### 什么是内存优化？为什么说内存泄漏只是冰山一角？

内存优化不只是消灭泄漏，还包括控制对象生命周期、减少无意义占用、避免频繁 GC 抖动。

为什么说内存泄漏只是冰山一角：

1. 有些问题不是绝对泄漏，而是长期高占用。
2. 对象虽然能回收，但回收太慢也会导致卡顿。
3. 频繁创建大对象、缓存失控、DOM 过多都会带来性能问题。

### 常见的内存泄漏场景有哪些？

高频场景包括：

1. `window` 上的监听事件没有正确移除。
2. 定时器没有清理。
3. 闭包错误持有大对象。
4. EventBus 事件没有解绑。
5. 脱离文档的 DOM 仍被引用。
6. 全局缓存无上限增长。
7. 控制台对象引用导致内存无法释放。

### 前端内存分析的流程是什么？

常见流程：

1. 先确认问题现象，是持续上涨、偶发飙升还是交互后不回落。
2. 用 Performance 和 Memory 工具记录行为过程。
3. 对比多次堆快照，找出增长对象。
4. 分析保留路径，定位是谁持有引用。
5. 修复后重新验证是否回落。

### Memory 工具中的关键点有哪些？

做内存排查时常关注：

1. Heap Snapshot
2. Allocation Timeline
3. Detached DOM
4. Retainers / retaining path
5. 对象数量和体积变化趋势

### WeakMap、WeakSet 在内存优化中有什么作用？

它们的核心价值是“弱引用”。

如果对象只被 `WeakMap` 或 `WeakSet` 弱引用，而没有其他强引用，那么对象仍然可以被垃圾回收。

适合场景：

1. 以对象作为键的缓存。
2. 附加元数据管理。
3. 避免因缓存导致对象无法释放。

## 附：常见代码实现

### 1. 图片懒加载

```javascript
const observer = new IntersectionObserver(entries => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      const img = entry.target
      img.src = img.dataset.src
      observer.unobserve(img)
    }
  })
})

document.querySelectorAll('img[data-src]').forEach(img => observer.observe(img))
```

### 2. 长任务拆分示例

```javascript
function chunkTask(list, handler, size = 20) {
  function run() {
    const part = list.splice(0, size)
    part.forEach(handler)
    if (list.length) requestAnimationFrame(run)
  }
  requestAnimationFrame(run)
}
```

### 3. Web Worker 示例

```javascript
const worker = new Worker('./worker.js')
worker.postMessage({ nums: [1, 2, 3] })
worker.onmessage = e => console.log(e.data)
```
