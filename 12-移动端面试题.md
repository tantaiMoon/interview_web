# 移动端面试题

## 一、移动端基础认知

### 说说你对移动端开发的理解。

移动端开发不仅是把页面缩小到手机上显示，而是围绕移动设备特性做整体适配和体验优化。

它和 PC 端相比，通常要重点关注：

1. 屏幕尺寸和分辨率差异。
2. 触摸交互而不是鼠标交互。
3. 网络环境不稳定。
4. 设备性能和内存限制。
5. 浏览器兼容性和 WebView 差异。

一句话总结：移动端开发的核心是“适配 + 性能 + 交互体验 + 稳定性”。

### 移动端开发常见方案有哪些？

常见方案有：

1. 响应式 Web。
2. 单独维护移动端 H5 页面。
3. Hybrid 混合开发。
4. 小程序。
5. 跨端框架，如 React Native、Flutter、uni-app、Taro。
6. 原生开发。

不同方案的取舍主要看：

1. 性能要求。
2. 跨平台复用需求。
3. 团队技术栈。
4. 开发和维护成本。

### H5、Hybrid、小程序、原生 App 的区别是什么？

可以从运行环境和能力边界理解：

1. H5 运行在浏览器中，开发快、发布快，但性能和系统能力受限。
2. Hybrid 是 Web + 原生容器结合，能调用部分原生能力。
3. 小程序运行在平台容器内，体验通常比普通 H5 更稳定，但受平台规则限制。
4. 原生 App 性能和系统能力最强，但开发成本也最高。

面试时最好补一句：没有绝对最优方案，关键看业务场景。

## 二、适配与布局

### 移动端适配的常见方案有哪些？

移动端适配常见方案有：

1. 百分比布局。
2. `rem` 适配。
3. `vw/vh` 适配。
4. Flex 和 Grid 布局。
5. 媒体查询。
6. 响应式图片。

实际项目里常见组合是：

1. `meta viewport`
2. Flex 布局
3. `rem` 或 `vw`
4. 媒体查询做补充

### `rem`、`vw`、`px` 在移动端适配中怎么选？

三者不是完全对立，而是适用于不同场景。

1. `px` 适合固定尺寸场景。
2. `rem` 适合基于根字号整体缩放。
3. `vw` 适合直接按视口宽度做比例适配。

常见经验是：

1. 页面级整体适配可以用 `rem` 或 `vw`。
2. 边框、图标、特殊精细元素仍可能用 `px`。

### viewport 是什么？移动端为什么一定要配置它？

`viewport` 是浏览器中页面显示区域的抽象概念。

移动端如果不配置 viewport，浏览器可能会按桌面网页宽度渲染，再缩放显示，导致页面尺寸和交互都异常。

最常见配置：

```html
<meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0, user-scalable=no">
```

面试时可以说明：不同项目是否禁用缩放，要结合可访问性需求谨慎处理。

### 移动端 1px 问题是什么？怎么解决？

在高 DPR 屏幕下，CSS 中的 `1px` 可能看起来比预期更粗，这就是常说的 1px 问题。

常见解决方案：

1. 使用 `transform: scale()` 缩放伪元素边框。
2. 使用 viewport 缩放方案。
3. 使用 `box-shadow` 模拟细线。
4. 使用 SVG 绘制线条。

### 如何处理刘海屏、安全区域适配问题？

全面屏和刘海屏设备会存在顶部、底部安全区域问题。

常见做法：

1. 配置 viewport 支持安全区域。
2. 使用 `env(safe-area-inset-top)`、`env(safe-area-inset-bottom)` 等环境变量。
3. 底部操作栏、吸底按钮、顶部导航要预留安全边距。

## 三、触摸事件与移动端交互

### 移动端常见触摸事件有哪些？

常见触摸事件有：

1. `touchstart`
2. `touchmove`
3. `touchend`
4. `touchcancel`

与 PC 端不同，移动端更多围绕手势和触摸轨迹处理交互。

### click 300ms 延迟是什么？怎么解决？

早期移动浏览器为了判断用户是否要双击缩放，会在点击后等待一段时间再触发 `click`，这就形成了 300ms 延迟。

常见解决方案：

1. 合理配置 viewport，禁用双击缩放相关行为。
2. 使用更现代的浏览器事件机制。
3. 使用 `touch` 事件或成熟手势库。
4. 在旧项目中用 FastClick 一类方案处理。

### 移动端手势冲突常见于哪些场景？怎么处理？

常见场景：

1. 页面滚动与轮播滑动冲突。
2. 下拉刷新与内部滚动区域冲突。
3. 横向滑动组件与竖向页面滚动冲突。
4. WebView 与原生容器手势冲突。

处理思路：

1. 区分横向和纵向手势。
2. 合理阻止默认行为。
3. 控制触发阈值。
4. 将交互优先级设计清楚。

### 移动端如何实现下拉刷新和上拉加载？

常见思路：

1. 监听滚动位置或触摸位移。
2. 下拉超过阈值时触发刷新。
3. 上拉接近底部时加载下一页。
4. 处理并发请求和重复触发。

如果是小程序或框架环境，也可以优先使用平台或组件提供的能力。

## 四、移动端性能优化

### 移动端性能优化应该重点关注哪些方面？

移动端性能优化通常更敏感，重点关注：

1. 首屏加载速度。
2. 包体和资源体积。
3. 图片加载和渲染。
4. 滚动和动画流畅度。
5. 请求数量与弱网表现。
6. 内存占用与长任务。

### 移动端首屏优化有哪些常见手段？

常见做法：

1. 路由和组件懒加载。
2. 骨架屏。
3. 精简首屏数据。
4. 首屏接口预拉取。
5. 图片懒加载和压缩。
6. 关键资源优先加载。
7. 使用 CDN。
8. 非关键逻辑延后执行。

### 移动端列表渲染卡顿怎么优化？

常见思路：

1. 分页加载。
2. 虚拟滚动。
3. 减少单项 DOM 复杂度。
4. 避免频繁重排重绘。
5. 对图片、动画、事件监听做优化。

### 移动端动画卡顿怎么优化？

常见做法：

1. 优先使用 `transform` 和 `opacity`。
2. 使用 `requestAnimationFrame`。
3. 避免动画中频繁读写布局。
4. 降低动画元素数量和复杂度。
5. 把重计算任务移出主线程。

### 弱网环境下怎么优化移动端体验？

常见做法：

1. 减少请求数量。
2. 压缩资源体积。
3. 做缓存和离线兜底。
4. 接口超时和重试策略。
5. 首屏优先加载关键内容。
6. 使用占位图、骨架屏、渐进式展示。

## 五、兼容性与浏览器问题

### 移动端常见兼容性问题有哪些？

常见问题包括：

1. 不同 WebView 行为差异。
2. iOS 和 Android 输入框、滚动表现不同。
3. fixed 定位异常。
4. 键盘弹起导致布局错乱。
5. viewport 和安全区域差异。
6. 某些 CSS 特性支持不一致。

### iOS 和 Android 在前端表现上有哪些常见差异？

常见差异：

1. 输入框聚焦后的滚动和布局行为不同。
2. fixed、sticky 在某些 WebView 表现不同。
3. 日期选择、表单控件默认样式不同。
4. 音视频自动播放、权限策略不同。
5. 滚动回弹和惯性效果不同。

### 移动端键盘弹起导致页面错位怎么办？

这是移动端高频问题。

常见处理思路：

1. 监听输入框聚焦和失焦。
2. 针对 iOS 和 Android 分开处理。
3. 调整底部固定区域位置。
4. 必要时在键盘弹起时禁用某些布局行为。
5. 对滚动容器做回滚修正。

### `position: fixed` 在移动端常见问题有哪些？

常见问题：

1. 某些 WebView 中 fixed 元素抖动。
2. 键盘弹起后 fixed 元素错位。
3. 滚动容器嵌套时 fixed 表现不符合预期。

常见处理方式：

1. 简化滚动层级。
2. 结合 `transform`、`sticky` 或容器布局替代。
3. 针对键盘和安全区做额外适配。

## 六、H5 与原生能力交互

### H5 如何与 App 原生通信？

常见方式有：

1. JavaScript Bridge。
2. URL Scheme。
3. WebView 注入对象。
4. postMessage。

通常流程是：

1. H5 调原生能力。
2. 原生执行后回调 H5。
3. 双方约定统一协议和参数格式。

### Hybrid 应用中常见问题有哪些？

常见问题：

1. WebView 性能瓶颈。
2. H5 与原生通信时序问题。
3. 路由和返回栈管理复杂。
4. 兼容性和版本碎片化。
5. 调试链路复杂。

### H5 页面唤起 App 的常见方式有哪些？

常见方式：

1. URL Scheme。
2. Universal Link。
3. Android Intent。
4. 通过中间落地页唤起。

面试时可以补一句：唤起能力通常要考虑安装态、浏览器限制和失败兜底页。

## 七、移动端安全与发布

### 移动端 H5 安全需要重点关注什么？

重点仍然是 Web 安全，但移动端场景更关注：

1. XSS。
2. CSRF。
3. Token 存储安全。
4. H5 与原生通信安全。
5. 敏感参数暴露。
6. 非法唤起和中间人攻击。

### 移动端项目发布时通常要注意什么？

常见要点：

1. 环境变量区分清楚。
2. CDN 和缓存策略正确。
3. Source Map 管理得当。
4. 监控和埋点提前接好。
5. 弱网和低端机要做回归验证。
6. H5、App、小程序如果多端并存，要确认版本兼容策略。

## 八、移动端面试高频综合题

### 如果让你负责一个移动端项目优化，你会怎么做？

比较完整的回答可以按这个顺序组织：

1. 先定指标：首屏、白屏、交互耗时、崩溃率、错误率。
2. 再做定位：启动慢、请求慢、渲染慢、交互卡顿分别排查。
3. 然后分层优化：资源、网络、渲染、缓存、交互。
4. 最后持续监控：埋点、性能分析、灰度发布、回归验证。

### 移动端项目最容易踩的坑有哪些？

常见坑有：

1. 只做视觉适配，不做交互适配。
2. 忽略 iOS 和 Android 差异。
3. 图片和资源体积过大。
4. 列表和动画过重导致卡顿。
5. 键盘、安全区域、fixed 定位处理不到位。
6. 弱网和低端机体验验证不足。

### 移动端项目中，你会优先建立哪些规范？

常见规范包括：

1. 适配方案统一。
2. 资源尺寸和图片格式规范。
3. 请求和缓存规范。
4. 埋点与监控规范。
5. 兼容性验证规范。
6. 发布和回滚规范。

一句话总结：移动端项目最怕“每个人按自己的方式兼容”，所以规范比技巧更重要。
